# 架构与系统设计题

100 道关于软件架构、分布式系统模式、可扩展性、可靠性、数据建模以及经典系统设计练习的高频题目。

---

### 1. 单体 vs 微服务 vs 模块化单体

**答案：** 单体作为一个可部署单元发布；开发、部署和理解都简单，但会把团队速度和扩展能力耦合在一起。微服务按界限上下文拆分，允许独立部署、多语言技术栈以及按服务扩展，代价是运维复杂度（网络故障、分布式追踪、数据一致性、部署流水线）。模块化单体保留单一部署，但通过包/程序集结构与显式接口强制模块边界；既享有重构安全性，又不必承担分布式系统的"税"。默认选模块化单体，直到团队规模、部署节奏或体量倒逼拆分。当团队数量超过邓巴数量级、服务扩展曲线差异巨大或合规要求隔离时，微服务才划得来。

**要点：**
- 康威定律占主导：架构映射组织结构。
- 微服务带来延迟、部分失败、最终一致性。
- 模块化单体用单一部署获得大部分模块化收益。
- 拆分成本高；在拆之前先把接缝跑通。

---

### 2. SOA vs 微服务

**答案：** SOA 在 2000 年代围绕企业集成兴起：重型 ESB、规范化 XML schema、WS-* 契约和共享数据库非常常见。微服务借用了"服务"这一抽象，但反转了若干默认值：智能端点/笨管道（无 ESB 编排）、去中心化的数据所有权（无共享 schema）、多语言持久化、轻量级 HTTP/gRPC 契约以及按服务持续交付。SOA 优化的是企业级复用和治理；微服务优化的是团队自治和部署速度。实践中加上服务网格或 API 网关后界限会模糊，但文化默认值（谁拥有 schema、业务逻辑放在哪）依然是真正的区分点。

**要点：**
- ESB vs 笨管道是架构分叉点。
- SOA 共享数据；微服务按服务拥有数据。
- 微服务假设有 CD 和容器编排。
- 二者都可做好或做砸；标签远不如耦合度重要。

### 3. 事件驱动架构

**答案：** 服务通过发布不可变事件来通信，而不是同步互相调用。生产者不知道消费者，从而实现松耦合、时间解耦（消费者按自己的节奏处理）以及面向新用例的便捷扇出。权衡：端到端流程更难推理、最终一致性、需要幂等消费者、需要回放/DLQ 工具，以及 schema 演化的纪律（Avro/Protobuf 配合 registry）。当工作流本质上是异步、需要对同一事实做多个下游反应，或希望把写路径与读模型解耦（CQRS）时选 EDA。需要强一致性和即时确认的简单请求/响应场景请避免使用。

**要点：**
- 事件是事实；命令是意图。
- 需要 schema registry、DLQ、幂等键。
- 默认支持 CQRS、事件溯源和审计。
- 跨跳追踪需要关联 ID 与 OpenTelemetry。

---

### 4. CQRS

**答案：** 命令查询职责分离把写模型（变更状态的命令，按不变量校验）和读模型（针对特定视图优化的查询）拆开。写入打到规范化存储；事件或变更数据捕获把每种查询所需的反规范化读模型投影出来（搜索索引、缓存、物化视图）。好处：读写独立扩展、读侧代码更简单、读模型可以独立演化而不动写侧。代价：两侧之间最终一致、组件更多、需要投影重建逻辑。读写不对称大或查询形态众多时使用 CQRS；简单 CRUD 不要套。

**要点：**
- 两个模型，通常两套数据库。
- 读相对于写最终一致。
- 自然搭配事件溯源，但并非必须。
- 当心"读自己写"的 UX 问题。

---

### 5. 事件溯源

**答案：** 把状态以追加式领域事件日志的方式持久化；当前状态通过折叠事件推导。提供完美的审计、时间旅行以及从历史中新增投影的能力。聚合通过回放其事件流来重建（用快照提升性能）。挑战：事件 schema/版本要永远兼容、GDPR 删除（加密粉碎）、复杂查询（需要 CQRS 读模型）以及陡峭的学习曲线。最适合：历史本身是一等业务关注点的领域（金融、账本、合规），或者多个消费者需要相同事实。简单 CRUD 或团队运维成熟度不足时请避开。

**要点：**
- 事件不可变且仅追加。
- 快照加速聚合重建。
- GDPR 需要加密粉碎或墓碑。
- 几乎总是与 CQRS 结合。

---

### 6. 六边形/端口与适配器

**答案：** 应用核心定义其所需一切（数据库、消息总线、HTTP）的端口（接口）。适配器为具体技术实现这些端口。核心不依赖框架或 I/O，因此可被轻易单元测试，并能在不同投递机制（HTTP、CLI、队列消费者）之间互换。权衡：更多样板代码与间接层，容易过度抽象。当领域逻辑丰富且生命周期长，或预期有多种投递通道时最有价值。对于薄 CRUD 服务，这些仪式属于过度设计——用框架默认就好。

**要点：**
- 领域核心与框架无关。
- 驱动方（HTTP、CLI）与被驱动方（数据库、broker）适配器。
- 用内存适配器实现快速测试。
- 易过度工程；视领域价值再用。

---

### 7. 整洁架构（Uncle Bob）

**答案：** 同心圆层级，依赖只能向内指向：实体（企业规则）、用例（应用规则）、接口适配器（控制器、展示器、网关）、框架与驱动（Web、数据库、UI）。外层通过接口依赖内层（依赖反转）。精神上与六边形相似，但对分层更加规定化。优点：业务规则被隔离且可测，框架升级不会扩散。缺点：层与层之间映射很多（到处都是 DTO），小服务会臃肿。选择性应用——保留依赖规则，当严格的四层模板带来噪音时果断丢弃。

**要点：**
- 依赖规则：只能向内。
- 用例编排实体；适配器做翻译。
- DTO 和映射器泛滥——注意成本。
- 适合复杂、长生命周期的领域逻辑。

---

### 8. DDD：界限上下文、聚合、通用语言

**答案：** 领域驱动设计通过围绕业务建模来应对复杂性。界限上下文定义一条显式边界，在边界内某个模型及其通用语言保持一致——同一个词（Customer、Order）在不同上下文里含义可以不同。聚合是一致性边界：一个根实体加上必须在一个事务中共同维护不变量的对象。跨聚合一致性是最终的，通过领域事件实现。界限上下文常常映射到服务边界；上下文映射图记录上下文之间的关系（共享内核、客户/供应商、防腐层）。做得好时，DDD 让代码与对话对齐；做成仪式后，会产出贫血模型加一堆多余的类。

**要点：**
- 通用语言：代码与业务词汇一致。
- 聚合 = 事务/一致性边界。
- 界限上下文通常 = 服务边界。
- 上下文映射图记录上下文间关系。

### 9. 绞杀者无花果迁移

**答案：** 增量替换遗留系统：把特定功能路由到新服务，旧系统并行运行，直到遗留被完全绞杀并下线。一个 facade 或代理（通常是 API 网关或反向代理）按请求决定打到老系统还是新系统。让你能持续交付价值、降低大爆炸风险，并按切片提供回滚能力。风险：双跑时间被拉长、新旧之间的数据同步、业务规则漂移。配合特性开关、（谨慎使用的）双写或 CDC，并为每个模块制定明确的下线计划。把"绞杀百分比"作为先导指标跟踪。

**要点：**
- 代理增量切流量。
- 避免大爆炸重写风险。
- 需要数据同步策略（双写或 CDC）。
- 设定显式的下线里程碑。

---

### 10. 防腐层

**答案：** 在你的界限上下文与外部/遗留模型之间的翻译层，防止外来模型渗入你的领域。实现为一组适配器和翻译器，把外部概念映射到你的通用语言。在集成遗留系统、第三方 API 或他人拥有的上下文时至关重要。代价：额外映射代码、性能开销、外部模型变更时的维护。琐碎集成可跳过；当外部模型混乱、不稳定或与你文化迥异时再用。常出现在绞杀者无花果迁移的边缘。

**要点：**
- 保护领域纯洁性，隔离外来模型。
- 通过适配器与翻译器实现。
- 遗留迁移期不可或缺。
- 维护有成本——按每个集成单独评估。

### 11. BFF 模式

**答案：** Backend-for-Frontend 为每种客户端类型（Web、iOS、Android、合作伙伴）放置一个专属 API 层。每个 BFF 聚合下游微服务、按客户端需要塑形负载、处理客户端特定的认证流程，避免那种"一个 API 包打天下"的妥协（导致载荷臃肿、移动端变慢）。理想情况下由拥有该客户端的同一团队负责。权衡：要运维更多服务、各 BFF 间可能存在逻辑重复（抽公共库/服务）、BFF 与客户端发布节奏的耦合更紧。GraphQL 在某些场景下可让客户端按需投影字段，从而替代 BFF。

**要点：**
- 每种客户端体验对应一套后端。
- 减少过度获取与往返。
- 由客户端团队拥有。
- GraphQL 在部分场景可作为替代。

---

### 12. Sidecar 模式

**答案：** 与主应用部署在同一 pod/主机里、共享生命周期与网络命名空间的辅助进程。常见用途：服务网格代理（Envoy）、日志投递、配置重载器、密钥获取。应用专注业务逻辑；横切关注点放在 sidecar 中并可独立升级。代价：每个 pod 额外的资源开销、部署复杂度、调试"这次请求是谁处理的"（应用还是 sidecar）。适用于辅助进程在许多服务间复用、并由基础设施团队拥有的场景。一次性需求请放回应用里。

**要点：**
- 同位部署的辅助进程。
- 与主容器共享网络/存储。
- 支撑服务网格、日志投递、密钥。
- 每个 pod 都增加开销——要度量。

---

### 13. Ambassador 模式

**答案：** 一个与客户端并行（或作为 sidecar）运行的代理，处理出向关注点：服务发现、重试、断路、TLS、可观测性。应用与 localhost 对话；ambassador 处理网络。让你能升级网络行为而无需修改应用代码，并支持无法被修改的遗留客户端。与 sidecar（通用同位辅助者）的区别在于它特指出向调用的网络代理。服务网格数据面将其一般化。权衡：多一跳、调试复杂度、需要平台投入。

**要点：**
- 与客户端同位的出向代理。
- 处理重试、TLS、发现。
- 把网络与应用代码解耦。
- 服务网格将其一般化。

---

### 14. 服务网格（Istio/Linkerd）

**答案：** 通过 sidecar 代理（数据面）处理服务间通信、由中心控制面控制的基础设施层。提供 mTLS、流量切换（canary、blue/green）、重试、超时、断路、细粒度授权和丰富遥测，且无需修改应用代码。权衡：显著的运维复杂度、每跳延迟开销（约 1-5ms）、每 pod 资源成本，以及又多一个组件要排障。Istio 功能丰富但偏重；Linkerd 更精简且基于 Rust。当服务众多、多语言并存且有平台团队运维时再采用。否则坚持基于库的弹性方案（如 gRPC 重试、Resilience4j）。

**要点：**
- 数据面（sidecars）+ 控制面。
- 提供 mTLS、流量策略、遥测。
- Linkerd 更轻；Istio 更全。
- 用服务数量与团队容量来论证。

---

### 15. Saga（编排 vs 编舞）

**答案：** Saga 把跨服务的长事务协调为一系列局部事务，并在某步失败时执行补偿动作。编排（orchestration）：中心编排器（如 Temporal、Camunda）显式调用每一步并在失败时触发补偿——流程可见、易调试，但编排器是耦合点。编舞（choreography）：每个服务对事件作出反应并发出新事件——耦合松，但流程是隐式的，规模化后难以追踪。编排在带分支的复杂流程下扩展更好；编舞适合简单线性的扇出。两种情况下，每一步都需要幂等的补偿动作。

**要点：**
- 用补偿替代 2PC。
- 编排：中心大脑，更易调试。
- 编舞：只用事件，松散但不透明。
- 每一步都需要幂等的回退。

---

### 16. Outbox 模式

**答案：** 解决数据库与消息 broker 之间的双写问题。服务在一个事务中把业务变更和出向事件写入同一个数据库（"outbox"表）。一个中继（轮询 worker 或 Debezium 这类 CDC 流）读取 outbox 并发布到 broker，标记行已发送。在不需要分布式事务的前提下保证至少一次发布，让数据库与 broker 最终一致。权衡：提交到发布之间有微小延迟、outbox 表会增长（需归档），且消费者必须幂等。与事件驱动和 saga 架构配合良好。

**要点：**
- 状态与事件原子写入数据库。
- 中继（轮询或 CDC）向下游发布。
- 至少一次投递；消费者必须去重。
- 避免对 broker 的不可靠双写。

### 17. 协作状态的 CRDT

**答案：** Conflict-free Replicated Data Types 是一类数据结构，多个副本上的并发更新可以在无协调的情况下被确定性合并，达成强最终一致。例子：G-Counter、PN-Counter、OR-Set、LWW-Register、文本用的 RGA/Yjs。基于状态的 CRDT 发送完整状态并通过 join 函数合并；基于操作的通过可靠因果广播发送 op。被用于协作编辑器（Figma、Linear、Notion 本地优先）、离线优先应用以及多区域数据库（Riak、Redis Enterprise）。权衡：元数据开销增长、意图（相对于原始合并）可能产生意外结果，并非每个问题都能映射到 CRDT。

**要点：**
- 数学上保证收敛。
- 无需协调，无需中心权威。
- Yjs/Automerge 在协作文本中流行。
- 元数据开销可能很大。

---

### 18. Lambda 架构 vs Kappa 架构

**答案：** Lambda 并行运行两条数据流水线：批处理层（慢、准、从原始重算）和速度层（快、近似、近实时），由服务层合并两侧。优点：能处理迟到数据和重算。缺点：两套代码、两套系统、同步漂移。Kappa 通过使用单一可从持久日志（Kafka）回放的流式管道来简化：需要重算？把日志回放进新的流任务即可。现代流处理器（Flink、Spark Structured Streaming）模糊了边界。当 broker 能回放历史时默认选 Kappa；只有当批处理工具对冷计算明显更便宜时再保留 Lambda。

**要点：**
- Lambda：批 + 速度层，逻辑重复。
- Kappa：纯流式，从日志回放。
- Kappa 需要持久且可回放的日志（Kafka）。
- 现代引擎减少了这种二分。

---

### 19. 管道-过滤器 vs 编排工作流

**答案：** 管道-过滤器：通过流或队列连接的独立阶段，每个阶段读输入、变换、写输出——非常适合 ETL、媒体管道、日志处理。各阶段可独立扩展，过滤器可插拔。编排工作流：工作流引擎（Temporal、Airflow、Step Functions）定义步骤、重试、分支、定时器和人工介入——非常适合有状态、有分支、长时长的业务流程。管道擅长高吞吐数据变换；编排器擅长正确性、可见性和长时间协调。许多系统两者都用：在编排工作流的某一步里嵌套管道。

**要点：**
- 管道：流式、无状态阶段。
- 编排器：有状态、分支、长跑。
- 两者可在同一系统并存。
- 按延迟、状态与可见性需求选择。

---

### 20. 无服务器 vs 容器 vs 虚拟机

**答案：** 虚拟机提供完整的 OS 隔离、可预测的性能、最长的冷启动、最高的运维开销。容器打包应用 + 依赖，共享内核，启动快，密度高——是有状态或常在工作负载的默认选择。无服务器（如 Lambda 的 FaaS 或基于容器的 Cloud Run）完全抽象掉服务器，可缩到零，按调用计费，但有冷启动、运行时限制和厂商锁定。把无服务器用于尖峰、事件驱动或低流量场景，那里运维节省占主导。把容器用于稳态服务，那里单位成本重要、需要完全掌控。把虚拟机留给遗留、GPU 或需要强隔离的合规工作负载。

**要点：**
- 无服务器：按使用付费、冷启动、缩到零。
- 容器：密度与控制力最佳。
- 虚拟机：隔离最强，开销最高。
- 成本交叉点大约在 30-50% 利用率。

---

### 21. 水平扩展 vs 垂直扩展

**答案：** 垂直（scale-up）给单节点加 CPU/RAM——简单、无需改代码，但有天花板且是单点故障。水平（scale-out）在负载均衡器后加节点——理论上无上限、容错，但要求无状态或把共享状态外置，并增加协调开销。大多数现代服务起步就把无状态服务放在 LB 后，所以水平扩展是平凡的。数据库历史上以垂直为主；现代系统（Cassandra、Spanner、CockroachDB）通过分片/共识进行水平扩展。混合很常见：先垂直扩展直到成本或机型上限，再分片。

**要点：**
- 垂直：简单、有天花板、单一故障域。
- 水平：需要无状态或外置状态。
- 数据库最难水平扩展——需要分片。
- 组合：更大节点 + 更多节点。

---

### 22. 无状态 vs 有状态

**答案：** 无状态服务不在本地保存会话/数据；任何实例都能服务任何请求，使水平扩展、滚动部署、故障恢复都变得平凡。状态放在数据库、缓存或对象存储里。有状态服务把状态保存在内存或本地磁盘（Kafka broker、Elasticsearch 节点、游戏服、WebSocket 连接状态），需要粘性路由、扩缩时小心再平衡、恢复更慢。目标：无状态应用层 + 把状态推到托管的有状态系统。当必须有状态时，使用一致性哈希、持久卷和 pre-stop 钩子优雅排空。

**要点：**
- 无状态 = 水平扩展平凡。
- 有状态需要粘性路由与再平衡。
- 尽量把状态推到托管存储。
- HTTP 的粘性会话是反模式。

---

### 23. 负载均衡（RR、最少连接、哈希、加权）

**答案：** 轮询（Round-robin）在请求均匀时分布均匀；便宜简单。最少连接路由到活跃连接数最少的实例——更适合长连接或耗时变化大的请求。一致性哈希按 key（用户 ID、租户）路由，让相同 key 持续命中相同后端——对缓存和有状态系统至关重要，拓扑变化时最小化再洗牌。加权变体可处理异构实例规格或 canary 流量切换。带"两个选择"的随机（power-of-two）通常在尾延迟上优于 RR。按流量形态选：均匀短请求 = RR、可变 = 最少连接、缓存亲和 = 哈希。

**要点：**
- RR：简单、假设请求均匀。
- 最少连接：处理可变时长。
- 一致性哈希：缓存与有状态亲和。
- power-of-two-choices 降低尾延迟。

---

### 24. L4 vs L7 负载均衡

**答案：** L4（TCP/UDP）在传输层做均衡——快、协议无关、不检查载荷（AWS NLB、HAProxy TCP 模式）。L7（HTTP、gRPC）理解应用协议——可按 host/path/header 路由、终止 TLS、做重试、粘性 cookie、限流和可观测性（Envoy、NGINX、ALB）。L4 更便宜、延迟更低；L7 更灵活，是当今多数现代 API 流量的栖息地。常见栈：L4 用于原始 TCP/UDP（游戏、数据库），L7 用于 HTTP API，L4 在 L7 集群前承担 DDoS 规模。服务网格本质上就是分布式 L7 LB。

**要点：**
- L4：TCP/UDP，最快，协议盲。
- L7：HTTP 感知，路由/策略丰富。
- L7 终止 TLS 并做重试。
- 常用 L4 前置 L7 以应对规模。

---

### 25. 粘性会话

**答案：** LB 把客户端（cookie 或 IP 哈希）固定到某个后端，让内存中的会话状态能用。便宜地解决了无状态空缺，但损害负载均衡、复杂化滚动部署（杀掉实例就丢失会话，除非排空），并产生热点。更好的替代：把会话外置到 Redis/Memcached，使用签名 JWT 让任何实例都能验证，或 SPA + token 鉴权。粘性仍有狭义合理用途：WebSocket 连接（TCP 套接字本身就是粘性的）、某些遗留应用，以及面向缓存亲和的一致性哈希路由（与粘性不同）。

**要点：**
- 通过 cookie/IP 把客户端绑定到后端。
- 破坏均匀负载与干净发布。
- 优先外置会话或 JWT。
- WebSocket 天然粘性。

---

### 26. 自动扩缩触发器

**答案：** 反应式：按 CPU、内存、请求率、队列深度、p95 延迟扩缩——简单但对突发滞后几分钟。预测式：按计划扩缩（高峰时段）或 ML 预测——能应对已知模式但错过意外。最佳实践：组合。可能时优先用基于工作量的信号（每副本 RPS、队列深度）而非 CPU——在 I/O 密集型工作负载下 CPU 会说谎。设保守的缩容以避免抖动，快速扩容以吸收尖峰。封顶上限以防反馈循环导致失控成本。用负载测试验证扩缩在现实坡度下能保 SLO，而不仅是在稳态。

**要点：**
- 优先用基于工作量（RPS、队列深度）的信号而非 CPU。
- 扩容要快，缩容要慢（冷却）。
- 反应式 + 计划预测式相结合。
- 封顶以约束成本。

---

### 27. CAP 与 PACELC 的实操

**答案：** CAP 说在网络分区时必须在一致性和可用性之间选。PACELC 扩展之：即便没有分区（Else），也要在延迟（Latency）和一致性之间权衡。真实系统按操作做选择：银行转账偏好 CP（分区时拒绝），社交动态偏好 AP（陈旧但可用）。许多"一致"系统是 PC/EL（任何时候强一致，正常时低延迟），如 Spanner；许多"可用"系统是 PA/EL（始终可用、快、最终一致），如 Dynamo。把 CAP 当作框架工具而非打勾——按工作负载、按操作分析。

**要点：**
- CAP 只在分区时适用。
- PACELC 增加正常情况下延迟 vs 一致性。
- 选择按操作，而非按系统。
- Spanner ≈ CP；Dynamo ≈ AP。

---

### 28. 最终一致性模式

**答案：** 用工具拥抱陈旧并将其掩盖：读自己写（写后把读路由到主或带写 token）、单调读（粘性会话到同一副本）、因果一致（用矢量时钟或会话 token 跟踪 happens-before）、有界陈旧（Cosmos DB 风格：最多落后 N 秒或 M 操作）。对 UX，乐观 UI（本地先应用，确认后协调）能掩盖延迟。对数据正确性，幂等写入和 CRDT 消除合并冲突。按端点记录一致性契约，让消费者不要假设强语义。

**要点：**
- 通过主路由或 token 实现读自己写。
- 有界陈旧提供可度量的 SLA。
- 乐观 UI 对用户隐藏延迟。
- 按 API 记录一致性契约。

---

### 29. 强一致性 vs 最终一致性 vs 因果一致性

**答案：** 强（线性化）：每次读看到最新提交的写；昂贵、需要共识、限制可用性与延迟。最终：副本在写停止后会收敛；最便宜、最可用，但读可能陈旧或乱序。因果：保留 happens-before 关系（若 A 引起 B，所有观察者都先看到 A 再看到 B）而不要求全序——协作应用和聊天的甜点。顺序一致介于强与因果之间。按操作选择：账户余额读 = 强；社交时间线 = 最终；线程内的聊天消息 = 因果。

**要点：**
- 强：线性化，昂贵。
- 最终：便宜，可能陈旧或乱序。
- 因果：保留因果序。
- 按操作选，而非按系统。

---

### 30. 分布式事务：2PC vs sagas

**答案：** 两阶段提交通过协调器（prepare/commit）跨多个资源协调一个事务。保证原子性但在协调器失败时阻塞、扩展不佳、把服务紧紧耦合在一起——在微服务间很少用。Saga 把事务拆为局部事务加补偿：若第 3 步失败，对 1、2 步执行回退。在不加锁的情况下提供最终原子性、可扩展，但补偿必须幂等且语义有意义（你不能"取消扣款"，必须执行真实退款）。2PC 只在单个数据库集群或 XA 感知基础设施里使用；跨服务用 saga。

**要点：**
- 2PC：阻塞，跨服务不可扩展。
- Saga：局部事务 + 补偿。
- 补偿必须幂等且有意义。
- 编排式 saga 比编舞式更易调试。

---

### 31. 领导者选举（Raft 基础、etcd/ZK）

**答案：** 分布式系统常需要单一领导者来协调（配置、锁、序列号）。Raft 通过随机超时和多数派投票选举领导者；领导者把日志复制给跟随者，多数派确认后提交条目。etcd 和 Consul 使用 Raft；ZooKeeper 使用 ZAB（类似保证）。客户端把写路由到领导者；读可以从跟随者以有界陈旧或经由领导者以线性化方式服务。故障切换约几百毫秒到秒级。把这些系统用于服务发现、领导者选举、分布式锁和配置——千万别当通用数据库用。

**要点：**
- Raft = 领导者 + 日志复制 + 多数派提交。
- etcd、Consul 用 Raft；ZK 用 ZAB。
- 用于锁、发现、配置——不用于数据。
- 故障切换以秒计；需要法定人数。

---

### 32. 法定人数与副本数

**答案：** N 个副本时，写需要 W 个确认，读取需要 R 个回复；强一致需要 W + R > N（重叠保证看到最新的写）。RF=3、W=2、R=2 是 Dynamo 经典：容忍一台故障且保强读。降低 W（=1）写更快但故障时有丢数据风险；降低 R（=1）读更快但可能错过最新写。提示交接（hinted handoff）和读修复（read-repair）异步修复不一致。更高 RF 提升耐久性但增加存储与写延迟。按工作负载调优：写多的系统降 W 并依赖读修复；读多的系统降 R。

**要点：**
- W + R > N 才能强一致。
- RF=3 配 W=R=2 是甜点。
- Hinted handoff 修复临时故障。
- 更高 RF = 更耐久、更高成本。

---

### 33. 只读副本与复制延迟

**答案：** 通过把数据复制到 N 个副本（异步或半同步）将读流量从主库卸载。异步快但读可能陈旧（在大量写、网络问题或副本重启时延迟会飙升）。半同步在提交前等待至少一个副本确认——故障切换更安全，写稍慢。监控延迟（秒数和落后字节）并按阈值告警（如 >5s）。把关键读（账户余额、写后读）路由到主，或使用"读自己写"token。注意：用副本扩展读，能撑到主库写吞吐成为瓶颈为止——之后就要分片了。

**要点：**
- 异步 = 快但陈旧；半同步 = 更安全。
- 同时监控延迟的秒数与字节数。
- 写后读路由到主。
- 副本不能扩写——之后分片。

---

### 34. 分片策略与再平衡

**答案：** 哈希分片：shard = hash(key) % N；分布均匀，但 N 变化时会重洗（用一致性哈希或虚拟节点缓解）。范围分片：每个分片一段连续范围（适合范围查询，易热点）。目录/查找：显式分片映射（灵活，但映射本身是瓶颈/SPOF）。地理分片：按地区获得局部性。再平衡是难点——无停机数据迁移需要双写、回填和切换，或选一个原生支持的系统（Cassandra、Vitess、Cockroach）。一定在上线前规划再分片；起步就选高数量的虚拟分片。

**要点：**
- 哈希：均匀但无范围查询。
- 范围：范围扫描，热点风险。
- 一致性哈希最小化重洗。
- 用大量虚拟分片便于再平衡。

---

### 35. 热点 key 问题

**答案：** 单个 key（Bieber 的 Twitter、爆款商品）给一个分片或缓存条目带来不成比例的负载，把它打饱，而集群其余空闲。修复：客户端请求合并（singleflight）、短 TTL 的本地缓存、key 拆分（在 key 后缀加随机扇出并聚合）、专门的热 key 缓存层，或为该 key 加只读副本。对写入，可以批量并异步聚合（如计数器分片：写到 N 个副本之一，读时求和）。通过每 key 指标或采样追踪检测；在 on-call 被叫醒之前先缓解。

**要点：**
- 通过每 key 指标、采样 trace 检测。
- Singleflight 与本地 TTL 缓存吸收读。
- key 扇出（分片计数器）用于写。
- 在爆红之前规划，而不是事后。

---

### 36. 缓存层与失效

**答案：** 层次：浏览器、CDN、边缘缓存、API 网关缓存、应用缓存（进程内）、共享缓存（Redis/Memcached）、数据库缓冲池。策略：cache-aside（懒加载，应用管理）、read-through（缓存在 miss 时取）、write-through（同步写入缓存 + 数据库）、write-behind（异步写入，有丢失风险）。失效选项：TTL（最简单，接受陈旧）、写时显式失效（一致但耦合）、事件驱动（CDC 触发失效）。雪崩通过请求合并、抖动 TTL 和 stale-while-revalidate 缓解。缓存命中率与尾延迟是重要指标。

**要点：**
- 层次众多——每一层都要度量。
- cache-aside 是默认。
- TTL + 抖动防止雪崩。
- 失效是难题。

---

### 37. 背压与边缘限流

**答案：** 没有背压时，慢消费者会让队列无限膨胀、延迟飙升并 OOM。在每个边界都施加：满则丢或阻塞的有界 channel/队列、最大在途信号量、边缘按客户端/租户的令牌桶限流、每后端并发上限。返回 429 + Retry-After 加少量抖动退避建议。自适应并发限流器（Netflix concurrency-limits、AIMD 风格）按观测到的延迟动态调整。区分：限流防滥用，背压防过载。两者结合。

**要点：**
- 队列处处有界；永远不要无界。
- 边缘按租户使用令牌桶。
- 429 + Retry-After + 抖动。
- 自适应并发限制优于静态。

---

### 38. 断路器

**答案：** 包裹远程调用；跟踪失败率；超过阈值时打开（快速失败）以停止猛击坏依赖；冷却后半开来探测恢复；成功则关闭。防止级联失败、释放被重试循环占住的资源。配置：错误阈值（如最近 100 请求中 50%）、打开时长（10s）、半开探测次数。重要：调优回退行为——陈旧缓存、默认响应、入队稍后处理。不能取代超时（永远先设超时）。在库（Resilience4j、Polly）或服务网格中实现。

**要点：**
- 状态：关闭、打开、半开。
- 防级联并节省资源。
- 配合超时与回退。
- 库或网格提供。

---

### 39. 舱壁（Bulkhead）

**答案：** 隔离资源，使一处故障不会沉船——名字来自船的隔舱。具体做法：为每个下游依赖分配独立线程池或连接池，让慢的第三方 API 不会耗尽你的共享池而饿死其他端点。在服务层面：租户隔离池、按优先级队列。在基础设施层面：为关键工作负载分集群。权衡：每个资源切片不能互相共享，最大利用率下降。但对任何延迟特征多样、关键 vs 尽力交付流量并存的系统，缩小爆炸半径很值。

**要点：**
- 每依赖线程/连接池隔离。
- 限制单点故障的爆炸半径。
- 利用率降低，韧性提升。
- 与断路器组合使用。

---

### 40. 超时、重试、指数退避、抖动

**答案：** 每次远程调用都要超时——无限等待会级联成大故障。把超时设得比调用者的超时短（预算传播）。只对幂等操作重试；限制次数（通常 3 次足够）；用指数退避避免惊群；加上完全抖动让重试不重新同步。不重试 4xx（客户端错误）；只重试 5xx 和网络错误。与断路器结合：依赖坏了还重试只会更糟。把重试率作为健康信号——突然飙升常常预示故障。

**要点：**
- 处处设超时；比父级短。
- 只对幂等操作重试，有上限。
- 完全抖动 > 无抖动以打散惊群。
- 断路打开时跳过重试。

---

### 41. SQL vs NoSQL 决策矩阵

**答案：** SQL（Postgres、MySQL）：查询丰富、事务、连接、工具成熟、强 schema——没有特殊原因就是默认。文档（MongoDB、DynamoDB）：schema 灵活、单文档原子性，适合形态多变与高写吞吐。宽列（Cassandra、ScyllaDB）：海量写吞吐、可调一致性、时序。键值（Redis、DynamoDB）：简单访问模式、亚毫秒延迟。图（Neo4j）：关系密集查询。按访问模式（提前已知的查询适合 NoSQL 建模）、一致性需求、规模（多数应用在 Postgres 里能跑很久）和运维成熟度决定。不要"为了规模"选 NoSQL，除非 SQL 已证明不够。

**要点：**
- SQL 是默认，直到被证伪。
- NoSQL 的选择依赖访问模式。
- 大规模写多用宽列。
- 真正关系查询用图数据库。

---

### 42. 多语言持久化

**答案：** 在一个系统里为不同需求使用不同存储技术：Postgres 用于事务、Redis 用于缓存/队列、Elasticsearch 用于搜索、S3 用于二进制对象、ClickHouse 用于分析、Neo4j 用于图。让每种工作负载用最合适的工具。代价：更多运维负担（每种存储的备份、升级、安全）、存储间的数据同步（CDC、双写）、团队专业能力被摊薄，以及跨存储事务更难。从单一存储开始；只有当访问模式清晰时才加新存储。把次级存储视为系统的真相记录的投影。

**要点：**
- 按访问模式选对工具。
- 每种存储都加运维负担。
- 通过 CDC/事件从真相源同步。
- 抵制过早过度碎片化。

---

### 43. CDC 与下游扇出

**答案：** Change Data Capture 接入数据库的复制日志（WAL/binlog），把行变更事件输出到流（Debezium → Kafka）。消费者将其投影到搜索索引、缓存、数据湖、其他服务——无需触碰应用代码。优点：应用保持简单；捕获可靠（应用在数据库和 broker 之间崩溃不会漏事件）；获得可回放的事件日志。缺点：事件暴露物理 schema（耦合消费者）、顺序与 exactly-once 语义需要小心、schema 变更会破坏消费者。对领域事件优先用 outbox（来自应用的语义事件）；对底层数据同步，CDC 理想。

**要点：**
- 读 WAL/binlog；无需改应用。
- 可靠：应用崩溃不会漏。
- 把消费者耦合到物理 schema。
- CDC 做数据同步，outbox 做领域事件。

---

### 44. Outbox vs CDC 做事件

**答案：** Outbox 在与状态变更相同的事务中写入具有语义意义的领域事件（OrderPlaced），再由中继发布。事件由应用定义、稳定、与 schema 解耦。CDC 从数据库日志派生低层行变更（UPDATE orders SET status='paid')——无需应用代码、无需额外表，但消费者看到物理 schema 并需要重建意图。当你掌控生产者并想要稳定的领域事件时用 outbox。当你无法修改生产者（遗留数据库）或确实想要数据级同步（搜索索引、湖）时用 CDC。

**要点：**
- Outbox：领域事件、schema 稳定。
- CDC：行级、无需应用改动。
- Outbox 需要中继；CDC 需要 Debezium。
- 两者都是至少一次；消费者必须去重。

---

### 45. Broker 选择：Kafka vs RabbitMQ vs SQS vs Pulsar

**答案：** Kafka：耐久、分区日志；高吞吐；可回放；消费者管理偏移；最适合事件流、CDC、分析管道。RabbitMQ：带丰富路由（交换机、主题）的经典 broker；适合任务队列、复杂路由，吞吐较低；消息通常在 ack 时删除。SQS：托管、简单、无限扩展、无顺序（FIFO 变体有上限）；非常适合云原生任务队列。Pulsar：日志 + 队列混合，多租户，原生地理复制，分层存储——Kafka 的竞争者，运维模型更好，但生态更小。按回放需求（Kafka/Pulsar）、路由复杂度（Rabbit）和运维胃口（SQS）选择。

**要点：**
- Kafka：日志、回放、高吞吐。
- Rabbit：路由丰富的任务队列。
- SQS：托管、简单、无运维。
- Pulsar：日志 + 队列，多租户。

---

### 46. Exactly-once 语义

**答案：** "Exactly once"通常指实际上一次：至少一次投递 + 幂等处理。真正的端到端 exactly-once 要求生产者、broker 和消费者都参与（Kafka 事务搭配幂等生产者和事务性消费者偏移，或 Flink 的两阶段提交 sink）。实践中把消费者设计成幂等：按事件 ID 去重、按主键 upsert、在 HTTP 中使用幂等键、保存已处理的事件 ID（TTL 窗口）以拒绝回放。这种方式统一处理 broker 重投、应用重试和网络重复。比追求真 exactly-once 更便宜、更可靠。

**要点：**
- 真 exactly-once 罕见且昂贵。
- 实际上一次 = 至少一次 + 幂等。
- 按事件 ID 去重或按主键 upsert。
- 每个入口都用幂等键。

---

### 47. 流处理（Flink、Kafka Streams、Spark）

**答案：** 处理无界事件流，包括窗口、连接、聚合和有状态算子。Flink：真流式、低延迟、复杂事件时间语义、基于检查点的 exactly-once，复杂有状态管道的最强选手。Kafka Streams：嵌入应用中的库，运维更简单，按分区扩展，与 Kafka 紧绑定。Spark Structured Streaming：流式 API 下的微批，对已在 Spark 上的 ETL 团队很合适，延迟更高。按延迟（Flink 到毫秒）、团队熟悉度（批处理团队用 Spark）和复杂度（嵌入用 Kafka Streams）选择。

**要点：**
- 事件时间 vs 处理时间很重要。
- 水印处理迟到数据。
- Checkpoint 用于 exactly-once 状态。
- Flink 适合最难的有状态工作负载。

---

### 48. 物化视图与读模型

**答案：** 预先计算查询结果以使读快且便宜。在 SQL 数据库（Postgres、Snowflake）中物化视图按计划或提交时刷新。在事件驱动系统中，投影消费事件并写入按查询定制的反规范化读模型——同样思路、不同机制。权衡：写放大、相对源最终一致，以及重建的成本（必须能从原始事件或数据库回放重建）。设计时始终考虑重建：为投影 schema 版本化、把事件保留足够长以重建、在切换期把新投影与旧并行运行。

**要点：**
- 预计算、反规范化的读形态。
- 相对源最终一致。
- 可从事件重建是不可妥协的。
- 为 schema 变更对投影做版本化。

---

### 49. 时序存储

**答案：** 针对按时间戳追加写入、窗口聚合以及保留/降采样进行了优化。可选：InfluxDB、TimescaleDB（Postgres 扩展）、Prometheus（拉、临时）、VictoriaMetrics、ClickHouse（通用但对时序极佳）。关键特性：列式存储、按时间分区的 chunk、自动降采样/上卷、基于 TTL 的保留、快速范围扫描。规模大了别用行存 SQL 数据库——索引膨胀、查询变慢。基数（label 组合的唯一数）是指标系统的隐形杀手；明确预算，并在 ingest 阶段拒绝高基数标签。

**要点：**
- 列式 + 按时间分区 chunk。
- 内置降采样与保留。
- 基数是失败模式。
- ClickHouse 适合即席分析。

---

### 50. 全文搜索（把 Elasticsearch 当投影）

**答案：** 搜索引擎（Elasticsearch、OpenSearch、Solr、Meilisearch）提供倒排索引、分词、analyzer、faceting、打分和聚合——这些 SQL LIKE 做不到。把搜索当作投影：真相源是数据库；事件或 CDC 保持索引更新。别把 ES 当主存——它并非设计为在所有失败模式下提供强一致性或耐久性。规划重建索引（schema/analyzer 变更）、ingest 的背压和资源隔离（搜索与索引会竞争）。对中小规模，Postgres 全文或 pgvector + tsvector 往往够用。

**要点：**
- 倒排索引、analyzer、faceting。
- 当作投影，而非真相源。
- 重建索引管道是必备。
- Postgres FTS 适合中小规模。

---

### 51. 向量数据库与 RAG

**答案：** 向量数据库（Pinecone、Weaviate、Qdrant、pgvector、Milvus）存储高维嵌入并回答近似最近邻查询（HNSW、IVF）。用于 RAG：分块文档、嵌入、存储；查询时嵌入查询，检索 top-K 相关块，塞入 LLM 提示。架构关切：分块策略、嵌入模型版本化（变更要重嵌入）、混合搜索（向量 + 关键词 BM25）、用交叉编码器重排、新鲜度（文档更新时重嵌入），以及访问控制过滤。多数应用 pgvector 嵌入 Postgres 避免新增存储；在极高规模或极低延迟时专用数据库胜出。

**要点：**
- ANN 索引：HNSW 常见。
- RAG = 检索 top-K + LLM 提示。
- 混合（向量 + BM25）优于纯向量。
- pgvector 适合中小规模。

---

### 52. 多租户隔离策略

**答案：** 三种模式：silo（每租户一库/一集群——隔离最强、成本最高、合规最易）、pool（共享库，每行都带 tenant_id——最便宜、隔离最难、有邻居噪音风险）、bridge（共享基础设施，每租户一个 schema——Postgres 里的中间地带）。按租户规模与合规选择：企业租户常要求 silo；SMB/自助租户活在 pool 中并加配额。许多系统用分层：免费/小用 pool，付费用 bridge，企业用 silo。关键：每个查询带 tenant_id（中间件或 RLS 强制）、按租户限流、按租户可观测性。

**要点：**
- silo、pool、bridge：成本 vs 隔离。
- 分层模型：自助用 pool，企业用 silo。
- 每行 tenant_id + 中间件强制。
- 每租户配额防邻居噪音。

---

### 53. 租户分区

**答案：** 在 pool 多租户内，把租户路由到分片以防止邻居噪音级联。策略：hash(tenant_id) 均匀分布、查找表灵活（把大租户搬到独立分片），或混合（多数租户哈希、Top-N 钉住）。规划租户迁移：批量导出、双写窗口、切换。每分片容量上限防止单租户填满分片。按租户按分片的可观测性能尽早发现热租户。结合按租户限流和断路器，分区是抵御多租户过载的主要手段。

**要点：**
- 哈希、查找或混合分片路由。
- 把大租户钉到独立分片。
- 提前规划租户迁移。
- 每租户可观测性是必备。

---

### 54. 地理分布的数据

**答案：** 把数据放在用户附近以获得延迟和满足驻留法律。模式：读本地/写全局（写入回到主区域，读在本地服务——简单，写延迟代价）、本区域（每用户钉在一个区域，复制到其他区域做 DR），或带 CRDT/冲突解决的 active-active 多主（Cassandra、Spanner、CockroachDB）——最难。Spanner 用 TrueTime 实现全球强一致；Cosmos DB 提供可调一致性。权衡：一致性、写延迟、复杂度 vs 本地化收益。从本区域起步，按延迟或合规需要加复杂度。

**要点：**
- 每用户钉本区域是常见默认。
- Spanner/CockroachDB 用于全球 SQL。
- Cassandra/Cosmos 用于可调 AP。
- 驻留规则把数据钉到司法管辖区。

---

### 55. GDPR / 数据驻留

**答案：** GDPR 规定处理需有合法依据，赋予访问、更正、删除（被遗忘权）和数据可移植权；欧盟/中国/俄罗斯/印度的数据驻留规则要求数据留在区内。架构影响：按区域存储（或假名化）、按驻留区分别的 prod 环境、细粒度数据血缘、按租户密钥的静态加密（可实现选择性删除的加密粉碎）、事件溯源系统需要粉碎策略（墓碑不满足 GDPR——要删除或加密粉碎）。Ingest 时 PII 分类、数据目录、DPIA 流程对合规不可妥协。

**要点：**
- 按区域存储以满足驻留。
- 事件溯源用加密粉碎实现删除。
- 按租户密钥支持选择性删除。
- PII 分类与血缘必备。

---

### 56. 认证 vs 授权架构

**答案：** 认证回答"你是谁"（登录、会话、MFA、无密码）；授权回答"你能做什么"（RBAC/ABAC、策略评估、资源范围）。要分开。认证通常集中在身份提供方（Auth0、Okta、Cognito、自研）；授权可以集中（OPA、Cedar、AuthZed/SpiceDB）或分散（每服务策略带共享库）。Token 承载身份（JWT/OIDC）有时带粗粒度角色；细粒度授权通常在资源层用策略引擎评估。审计每一次授权决策的主体、动作、资源、决策和原因。

**要点：**
- 认证 = 身份；授权 = 权限。
- 集中 IdP 做认证；引擎做授权。
- JWT 承载身份，不是细粒度权限。
- 带完整上下文审计决策。

---

### 57. OAuth 2.0 / OIDC 流程

**答案：** OAuth 2.0 委派授权（用于 API 的 access token）；OIDC 加上认证（ID token）。流程：Authorization Code + PKCE（Web/移动/SPA——现代默认）、Client Credentials（服务对服务）、Device Code（电视、CLI）。Implicit 和 Resource Owner Password 已弃用。Token：access token（短时，发给 API）、refresh token（长时、轮换）、ID token（身份声明）。公共客户端必须用 PKCE；用 HTTPS；校验 `iss`、`aud`、`exp`、`nonce`。安全存储 refresh token（HTTP-only cookie 或安全存储）、使用时轮换。

**要点：**
- Authorization Code + PKCE 是默认。
- Access token 短，refresh token 轮换。
- OIDC 在 OAuth 之上加 ID token。
- 总是校验 iss/aud/exp/nonce。

---

### 58. Token：不透明 vs JWT；撤销

**答案：** 不透明 token 是随机字符串；签发方持有状态，可通过查找即时撤销——简单安全但每次 API 调用都需要内省。JWT 自包含（签名声明），本地校验无需往返——快且无状态，但过期前难撤销。缓解：短 TTL（5-15 分钟）+ refresh token、强制撤销的 jti 黑名单、密钥轮换、aud 限定。对 B2B/内部，不透明 + 缓存通常更简单。对高流量公共 API，JWT 是性能赢家。混合：快速路径用 JWT + 敏感操作查黑名单。

**要点：**
- JWT：无状态、快、难撤销。
- 不透明：有状态、易撤销、需查找。
- 短 TTL + refresh 缓解 JWT 撤销。
- 黑名单用于紧急撤销。

---

### 59. 零信任网络

**答案：** "永不信任，始终校验"：不基于网络位置的隐式信任（没有平坦的内网）。每次请求——即使是东西向——都做认证、授权和加密。通过服务身份（SPIFFE/SVID、workload identity）、服务间 mTLS、面向用户的身份感知代理（BeyondCorp）、短时凭证（Vault、IAM 角色）和每请求策略（OPA、网格策略）实现。在许多用例中取代 VPN。代价：大量平台投入、复杂度、额外检查带来的延迟。收益：泄露的爆炸半径骤减；内部威胁和横向移动变得更难。

**要点：**
- 网络无隐式信任。
- mTLS + 服务身份 + 每请求策略。
- 取代 VPN（BeyondCorp 模型）。
- 平台投入巨大，安全收益大。

---

### 60. 网格中的 mTLS

**答案：** 双向 TLS 用证书认证客户端与服务端——每次服务间调用都加密学地证明身份。服务网格（Istio、Linkerd）自动化此过程：sidecar 从内部 CA 轮换短时证书（每 24 小时或更短），应用不需要 TLS 代码。在集群内实现零信任、加密所有东西向流量，并为授权提供强身份。代价：CA 管理、握手有少量延迟（用 session resumption 缓解）、调试"为什么被拒"需要良好的策略工具。在多服务的成熟平台里视为基本配置。

**要点：**
- 双方都用证书认证。
- Sidecar 透明地处理轮换。
- 零信任 + 授权的基础。
- 内部 CA 是关键。

---

### 61. 大规模密钥管理

**答案：** 永远不要把密钥写入代码、镜像或提交到仓库的环境变量。集中到密钥管理器（Vault、AWS Secrets Manager、GCP Secret Manager）；启动或运行时获取；自动轮换；审计访问。使用 workload identity（IAM 角色、SPIFFE、Vault 认证方法）让应用无需引导密钥即可向密钥库认证——避免"乌龟驮乌龟"问题。Sidecar 或 CSI 驱动把密钥注入 pod。短时动态密钥（按会话生成的数据库凭证）优于长时静态密钥。在 CI 中扫描仓库和镜像里泄露的密钥。

**要点：**
- 集中存储，代码无密钥。
- Workload identity 优于引导密钥。
- 尽量用短时动态密钥。
- CI 扫描防意外泄露。

---

### 62. 纵深防御

**答案：** 不依赖任何单一控制。层次：网络（防火墙、分段、mTLS）、身份（MFA、SSO、最小权限）、应用（输入校验、输出编码、参数化查询、依赖扫描）、数据（静态与传输加密、tokenization、密钥管理）、监控（审计日志、异常检测、SIEM）、流程（代码评审、威胁建模、事件响应）。每一层假设其他层可能失败。对新特性做威胁建模（STRIDE）并假设有内部威胁。及时打补丁、持续扫描、演练事件响应——各层必须真正工作，而不仅仅存在。

**要点：**
- 假设任意单点控制失效。
- 分层网络、身份、应用、数据、监控。
- 对新特性做威胁建模。
- 演练事件响应；别只写 runbook。

---

### 63. 限流与滥用检测架构

**答案：** 多层：边缘（CDN/WAF 做粗 IP 级限制与 DDoS）、API 网关（按 API key/用户 RPS，用令牌桶或滑窗）、按服务（Redis 支撑的分布式计数器，或单实例时的进程内）。按用户层（匿名、免费、付费）和按端点（登录 vs 只读）使用不同限制。通过异常信号（突发 RPS、登录失败、撞库模式）检测滥用，输入到行为系统下发挑战（CAPTCHA）、阶梯认证或临时封禁。始终返回 429 + Retry-After 并公开限制。

**要点：**
- 分层：边缘、网关、服务。
- 令牌桶或滑窗 + Redis。
- 按用户类别与端点分级。
- 配合异常检测与挑战。

---

### 64. WAF 放在哪里

**答案：** Web 应用防火墙检查 HTTP 流量中的 OWASP 类攻击（SQLi、XSS、RCE、路径穿越）。放在边缘（Cloudflare、AWS WAF、Akamai），让流量在到达你的基础设施之前就被便宜地吸收 bot 和已知漏洞模式。权衡：误报会阻断合法流量，必须按应用调优，托管规则相对新型攻击滞后。配合限流、bot 管理、运行时应用保护（RASP）实现纵深防御。WAF 必要但不充分——安全编码与依赖卫生才是真正防线。

**要点：**
- 放在边缘（CDN/WAF 服务）。
- 阻断 OWASP 类攻击。
- 调优以避免误报。
- 不能替代安全代码。

---

### 65. RBAC vs ABAC

**答案：** RBAC：把用户分配到角色，角色对资源类型授予权限。简单、易审计，但当权限随资源不同（每文档、每租户）变化时会爆炸——最终上千个角色。ABAC：策略评估主体、资源、动作和环境的属性（department=engineering AND resource.owner=subject.id）。灵活、能扩展到细粒度策略，但更难推理与审计。真实系统会混用：用角色做粗粒度访问，用属性做资源范围。现代引擎（OPA、Cedar、SpiceDB/Zanzibar）两者都支持；基于关系的（ReBAC）优雅处理"分享给我"。

**要点：**
- RBAC：角色、简单、粒度细时扩展差。
- ABAC：基于属性策略、灵活、复杂。
- ReBAC（Zanzibar）适合关系分享。
- 多数系统混用 RBAC + ABAC。

---

### 66. 十二要素应用

**答案：** 云原生应用的原则集合：I 一个代码库放在版本控制，II 显式依赖，III 配置在环境变量里，IV 把后端服务当作附加资源，V 严格分离构建/发布/运行，VI 无状态进程，VII 通过端口绑定导出服务，VIII 通过进程模型扩展，IX 快速启动/优雅关停，X 开发与生产对等，XI 把日志当事件流写到 stdout，XII 后台任务作为一次性进程。仍是容器/PaaS 友好应用的基线。某些要素演化了（配置从环境变量 + 密钥管理器，日志通过 sidecar），但核心理念依然成立。

**要点：**
- 配置在环境变量，密钥通过管理器。
- 无状态进程，按数量扩展。
- 日志输出到 stdout；收集器负责路由。
- 严格分离构建/发布/运行。

### 67. Blue/Green vs Canary vs 滚动

**答案：** 滚动：每次替换 N 个 pod——简单、不需要额外容量，但回滚慢（每次只回滚 N 个）且坏版本立刻触达部分用户。Blue/Green：把 v2 与 v1 并行部署，原子切换流量，保留 v1 以快速回滚——短期需要 2 倍容量，切换干净。Canary：把一小比例流量路由到 v2，观察 SLO，逐步放大——以最小爆炸半径捕获问题，需要良好可观测性和自动化分析（Flagger、Argo Rollouts）。多数团队对风险变更用 canary，对常规部署用滚动。数据库耦合或有状态升级用 blue/green。

**要点：**
- 滚动：便宜，回滚慢。
- Blue/green：瞬时回滚，2 倍容量。
- Canary：渐进，依赖可观测性。
- 按风险等级混用。

---

### 68. 特性开关解耦部署与发布

**答案：** 发布隐藏代码，稍后通过运行时开关（LaunchDarkly、Unleash、Flagsmith、自研）按用户/群组启用。把部署（低风险、频繁）与发布（业务决策）解耦。支持按用户分群的 canary 发布、A/B 测试、事件中的 kill switch，以及无长期分支的主干开发。代价：开关债（清理旧开关！）、条件逻辑爆炸、所有组合无法测试。纪律：每个开关都有 owner、过期日期与移除任务。用开关管理在途特性与运维切换，不要用于永久配置。

**要点：**
- 把部署与发布解耦。
- 支持定向 canary、A/B、kill switch。
- 开关债真实存在——过期并移除。
- 不是配置管理的替代。

### 69. CD 中的数据库迁移

**答案：** Schema 变更必须向后兼容以允许滚动部署。模式：expand-migrate-contract（扩展-迁移-收缩）。扩展：增加新列/表（可加性，两版本都能跑）。迁移应用代码使用新 schema，必要时双写，回填数据。收缩：在所有实例都升到新版本后移除旧列/表。在大表上避免阻塞 DDL（用 pt-online-schema-change、gh-ost 或原生 online DDL）。迁移作为 CI 单独步骤运行在应用部署前，绝不要在应用启动时跑（竞态、多实例）。在 staging 用 prod 形状数据预演。

**要点：**
- expand-migrate-contract 保证向后兼容。
- 大规模使用在线 schema 变更工具。
- 作为独立 CI 步骤运行，而非应用启动时。
- 用真实量级数据预演。

### 70. liveness/readiness/startup 探针

**答案：** Kubernetes 使用三种探针。liveness：进程是否活着？liveness 失败重启 pod——保持低成本（进程检查、简单心跳）；不要因下游问题失败，否则会级联。readiness：pod 能服务流量吗？readiness 失败从 LB 摘除——检查阻塞请求的依赖（数据库连接、预热缓存）。startup：对慢启动应用，在应用就绪前抑制 liveness 检查，避免启动期间被杀。配错的探针是可避免事故的头号原因——pod 反复重启或当 readiness 过严时级联失败。

**要点：**
- liveness：进程活着；便宜；不要级联。
- readiness：能服务；查依赖。
- startup：保护慢启动。
- 探针配错引发事故。

### 71. 优雅关停与连接排空

**答案：** 收到 SIGTERM：标记为未就绪（LB 停止发新请求），用宽限期排空在途请求，关闭空闲连接，flush 缓冲（日志、指标、事件），然后退出。Kubernetes 发送 SIGTERM 后等待 `terminationGracePeriodSeconds`（默认 30 秒）再 SIGKILL——按工作负载调整。配合 pre-stop 钩子让 LB 有时间注意到未就绪状态（5-10 秒睡眠）。对有状态服务，退出前还要重新分配分区/领导权。没有优雅关停的话，每次部署都会丢一小部分请求——表现为发布期间的错误率升高。

**要点：**
- SIGTERM、排空、再退出避免 SIGKILL。
- Pre-stop 钩子给 LB 反应时间。
- 按工作负载调 terminationGracePeriod。
- 有状态服务必须移交领导权。

---

### 72. DR：RTO vs RPO

**答案：** RTO（恢复时间目标）：灾后多久恢复服务。RPO（恢复点目标）：可接受的数据丢失多少。驱动架构与成本：RTO=分钟/RPO=零需要同步复制的 active-active 多区域（贵）。RTO=小时/RPO=15 分钟允许带异步复制的 pilot-light 或 warm-standby（便宜）。按服务层级定义：支付系统 RTO/RPO 紧；分析仪表板能容忍数小时。定期演练 DR——没演练过的 DR 计划不工作。备份、runbook 和故障切换自动化至少每季度演练一次。

**要点：**
- RTO = 停机预算；RPO = 数据丢失预算。
- 驱动复制与成本。
- 按服务分层避免过度花费。
- 没演练 = 没 DR。

---

### 73. 备份与恢复

**答案：** 没测过恢复的备份没用。策略：定期全备（每周）、之间增量、WAL/binlog 持续归档以做时间点恢复。异地存储（不同区域）、加密、带访问控制（不可变备份击败勒索软件）。每季度用真实数据量测试恢复——测量 RTO、验证完整性。按法规需要保留（金融 7 年）。应用一致备份（静默数据库、快照）对数据库优于崩溃一致。备份元数据（schema 版本、应用版本）以便恢复到兼容运行时。

**要点：**
- 全量 + 增量 + 持续 WAL。
- 异地、加密、不可变。
- 每季度测恢复。
- 数据库要应用一致备份。

---

### 74. 混沌工程

**答案：** 在类生产或生产环境中故意注入故障（杀 pod、隔离网络、慢盘、耗 CPU），在弱点找到你之前找到它们。从小开始（staging 的 game day）、扩展到受控生产实验（canary 爆炸半径），并升级到持续混沌（Chaos Monkey 风格）。前置条件：扎实的可观测性、SLO 预算、自动回滚。工具：Chaos Mesh、Litmus、Gremlin、Pumba。文化转变比工具更重要：无指责文化、假设驱动的实验，把事后复盘当学习而非惩罚。

**要点：**
- 注入故障以发现弱点。
- 从 staging 开始，逐步到生产。
- 可观测性与回滚是前置条件。
- 文化比工具更重要。

---

### 75. 日志 vs 指标 vs 链路追踪

**答案：** 可观测性三大支柱。日志：离散事件带丰富上下文——适合调试具体事故，规模化存储贵，难聚合。指标：带标签的数值时间序列——便宜、可聚合、完美用于仪表板/告警，要求低基数。链路追踪：跨服务的请求流程，按 span 显示耗时——适合延迟分析与依赖图。三者都用：指标用于 SLO 与告警、日志用于事故调试、追踪用于理解分布式流程。OpenTelemetry 统一了埋点表面；后端不同（Datadog、Honeycomb、Grafana 栈）。

**要点：**
- 指标做告警、日志做调试、追踪做流程。
- OpenTelemetry 标准化埋点。
- 基数是指标的杀手。
- 对追踪采样；永远不要采样错误。

---

### 76. SLI / SLO / SLA 与错误预算

**答案：** SLI：可度量的指标（请求成功率、p99 延迟）。SLO：对 SLI 的内部目标（30 天 99.9% 成功）。SLA：带罚则的对外契约（99.5% 不达赔付）。错误预算 = 1 - SLO；如果达 SLO，你就有预算去做风险变更；如果耗尽，冻结风险部署直到可靠性恢复。强制开发与运维就可靠性 vs 速度展开量化对话。挑以用户为中心的 SLI（用户请求是否成功？）而非基础设施指标（CPU）。多窗口多燃烧率告警同时捕捉快慢消耗。

**要点：**
- SLI 度量、SLO 目标、SLA 契约。
- 错误预算治理部署风险。
- 用户为中心的 SLI 优于基础设施。
- 多窗口燃烧率告警。

---

### 77. on-call、runbook、无指责事后复盘

**答案：** 轮值 on-call 以传播知识与防倦怠；限制每班 pager 次数（告警疲劳降低响应）。每个告警都要有 runbook 带诊断与处置步骤——理想情况下自动化。事后复盘要无指责：聚焦系统/流程而非个人；记录时间线、根因、贡献因素、有 owner 和截止日期的行动项。跟踪行动项到完成；否则相同事故会重现。广泛分享复盘——组织在学习。演练事件指挥（一名 IC、一名通讯负责人、记录员），让真实事件中的协调成为自然反应。

**要点：**
- 每告警一份 runbook；尽可能自动化。
- 限制 pager 防疲劳。
- 复盘无指责且行动项被跟踪。
- 演练事件指挥角色。

---

### 78. 成本可观测性与单位经济

**答案：** 不按特性、按租户、按请求度量成本的团队会被云账单震惊。给一切打标签（服务、团队、环境）、用成本分摊报表、计算单位成本（每用户、每请求、每 GB 存储的金额）。FinOps 实践：按团队 showback/chargeback、定期成本评审、对花费异常告警、自动化 rightsizing 建议。架构决策有成本形态：无服务器在流量未稳前出色（稳态后容器更便宜）；多 AZ 数据传输累加；日志是隐形预算杀手。把成本视为一等非功能性需求。

**要点：**
- 一切打标签；按团队/特性可见。
- 单位成本（每用户/请求）作为 KPI。
- 对日花费异常告警。
- 日志和出口流量是隐形杀手。

---

### 79. 多区域 active-active vs active-passive

**答案：** Active-passive：一个区域服务，另一个是温备做故障切换——简单，无写冲突，但切换耗分钟、备用容量浪费。Active-active：两个区域都服务——用户延迟更低、满负载利用，但写需要冲突解决（CRDT、最后写入获胜，或写路径的区域亲和），一致性更棘手。混合：本区域路由（每用户为写钉到一个区域，到处服务本地读）——通常是甜点。真正全球的应用用 active-active；合规驱动、切换少的 DR 用 active-passive。

**要点：**
- Active-passive：简单，温备容量浪费。
- Active-active：满利用，写冲突。
- 本区域路由是常见甜点。
- 故障切换自动化必须演练。

---

### 80. 边缘计算（Workers、Lambda@Edge）

**答案：** 在 CDN PoP 运行代码（Cloudflare Workers、Lambda@Edge、Fastly Compute、Deno Deploy）以极低延迟（10-50ms vs 源站 100-300ms）。用例：A/B 测试路由、边缘鉴权、个性化、地理路由、图片变换、带自定义逻辑的 API 缓存。约束：极小运行时（V8 isolate 或 Wasm）、短 CPU 预算（50-500ms）、库受限、最终一致或无持久存储（高读复制低写吞吐的 KV）。架构模式：边缘处理薄逻辑 + 缓存；源站处理重逻辑 + 写入。非常适合全球用户的读多工作负载。

**要点：**
- 代码运行在 CDN PoP；50ms 以下延迟。
- V8 isolate/Wasm；CPU 预算短。
- 存储是最终一致 KV。
- 边缘做薄逻辑；源站做重逻辑。

### 81. 短链接（bit.ly）

**答案：** 需求：把长 URL 缩短为短码、GET 时重定向、跟踪点击、处理 1000 亿 + 链接和 10 万 QPS 读（读写比 10:1）。组件：API 服务（POST /shorten、GET /:code）、码生成器（来自分片计数器的递增 ID 做 base62，或 Snowflake 风格）、KV 存储（Redis 缓存 + Cassandra/DynamoDB 做耐久）、分析管道（点击 → Kafka → ClickHouse）。数据模型：{short_code, long_url, owner, created_at, expires_at}。扩展：CDN 缓存重定向（301 vs 302——用 302 保留分析）、按 short_code 哈希分片 KV、本地缓存最热码。权衡：基于计数器的码可预测（私链有安全风险——用随机或哈希 ID）；301 永久缓存难以撤销。

**要点：**
- 来自分片计数器或 Snowflake 的 base62。
- CDN + Redis + 耐久 KV。
- 需要点击分析则用 302。
- 本地缓存热码吸收峰值。

### 82. Pastebin

**答案：** 需求：粘贴任意文本最多约 10MB、生成短 URL、支持过期与私有/未列出/公开、语法高亮、每天约 1 万次写、10 倍读。组件：API + Web（POST /paste、GET /:id）、对象存储存粘贴内容（S3——便宜耐久，对超大粘贴支持范围读）、元数据库（Postgres 存 {id, owner, visibility, expiry, mime, size}）、公开粘贴的搜索索引（Elasticsearch）。数据模型：小元数据行 + S3 对象。扩展：CDN 缓存公开粘贴；KV 缓存热粘贴。TTL 通过 S3 生命周期策略或定时清理。权衡：把内容存数据库内简化但规模化会爆——从第一天就卸到 S3。

**要点：**
- 元数据在 SQL，内容在 S3。
- 公开读用 CDN。
- 生命周期策略处理过期。
- 客户端做语法高亮省服务端。

### 83. Twitter 时间线（写时扇出 vs 读时扇出）

**答案：** 需求：发推、按时间或排序展示关注者的主页时间线、5 亿用户、1 亿粉丝的名人。写时扇出：通过把新推推送到每个粉丝的收件箱（每用户一个 Redis 列表）预计算时间线。读快但写贵（名人一条推会更新 1 亿收件箱）。读时扇出：在读时查询关注者的推并合并——写便宜但读贵。混合：普通用户用写时扇出；名人推放在独立存储，读时合并。组件：Tweet 服务（Postgres + Manhattan/Cassandra）、Timeline 服务（Redis 收件箱）、扇出 worker（Kafka）、排序服务（ML）、媒体服务。权衡：混合平衡写放大与读延迟；排序最终取代严格时间序。

**要点：**
- 混合扇出：普通写、名人读。
- 每用户 Redis 列表作收件箱。
- Kafka 把发推与扇出解耦。
- 在时间线合并之上加排序模型。

### 84. Instagram

**答案：** 需求：照片上传、动态（关注者的帖子）、Explore、Stories、约 20 亿用户。组件：上传服务（缩放 + 转码到多种尺寸、存到 S3 加 CDN）、元数据库（Cassandra 存帖子，按 user_id 分片）、Feed 服务（类似 Twitter 的混合扇出）、Search/Explore（Elasticsearch + ML 排序）、Stories（独立存储 24 小时 TTL）。数据模型：post {id, user_id, media_urls, caption, created_at}；关系存图中或反规范化表。扩展：CDN 吸收照片读（主要成本）、预生成缩略图尺寸、地理分布存储。权衡：ML 排序的 feed 取代时序；点赞用分片计数器避免热 key 的最终一致计数。

**要点：**
- 图片转码管道 + CDN。
- Cassandra 应对写多的帖子元数据。
- 混合时间线扇出。

### 85. WhatsApp / 聊天

**答案：** 需求：1:1 和群聊、在线状态、送达 + 已读回执、E2E 加密、离线消息排队、每天 1000 亿条消息。组件：连接服务（长连 WebSocket，单节点百万连接靠 Erlang 风格并发）、消息服务（按会话日志存 Cassandra 或自研存储）、状态服务（带 TTL 的 Redis）、推送通知（APNs/FCM）做离线送达、媒体服务（S3 + CDN，E2E 加密 blob）。路由：每用户按一致性哈希钉到一个连接节点；接收方查询 → 转发到其节点。E2E：Signal 协议（X3DH + Double Ratchet）——服务器只存密文。扩展：按用户分片、地理路由到最近 PoP、群组异步扇出。

**要点：**
- 每用户 WebSocket；一致性哈希到节点。
- Signal 协议做 E2E（X3DH + Double Ratchet）。
- 按会话日志存 Cassandra。
- APNs/FCM 做离线送达。

### 86. 通知系统

**要点：**
- 渠道网关后接队列。

### 87. YouTube

### 93. 分布式限流器

**答案：** 需求：在 N 个全球边缘节点上对每用户/每 key RPS 限制、低开销、分区时接受轻微超限。算法：令牌桶（允许爆发到桶大小、按速率补充）、滑窗日志（精确但内存重）、滑窗计数器（近似、便宜）。实现：Redis 用 INCR + EXPIRE（集中、简单但加延迟）；本地内存定期同步（最终、可能超）；Redis Lua 脚本做原子。全局限制用中心 Redis 集群或分层（本地软上限 + 中心硬上限）。权衡：精度 vs 延迟；允许小超限以低延迟。CRDT（PN-counter）适用于跨区域最终一致计数。

**要点：**
- 令牌桶是实用默认。
- Redis + Lua 做原子 check-and-decrement。
- 本地软 + 全局硬以低延迟。
- 允许小超限省往返。

### 94. 分布式缓存（类 Redis）

**答案：** 需求：低延迟 KV 存储、水平扩展、可选复制、淘汰策略、每秒数十亿 op。组件：分片（一致性哈希或分区槽——Redis Cluster 用 16384 个槽）、复制（每分片主 + 副本，异步或半同步）、客户端（槽感知、流水线化）、淘汰（LRU、LFU、allkeys-random、TTL）。故障：gossip 协议检测死节点，副本自动晋升。持久化：AOF（追加文件）或快照（RDB）。扩展：在线加分片；客户端跟随 MOVED/ASK 重定向。权衡：复制异步（故障切换可能小数据丢失）；大 key 引起热分片（拆或客户端分片）；流水线 + 连接池对吞吐至关重要。

**要点：**
- 一致性哈希或固定槽数。
- 异步复制；可能小数据丢失。
- Gossip + 自动晋升做 HA。
- 按工作负载选淘汰策略。

### 95. Typeahead / 自动补全

**答案：** 需求：随键入提供 100ms 以下建议、个性化 + 流行查询、多语言、数十亿查询。组件：流行查询的 Trie 或 FST（带分数，离线由查询日志构建，每小时/每天刷新）；内存放在建议节点上以高速。查询服务：每次按键触发轻量请求，返回按频率 + 新鲜度 + 个性化排序的 top-K 补全。个性化：把全局建议与用户历史混合（浏览器缓存或每用户索引）。扩展：按前缀分片或只读复制建议索引；CDN 缓存常见前缀；客户端 debounce（200ms）。权衡：新鲜度 vs 索引构建成本；流行度偏置压制长尾查询——从实时流混入新鲜查询。

**要点：**
- Trie/FST 离线由日志构建。
- 分片或只读复制索引。

### 99. 支付系统（幂等、账本、对账）

**答案：** 需求：在重试与部分失败下正确处理支付、绝不重复扣款、精确记账、审计轨迹、合规（PCI）。组件：支付 API（每个请求带幂等键——与响应一起存储，重试返回相同答案）、账本服务（复式记账：每笔事务有相等借贷，仅追加）、提供方适配器（Stripe、Adyen 等）带断路器和重试、对账作业（每日比对账本与提供方流水、标记差异）、Webhook 处理器（提供方通知——必须幂等）、KYC/AML、结算。数据模型：账户 + 条目（借贷对）。扩展：按账户分片账本；不可变条目让分片简单。权衡：同步确认简单但更慢；异步加状态轮询更可扩展。永远不要让重试导致重复扣款——幂等键是最重要的设计元素。
