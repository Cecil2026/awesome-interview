# 模拟面试：设计聊天应用（WhatsApp / Slack）

**类型：** 系统设计
**难度：** L5
**时长：** 60 分钟（模拟）
**标签：** #system-design, #realtime, #websockets, #message-queues, #fan-out, #presence

## 背景

候选人正在面试一家与即时通讯相关的公司（想想 Slack、Discord，或者自建聊天的金融科技公司）的 Staff 方向 L5 后端岗。面试官是一位 Principal 工程师，曾交付过大规模的消息系统。题目在这一轮开始时公布。候选人在一块虚拟白板上，面试官通过视频接入；面试官对一致性模型有强烈的看法，会专门考察候选人对消息排序的理解。

## 对话记录

**面试官：** 今天我希望你设计一个聊天应用 —— 想想 WhatsApp 或 Slack。慢慢来，需要问什么尽管问，哪里有意思我就会深挖。

**候选人：** 好。我先从范围开始 —— 这里覆盖面很广。先问几个澄清问题。我们做一对一聊天、群聊，还是两者都做？要不要语音和视频通话？只做消息？客户端是移动端、桌面端、Web 端，还是全部？

**面试官：** 所有客户端 —— 移动、桌面、Web。只做消息 —— 这一轮不做语音或视频。一对一和群聊都做。群可以小可以大；最大的可以到 10 万成员。

**候选人：** 好，这是一个很重要的约束 —— 大群会显著改变 fan-out 的玩法。消息相关的功能呢？输入指示、已读回执、在线状态、消息编辑、话题（threading）、文件附件、搜索？

**面试官：** 都在范围内，但优先级：消息送达必须有，在线状态和已读回执很重要，输入指示和编辑是 nice-to-have，附件和搜索可以勾勒。Threading 跳过。

**候选人：** 端到端加密呢？

**面试官：** 范围外。假设只有传输层安全。如果有时间最后可以聊聊。

**候选人：** 规模数据 —— 多少用户，每秒多少消息？

**面试官：** 5 亿 MAU，1 亿 DAU。每个用户每天发送的消息数你自己假设。

**候选人：** *（思考）* 活跃用户平均每天发送约 50 条消息。所以 1 亿 × 50 = 每天 50 亿条，平均每秒约 6 万条。乘上 5 倍峰值因子 —— 主要时区的晚上 —— 峰值算每秒 30 万写入。

**面试官：** 读取呢？

**候选人：** 读取更复杂。每条发送的消息都会被接收方读到 —— 1:1 里就是每写 1 读，群里则是每写 N 读，N 是群大小。大部分群很小；长尾里有少数超大群。假设平均读放大 3 倍 —— 即峰值每秒 90 万读，但热点处速率会高得多。

**面试官：** 区分一下"读"是指送达，还是显示。

**候选人：** 这是个好区分。送达意味着消息落到接收方的设备上。显示意味着用户实际看了它。送达量大约是写入的 3 倍 —— 每个活跃对话都会被推送。显示量更低，因为不是每个接收方都会立刻打开每条消息。还有一种是历史拉取 —— 向上滚动读旧消息 —— 对单个用户更尖峰，但总体量更低。

**面试官：** 存储如何？

**候选人：** 每天 50 亿条消息，算上元数据平均约 200 字节。每天 1 TB，每年 365 TB。再加上无限期保留的历史和副本，几年内就是 PB 级。

**面试官：** 延迟目标？

**候选人：** 消息送达 —— 从发送方按发送到接收方看到 —— 我会把目标设在跨区域 p50 低于 200ms、p99 低于 1 秒。输入指示和在线状态可以宽松一些 —— 几秒可接受。已读回执：低于 1 秒。

**面试官：** 好。你不为什么做设计？

**候选人：** 我不为语音/视频、端到端加密、跟其他聊天网络的联邦、单条消息的细粒度权限做设计。我还假设所有客户端都信任服务器 —— 没有 MLS 风格的群加密。

**面试官：** 好。看 API 吧。

**候选人：** *（写）*

```
WebSocket 会话：
  CONNECT /ws  （带 auth token）
  -> 服务端升级到 WS，注册会话

Client -> Server:
  SEND_MESSAGE { conversation_id, client_msg_id, body, type }
  MARK_READ { conversation_id, last_read_msg_id }
  TYPING { conversation_id, started: bool }
  PRESENCE { status: online | away | offline }

Server -> Client:
  MESSAGE { msg_id, conversation_id, sender_id, body, ts }
  READ_RECEIPT { conversation_id, user_id, last_read_msg_id }
  TYPING { conversation_id, user_id, started }
  PRESENCE { user_id, status }

REST（用于拉取和管理）：
  GET /conversations
  GET /conversations/{id}/messages?before=...&limit=...
  POST /conversations （创建）
  POST /conversations/{id}/members （添加）
  DELETE /conversations/{id}/members/{user_id}
```

**面试官：** 为什么用 WebSocket 而不是 long-polling 或 SSE？

**候选人：** 双向是胜出之处。用 SSE 还得另开一个 HTTP 通道做 client-to-server 的发送，连接开销更大，排序也会变复杂。Long polling 能用，但开销更高、延迟更差。WebSocket 是这种负载的标准，现代基础设施 —— 包括大多数 LB 和 CDN —— 都支持。

**面试官：** HTTP/3 server push 或 WebTransport 呢？

**候选人：** WebTransport 长期看更干净，也避免了单 TCP 连接上的队头阻塞，但采纳率和基础设施支持还在成熟中。我会从 WebSocket 开始，保留以后切换到 WebTransport 的路径。我们会把传输层抽象出来，让替换可控。

**面试官：** 讲讲 `client_msg_id`。为什么它在 SEND 请求里？

**候选人：** 幂等性。客户端发送前为每条消息生成一个 UUID。如果发送被重试 —— 网络抖动、客户端重连 —— 服务端用 client_msg_id 去重。同样的 UUID 对应同一条逻辑消息，不管传输多少次。

**面试官：** 服务端会返回自己的 msg_id？

**候选人：** 是的。服务端分配一个全局或每对话内的 msg_id，用于排序、历史拉取和引用。client_msg_id 是去重 key；server msg_id 是从此以后的规范标识符。服务端对 SEND_MESSAGE 的回复同时包含两者，让客户端对账。

**面试官：** 好。看数据模型。

**候选人：** *（写）*

```
users (
  user_id BIGINT PK,
  name VARCHAR,
  avatar_url TEXT,
  created_at TIMESTAMP,
  ...
)

conversations (
  conv_id BIGINT PK,
  type ENUM('direct', 'group'),
  name VARCHAR,
  created_at TIMESTAMP,
  ...
)

conversation_members (
  conv_id BIGINT,
  user_id BIGINT,
  joined_at TIMESTAMP,
  last_read_msg_id BIGINT,
  notification_settings JSONB,
  PRIMARY KEY (conv_id, user_id)
)

messages (
  conv_id BIGINT,
  msg_id BIGINT,           -- 在 conv 内单调递增
  sender_id BIGINT,
  body TEXT,
  type ENUM('text','image','file','system'),
  client_msg_id UUID,
  created_at TIMESTAMP,
  edited_at TIMESTAMP NULL,
  PRIMARY KEY (conv_id, msg_id)
)
```

**面试官：** 为什么 msg_id 是按对话而不是全局？

**候选人：** 两个原因。一，消息排序是每对话内的关注点 —— 两个不相关对话之间的消息没有有意义的顺序。让 ID 按对话生成意味着排序权威是本地的，不是全局的。二，这让分片变得简单 —— 按 conv_id 分区，一个对话的所有消息都在一个分片上。

**面试官：** 取舍？

**候选人：** 没有全局的"给我消息 #12345"这种查询 —— 但我们不需要。每次引用消息都带 conv_id，所以查询永远是 (conv_id, msg_id)。

**面试官：** 用什么数据库？

**候选人：** 消息我会放在宽列存储里 —— Cassandra 或 ScyllaDB。访问模式是"取对话 X 的最近 N 条消息"或"取对话 X 时间戳 T 之前的消息"，跟 partition-key + clustering-key 模型完美契合。按 conv_id 分区，按 msg_id 降序聚簇。写吞吐量巨大，通过加节点来扩缩容。

**面试官：** 为什么不用带分片的 Postgres？

**候选人：** Postgres 能用，但 30 万写/秒下你得激进分片，而跨对话事务收益不大。Cassandra 是为这种访问模式量身打造的。取舍是一致性 —— Cassandra 默认最终一致。对聊天而言基本可以接受；消息不需要严格的全局排序。

**面试官：** 用户和对话元数据呢？

**候选人：** 那是关系型的 —— Postgres。写入量低得多，受益于事务（尤其是成员变更），数据模型自然是关系型的。

**面试官：** Postgres 的写入量是多少？

**候选人：** 远低于消息。新建对话、加/减成员、资料更新、last_read 更新。最频繁的是 last_read —— 它跟读取活动一起增长。我们会在客户端做防抖和批量，写入前再在 write buffer 里聚合。分片 Postgres 处理起来从容。

**面试官：** 每条消息的 msg_id 生成器？

**候选人：** 每对话内的单调计数器。每个对话有自己的计数器，每次发送时递增。为避免争用，可以让计数器跟着对话的分片走，由该分片串行化 SEND。或者用混合方案：Snowflake 风格的 ID，时间戳前缀给出粗略顺序，序列后缀打破平局。

**面试官：** 反驳一下"通过一个分片串行化 SEND"。一个非常活跃的对话的吞吐量呢？

**候选人：** 单个对话每秒处理数千条消息很罕见，但确实会发生 —— 比如热门的 Discord 或 Slack 频道。*（思考）* 对这些，我们接受写入串行化到一个分片，但分片每秒能处理数万次写入。我们会先在单对话吞吐量上撞瓶颈，而不是在分片上。

**面试官：** 如果单个对话超过分片容量呢？

**候选人：** 那每对话单调计数器的假设就破了。我们会把这种对话切换到放松模型：msg_id 用带时间戳前缀和节点 ID 的 Snowflake ID，排序通过 Snowflake 比较。失去严格单调性，但换来水平扩缩容。这类对话会被打标并走不同的路由 —— 走"高 fan-out"层级。

**面试官：** *（记录）* 好，画一下高层架构。

**候选人：** *（画）*

```
[Mobile/Desktop/Web Clients]
       |
       v (WebSocket)
[Edge LB / TLS Termination]
       |
       v
[WebSocket Gateway Tier] <----> [Session Registry (Redis)]
       |
       v
[Message Service]
   |        \
   |         \--> [Conversation Sharder]
   v
[Cassandra: messages]   [Postgres: users, conversations, members]
   |
   v
[Kafka: message bus]
   |
   +---> [Fan-out Service] --> [WebSocket Gateway delivery]
   +---> [Push Notification Service]
   +---> [Search Indexer (Elasticsearch)]
   +---> [Analytics Pipeline]
```

WebSocket 网关是唯一有状态的层 —— 它持有连接。其他都是无状态的，水平扩缩容。

**面试官：** 走一遍 Alice 在 50 人群里发一条消息时发生了什么。

**候选人：** 一步一步。Alice 的客户端跟网关 G1 有一条 WebSocket 连接。她敲一条消息，按发送。她的客户端通过 WebSocket 发出 SEND_MESSAGE，带 conversation_id 和 client_msg_id。

G1 收到 SEND，校验 auth token，通过 gRPC 转发给 Message Service。Message Service 查到对话归属的分片，调用每对话内的 msg_id 计数器拿到下一个 msg_id，把消息写入 Cassandra。写入成功后，回 ACK 给 G1，G1 再把 ACK 转给 Alice 的客户端。这就是发送路径 —— Alice 的 UI 现在把消息显示为"已发送"。

与此同时，Message Service 把消息发布到 Kafka，以 conv_id 为 key。Fan-out Service 从 Kafka 消费，查到对话成员 —— 50 个 user_ids —— 对每个成员去 Session Registry 查他们当前的 WebSocket 网关，然后把消息推到每个网关，网关再通过打开的 WebSocket 送达每个连接的客户端。对离线成员，通过 APNs/FCM 触发推送通知。

**面试官：** 让我反驳一下。你说 Message Service 写入 Cassandra 然后发布到 Kafka。如果 Cassandra 写入成功但 Kafka 发布失败呢？

**候选人：** *（停顿）* 那是经典的双写问题 —— 消息持久化了，但从未 fan-out。我们会想要某种 outbox 模式。

**面试官：** 多说一点。

**候选人：** 两种主要做法。一：在一次 Cassandra batch 里同时写消息和一条 outbox 条目 —— Cassandra 在 partition 内的 batch 是原子的 —— 由一个单独的进程跟踪 outbox 并发布到 Kafka。二：通过 Debezium 这类工具做 Cassandra 到 Kafka 的 change-data-capture。两者都有"Kafka 发布在持久化写入下游"这个性质。

**面试官：** 你会选哪个？

**候选人：** Outbox 在运维上更简单 —— 比 CDC 少了活动部件，尤其在 Cassandra 上 CDC 工具不如 Postgres 上成熟。取舍是写入和发布之间有一小段滞后，但对聊天而言是几十毫秒。

**面试官：** 好。现在反过来 —— 如果 Kafka 发布成功但 Cassandra 写入失败呢？

**候选人：** 用 outbox 模式这不可能发生 —— Kafka 发布是由对 outbox 的成功写入门控的，而 outbox 跟消息在同一个 partition 里。它们要么一起提交，要么一起不提交。

**面试官：** 好。我们再深入 Fan-out Service。10 万人的群。会怎么样？

**候选人：** *（思考）* 朴素的 fan-out：一条消息进来，做 10 万次成员查找，推给每个人。如果每个成员都在线，一条消息就是 10 万次 WebSocket 写入。按我们的消息速率，fan-out 服务工作量很大。我们会按某种方式分片它 —— 也许按接收方 user_id 范围 —— 让每个 fan-out 节点处理成员的一个切片。

**面试官：** 成员列表本身呢？放在哪里？

**候选人：** 真理来源在 Postgres，但激进地缓存。fan-out 服务在内存里缓存成员列表，带 TTL 和变更时的 pub/sub 失效。对一个 10 万成员的群，成员列表大概几 MB —— 装得下内存。

**面试官：** 群的上限是多少？你说 10 万。如果是 100 万呢？

**候选人：** *（思考）* 到 100 万我们就进入广播领域了 —— 更接近 Twitch 聊天而不是私人群组。每接收方模型的 fan-out 成本变得不可承受。对超大房间我们会切换到 pull-based 模型：客户端定期轮询新消息，或者通过 pub/sub 频道订阅，服务端不必知道谁订阅了。

**面试官：** Pull-based —— 取舍是什么？

**候选人：** 接收方延迟更高 —— 不是亚秒级推送，而是按轮询间隔。但服务端更省：不必为超大房间维护每接收方状态。混合方案是给前 N 个热门成员推送、给长尾拉取，或者用 fan-out 树，中间节点再进一步 fan-out。

**面试官：** 对 10 万你选哪个？

**候选人：** 推送，但带优化。具体来说：只推给当前已连接的成员。对断开的成员，写入他们的未读队列，等他们重连时拉取。这就避免了"其实只有一小部分人在线打开应用"的情况下还做 10 万次 WebSocket 推送。

**面试官：** 未读队列住在哪里？

**候选人：** 概念上它就是"这个对话、这个用户中 msg_id > last_read_msg_id 的消息"。我们不把它物化为单独的队列 —— 从 messages 表和 conversation_members.last_read_msg_id 字段派生。客户端上线重连时，查询 `GET /conversations/{id}/messages?since={last_read_msg_id}` 来追赶。

**面试官：** 派生未读队列的替代方案是什么？应该把它物化吗？

**候选人：** 可以 —— 给每个用户一个收件箱，存消息引用。WhatsApp 历史上做过类似的事。取舍：物化收件箱拉取简单，但写入昂贵（每条消息产生 N 次收件箱写入）。在我们的规模下，从对话派生更便宜 —— 消息只存一次，未读状态从成员元数据算出。

**面试官：** 成本差异是什么样的？

**候选人：** 大致和群大小成正比。对 100 人的群，物化收件箱每条消息花 100 次写入，派生只花 1 次。平均 30 万消息/秒，平均群大小约 5，物化就是 150 万收件箱写入/秒 —— 能做但昂贵。派生便宜得多，而且我们本来就要按成员维护 `last_read_msg_id`。

**面试官：** 物化有其他优势吗？

**候选人：** 有 —— 跨对话查询很快。"显示我所有对话里所有未读消息"在物化收件箱上是一次查询；在派生上是跨我所有对话的 fan-in。对"统一收件箱"UI，物化要好得多。我们可以做混合：每用户最近的前 N 个对话物化，其余派生。

**面试官：** WebSocket 网关呢？它怎么知道把消息发到哪里？

**候选人：** Session Registry。客户端连到某个网关时，网关把 `(user_id, gateway_id, session_id)` 写入 Redis。fan-out 服务要送达某个用户时，先在 Redis 里查到该用户的会话，再转发到对应的网关。

**面试官：** 如果用户有多个设备呢？

**候选人：** 每个用户多个会话。每台设备一条自己的会话条目。fan-out 送达该用户的所有会话。Session Registry 按 user_id 做键，下面挂一组会话条目。

**面试官：** 如果一条消息送达到一台设备但用户从另一台读取了它，两台设备都知道它已读吗？

**候选人：** 是的 —— 用户从设备 A 读了之后，客户端发出 MARK_READ。服务端更新 last_read_msg_id，并向同一用户的其他会话以及对话的其他参与者 fan-out 一个 READ_RECEIPT。所有设备在一秒内收敛到同一已读状态。

**面试官：** 表情反应和编辑呢 —— 它们怎么传播？

**候选人：** 走同一条 fan-out 路径。一个反应就是一个小型的类消息事件：`(msg_id, user_id, emoji, action: add | remove)`，由 fan-out 送达对话成员。编辑类似 —— `(msg_id, new_body, edited_at)`。客户端把更新应用到本地副本并重新渲染。我们持久化最新版本，可选保留编辑历史以便审计。

**面试官：** 如果会话过期了 —— 网关崩溃没有清理呢？

**候选人：** 会话条目带 TTL，加上网关定期心跳。如果某个网关停止心跳，它的会话在一分钟内从注册表过期。fan-out 尝试送达过期会话时，网关已经不在了，推送失败，fan-out 丢弃这次送达 —— 消息仍在 Cassandra 里，客户端重连时会取到。

**面试官：** *（记录）* 好，让我深入排序。你说每对话内单调 msg_id。客户端看到什么顺序？

**候选人：** 客户端在一个对话内看到消息按 msg_id 排序。服务端是排序权威 —— 它按处理 SEND 的顺序分配 msg_id。

**面试官：** 如果同一个对话里两个发送方同时发送，来自不同区域，并且消息走系统里不同的路径呢？

**候选人：** *（停顿）* 排序由哪条先被对话归属的分片处理来决定。如果 Alice 和 Bob 在"同一时刻"发送，分片按请求到达的顺序分配 msg_id，而到达顺序取决于网络路径。没有因果排序 —— 实际上是"先到分片者赢"。

**面试官：** 对聊天来说可以吗？

**候选人：** 大部分情况下可以，有一个例外。如果 Bob 的消息是对 Alice 的回复 —— "感谢链接" —— 但 Bob 因为网络更快先到达分片，他的"回复"就拿到更低的 msg_id，在 Alice 的消息之前渲染。这会让人困惑。

**面试官：** 你怎么修？

**候选人：** 向量时钟或因果元数据。客户端在自己的 SEND 里带上它见过的最后一个 msg_id。服务端收到 Bob 的 SEND 带 `last_seen_msg_id = X` 时，确保 Bob 的消息排在 X 之后。如果 Alice 的消息已经有 msg_id > X，Bob 的消息就拿到比 Alice 更大的 msg_id。

**面试官：** 在存储层这具体是什么样子？

**候选人：** Bob 的 SEND 到达，服务端看到 `last_seen = X`。服务端查到对话中最新的 msg_id —— 比如现在是 Y > X，因为 Alice 的消息先到。服务端给 Bob 的消息分配 msg_id Y+1。因果 happens-before 得到保留。

**面试官：** 如果 Bob 的 `last_seen = X` 其实相当陈旧 —— 他很久没同步了 —— 而且有 1000 条更新的消息呢？

**候选人：** 约束仍然满足：Bob 的消息拿到 msg_id = current_max + 1 = 1000 + X + 1。我们没有延迟它 —— 只是确保它晚于他看到的。这没问题。

**面试官：** 跨对话的全序呢？这重要吗？

**候选人：** 不太重要。用户 UI 按最后消息时间戳顺序显示对话，是宽松的。在一个对话里，我们有每对话内的排序。跨对话，基于时间的排序就够好了，UI 用途上几秒的时钟偏差可接受。

**面试官：** 好。讲讲送达保证。你提供什么保证？

**候选人：** 对每个接收方至少一次送达，加上通过客户端去重做到的恰好一次可见送达。流程：服务端把消息提交到 Cassandra，fan-out 推到接收方网关，网关再推给客户端。如果客户端的 WebSocket 在推送途中关闭，消息在传输中丢失但仍在 Cassandra —— 重连时客户端从中断处拉取。

**面试官：** "恰好一次可见"到底是什么意思？

**候选人：** 用户看到每条消息恰好一次。系统在重连/重试中可能多次送达同一条消息，但客户端在渲染前按 server msg_id 去重。

**面试官：** 如果客户端从未看到消息呢？网络分区、应用崩溃等等。

**候选人：** 消息在 Cassandra 里是持久的。客户端重连并请求 last_read 之后的消息时，会拿到这条消息。只要服务端的"已接收"视图由持久存储门控，消息就不会被悄悄丢弃。

**面试官：** 从客户端回到服务端的 ACK 呢 —— 你跟踪送达状态吗？

**候选人：** 是的。客户端通过 WebSocket ACK 每条消息的接收。服务端按 (user_id, msg_id) 记录 "delivered_at" —— 存为 Cassandra 的稀疏列或单独的 delivery_status 表。这驱动用户看到的单勾 vs 双勾状态，类似 WhatsApp 的灰勾/蓝勾。量很大 —— 每条消息每个接收方一个 ACK —— 所以我们在客户端批量化 ACK，并以粗粒度持久化。

**面试官：** 离线到上线的转换 —— 回填排序呢？

**候选人：** 客户端请求 last_read_msg_id 之后的消息，按 msg_id 顺序拿到，按顺序应用。如果用户在很多对话里，我们会分页或排优先级：先取用户正在主动查看的对话，再回填其余的。

**面试官：** 客户端何时更新 last_read？

**候选人：** 用户实际查看消息时。客户端发出 MARK_READ，带上看过的最高 msg_id。服务端更新 `conversation_members.last_read_msg_id`，并向关心的其他参与者 fan-out 一个 READ_RECEIPT。

**面试官：** 多久持久化一次 MARK_READ？

**候选人：** 客户端防抖 —— 比如把 1-2 秒内的已读更新攒批再发。不要每条消息发一个 MARK_READ；滚动时那是浪费。服务端批量写入 Postgres，因为我们每个对话更新一行。

**面试官：** 好。讲讲在线状态。怎么知道 Alice 在线？

**候选人：** 在线状态从活跃 WebSocket 连接加上客户端发出的状态派生。Alice 的 WebSocket 连上时，网关发布一个 presence-update 事件。它断开 —— 不管是干净断开还是过期 —— 在线状态在宽限期后回退到离线。客户端也能发 PRESENCE 带显式状态 —— away、do-not-disturb。

**面试官：** 在线状态住在哪里？

**候选人：** Redis，按 user_id 做键，带 TTL。更新来自网关连接/断开事件。读取来自订阅某用户在线状态的客户端。

**面试官：** 客户端怎么订阅一个用户的在线状态？

**候选人：** 客户端打开一个对话时，隐式订阅其他成员的在线状态。服务端注册订阅，在目标用户状态变化时把更新推给订阅者。

**面试官：** 规模检查 —— 有多少订阅？

**候选人：** 每个用户对大约 100 个联系人感兴趣。1 亿 DAU 全球就是 100 亿订阅。我们不把所有这些都物化 —— 只在客户端打开相关视图时延迟订阅。活跃订阅峰值更像是 1 亿到 10 亿，分布在几千台在线状态服务器上是可管理的。

**面试官：** 10 万成员群的在线状态呢？

**候选人：** *（思考）* 那很昂贵。我们不会推送所有 10 万成员的在线状态更新。换成显示聚合统计 —— "23,000 在线" —— 定期计算。或者只对用户最近交互过的成员推送在线状态。出于纯粹规模考虑，我们会在大群里限制在线状态可见性。

**面试官：** 名册里有数千联系人的用户怎么处理？

**候选人：** 像我提到的延迟订阅，加上分页。用户的 UI 一次只显示少量联系人 —— 联系人列表顶部、当前打开的聊天。我们订阅可见联系人的在线状态，他们滚出视图或聊天关闭时取消订阅。未订阅的那批人的在线状态更新不推送；下次打开时取一个快照。

**面试官：** 快照拉取是什么样子？

**候选人：** `GET /presence?user_ids=[...]` 返回一批当前状态。Redis 查找很快 —— 100 个 key 亚毫秒。我们会在客户端缓存快照几秒，避免快速 UI 滚动时反复打。

**面试官：** 输入指示？

**候选人：** 思路一样，但是瞬态的。用户开始输入时客户端发 TYPING(started=true)，停止或超时后发 TYPING(started=false)。服务端 fan-out 到跟接收消息一样的受众。不做持久化 —— 输入事件不进 Cassandra。它们甚至可能不走 Kafka —— 直接通过 pub/sub 频道从网关到网关，获得超低延迟。

**面试官：** 输入指示的带宽成本是多少？

**候选人：** 朴素地做的话很大。如果 10 万人群里每个人都被告知谁在输入，fan-out 会大得离谱。我们对输入指示 fan-out 设上限：只在 1:1 和小群显示，在大房间隐藏。

**面试官：** 阈值是多少？

**候选人：** 凭启发式。也许在 20-50 人的群里显示。超过这个数，输入指示就变成噪音了 —— 多人同时输入，UI 反正也表示不出来。我们会用产品分析来验证；这跟工程一样是 UX 决定。

**面试官：** *（看时间）* 好。推送通知。讲讲它们怎么工作。

**候选人：** 服务端：一个 Push Notification Service 从 Kafka 消费 —— 跟 fan-out 同一个 topic，或者过滤了"任一接收方离线"消息的派生 topic。对每个离线接收方，它构造推送载荷，通过 APNs（iOS）或 FCM（Android）派发。客户端再通过 OS 接收推送，OS 唤醒应用或显示通知。

**面试官：** 怎么知道谁离线？

**候选人：** 跟 Session Registry 交叉比对。如果用户没有活跃会话，就是离线。如果所有会话都是后台/空闲，我们也会发推送，因为 WebSocket 可能开着但用户没在看。

**面试官：** 如果用户在桌面在线但移动离线呢？

**候选人：** 推送到移动，抑制（或静默发送）到 WebSocket 已经送达的桌面。按设备的会话跟踪让我们正确路由这个。

**面试官：** 如果推送失败 —— token 过期、APNs 限流等等？

**候选人：** 对瞬态错误带退避重试。token 过期就在用户资料里标记 token 无效并停止发送。被限流就排队并放慢。我们也有另一条路径：用户重新打开应用时，即使没收到推送也会从服务端拉未读消息。

**面试官：** 通知批量化呢？如果我离开时收到 50 条消息，我会得到 50 个推送吗？

**候选人：** 不，那是糟糕的 UX。我们在服务端做批量化：按接收方、按对话，带一个短的合并窗口。如果我们发了一个推送，5 秒内同一对话又来一条消息，就更新原推送的载荷而不是再发一个新的。APNs 和 FCM 都通过 collapse key 支持载荷替换。用户看到"Alice 给你发了 3 条消息"，而不是三条独立的横幅。

**面试官：** iOS 上的角标计数呢？

**候选人：** 服务端按用户维护。每个推送都带上当前未读计数。用户读了消息后客户端发 MARK_READ；服务端更新未读计数，并推送一个静默的仅角标更新，让其他设备上的角标也清零。

**面试官：** 勿扰 / 安静时间呢？

**候选人：** 用户偏好存在 Postgres。Push Notification Service 派发前先检查。安静时段我们要么完全丢推送，要么静默发送（无横幅但消息仍送达），要么攒成摘要等安静期结束再发。由用户偏好决定。

**面试官：** 切到媒体。文件附件怎么工作？

**候选人：** 带外。客户端通过预签名 URL 把文件上传到 blob 存储 —— S3 或同类。blob 存储返回一个内容 URL。客户端再发一个 MESSAGE，`type=image`（或 file 等），URL 放在 body 或 attachment 字段。接收方看到 URL，直接从 blob 存储拉内容。

**面试官：** 为什么带外？

**候选人：** 两个原因。一，消息通道是为小且频繁的消息优化的，不是数 MB 的 blob。通过消息总线发 50MB 视频会冲击 Kafka 和消息存储层。二，blob 存储有我们想要的特性 —— 内容寻址存储、CDN 集成、生命周期策略 —— 我们不想重新实现。

**面试官：** 缩略图或转码版本呢？

**候选人：** 单独的 Media Processing Service。上传时触发 —— S3 事件通知，或 blob 上传发布的 Kafka topic。这个服务生成缩略图、转码视频、提取 EXIF，再用派生 URL 更新消息元数据。接收方先看缩略图，按需加载完整文件。

**面试官：** 如果上传在消息发送之后才完成呢？

**候选人：** 顺序很重要。客户端先上传，拿到 URL，再发引用该 URL 的消息。上传失败，消息就不发了。上传成功但发送失败，blob 成了孤儿，由生命周期策略回收。

**面试官：** 你能在上传完成之前乐观地显示消息吗？

**候选人：** 可以 —— 客户端显示一个"上传中"占位符，带本地数据。上传完成且 SEND 成功后，占位符替换为真实消息引用。其他接收方在 SEND 落地前看不到任何东西。这是带媒体的聊天应用的标准 UX。

**面试官：** 媒体访问控制呢？非成员能拉取 blob 吗？

**候选人：** 没有 ACL 的话，预签名 URL 来门控访问。但预签名 URL 是 bearer token —— 任何拿到 URL 的人都能拉。对敏感内容，用短期预签名 URL（1 小时 TTL），认证成员每次访问时重签。更好：媒体代理服务在提供 blob 前检查授权。更贵但更私密。

**面试官：** 搜索？

**候选人：** 异步索引。Search Indexer 从 Kafka 消费并写入 Elasticsearch —— 按用户还是按对话建索引取决于访问模式。出于隐私考虑，索引应按用户范围：用户的搜索索引只包含他是成员的对话。我们要处理加/减成员 —— 用户加入一个对话时，会回填可搜索内容吗？通常不会；只有新消息可搜索。

**面试官：** 为什么按用户而不是按对话？

**候选人：** 两个原因。一，安全裁剪 —— 搜索必须只返回用户被允许看到的消息；按用户索引在构造上就保证了这点。二，查询性能 —— 用户的搜索只打一个索引，不必跨多个对话做 fan-in。缺点是索引重复：50 人群里的一条消息会存在 50 个索引里。我们接受存储成本以换访问模式上的胜出。

**面试官：** 你可以做按对话索引并在查询时做安全裁剪吗？

**候选人：** 可以 —— 每对话一个索引，查询时连接成员信息做过滤。存储更少、查询更复杂，对有大量对话的用户查询延迟也更差。我们会基于读模式来选。对消费者聊天，大多数用户很少搜索，按对话可能就够。对 Slack 风格的工作场所搜索这种主要功能，按用户值得。

**面试官：** 索引大小是多少？

**候选人：** 每天 1 TB 消息数据，索引大小类似 —— 经过压缩和词项索引后可能更小。我们会按 user_id 或 conversation_id 分片索引，并用 Elasticsearch 内置的分片。

**面试官：** 搜索延迟是多少？

**候选人：** 典型查询 p99 低于 1 秒。Elasticsearch 对关键词查询很快；全文相关性排名增加一些成本。

**面试官：** 按什么排序？

**候选人：** 默认按相关性（BM25），可选按时间或在特定对话内置顶。我们也支持过滤器：发送方、日期范围、有附件、对话。每个过滤器都是一个标准的 Elasticsearch term 查询。

**面试官：** 删除呢？如果用户删除消息，它会从搜索中消失吗？

**候选人：** 是的 —— 我们向 Search Indexer 发布删除事件，它从索引中移除文档。先软删除（标记删除但保留一段宽限期，以防误删），再硬删除 —— 比如 30 天后。索引器遵循同样的生命周期。

**面试官：** *（记录）* 好。再深入一个领域。选：WebSocket 网关扩展，或 PB 规模的消息存储。你选。

**候选人：** 选 WebSocket 网关吧。它运维上最有意思 —— 因为它是有状态的。

**面试官：** 开始。

**候选人：** 每个网关是一台持有打开 WebSocket 连接的服务器。每服务器的连接数受文件描述符、内存、CPU 的上限制约。经过仔细调优 —— Linux 内核设置、高效的 WS 库 —— 单个网关能持有 10 万到 50 万连接。我们规划数字按每网关 20 万算。

峰值 1 亿并发用户，那就是 500 台网关服务器。为降低延迟跨 3-5 个区域分布，算每区域 100-200 台。

**面试官：** 客户端怎么找到网关？

**候选人：** Geo-DNS 路由到最近的区域。在区域内，L4 LB 把新连接分配给网关。如果想要可预测性，LB 可以按客户端 IP 做粘性，但没必要 —— Session Registry 按连接处理路由。

**面试官：** 如果一个网关到达容量呢？

**候选人：** 健康检查失败或卸掉一些连接。LB 停止发新连接；现有连接保留。连接数接近阈值时通过加网关容量自动扩缩容。

**面试官：** 如果一个网关崩溃，失败模式是什么？

**候选人：** 它上面的所有连接都掉线。客户端通过 WebSocket close 检测到，重连。LB 路由到其他网关。重连时，客户端通过 REST 拉取断连期间错过的消息。用户总影响：几秒的"重连中"UI，没有消息丢失。

**面试官：** 20 万连接的重连风暴是什么样子？

**候选人：** 朴素地处理的话很糟。20 万客户端同时重连是雷霆群冲击。缓解：客户端用带抖动的指数退避 —— 重连前随机延迟 100ms 到 5s。其余网关逐步吸收负载。我们也会预先扩容，让幸存网关有冗余。

**面试官：** 冗余量是多少？

**候选人：** 规划 N+2 —— 任何两台网关同时下线，其余的能吸收负载。每网关 20 万连接、100 台网关，那是 20 万额外连接要分散到 98 台上 —— 每台约 2% 冗余。我们会规划到 20-30% 冗余，以应对多重故障或意外尖峰。

**面试官：** 部署的优雅关闭呢？

**候选人：** Drain 模式。网关停止接受新连接，向现有客户端发信号让其重连 —— 服务端发起的优雅关闭，带"请重连"代码。客户端转到其他网关。关闭中的网关等连接 drain 完或超时，再终止。

**面试官：** 部署时间线是什么？

**候选人：** 滚动：一次关 5% 的网关，等连接迁移，部署新版本，重新上线，再继续下一批。在我们的规模下整个机群滚动可能要 30-60 分钟。

**面试官：** 如果新版本有 bug 让所有重连的客户端把它搞崩呢？

**候选人：** *（思考）* 那是部署时的雷霆群冲击。我们会用金丝雀部署：先发布到 1% 的网关，观察连接尖峰下的行为，再扩大。我们也可以在部署期间在 LB 上对重连做限流。

**面试官：** 好。网关的跨区域呢？

**候选人：** 客户端连到最近的区域。区域 A 的网关可能要送达一条消息，但这个用户的会话挂在区域 B 的网关上。两个选项：通过 fan-out 服务把消息跨区域路由到目标区域的网关，或者让所有区域去查一个全局 Session Registry。

**面试官：** 哪个？

**候选人：** 都要，分层来做。每区域 Session Registry，大部分查找在本地 —— 联系人多在同区域的用户，fan-out 是本地的。对跨区域目标，fan-out 服务通过镜像到目标区域的 Kafka topic 转发，或直接 gRPC 调用目标区域的 fan-out 服务。

**面试官：** Session Registry 的复制？

**候选人：** 每区域一份 Redis，带跨区域复制用于跨区域查找。大部分用户只跟附近的人聊，所以跨区域流量是一个小比例。复制可以异步 —— 跨区域在线状态有几百毫秒的陈旧没问题。

**面试官：** 会话移动了区域的用户的消息送达呢？

**候选人：** 有竞态条件风险：用户从区域 A 断开、从区域 B 重连，区域 A 的 fan-out 还持有过期会话信息，推到不再有该连接的网关。推送失败，fan-out 丢弃，消息从推送路径丢失 —— 但仍在 Cassandra 里。客户端下次同步时会拉到。送达被延迟但没丢。要减少延迟，我们会加一个"推送失败就试备用区域"的重试，它去查全局注册表。

**面试官：** 但如果经常发生那就很贵。

**候选人：** 对 —— 所以我们会缓存每用户的"最后已知区域"，只在失败时去查全局。稳定用户（大多数）命中缓存；在基站间漫游的移动用户可能偶尔触发跨区域查找。成本有界。

**面试官：** *（看时间）* 好。失败场景。整个 Cassandra 集群宕了。会怎样？

**候选人：** 写入失败。Message Service 不能 ACK SEND，因为我们无法持久化。发送快速失败 —— 客户端看到错误，带退避重试。读取：最近消息在应用/客户端缓存里；更早的历史拉取失败。

**面试官：** 影响半径？

**候选人：** 全部 —— 没有消息能用。我们会要多区域 Cassandra 带主动复制，让单区域故障不会让全世界宕机。跨区域 Cassandra 支持得很好；每区域配 replication factor 3，用 `LOCAL_QUORUM` 写入以获得低延迟。

**面试官：** 故障切换期间呢？

**候选人：** 用 LOCAL_QUORUM，区域内单 AZ 故障不阻塞写入。完整的区域故障意味着客户端路由到其他区域，但他们的数据 —— 包括对话成员资格 —— 必须在那里可用。这要求 Postgres 侧也跨区域复制，这更难。

**面试官：** 为什么更难？

**候选人：** Postgres 没有像 Cassandra 那样的原生跨区域多主。我们会用逻辑复制并接受最终一致，或者用 CockroachDB 或 Spanner 这类全球分布式 SQL 来存元数据。增加了复杂度，但在这种规模下是正确的决定。

**面试官：** 跨区域新对话的元数据更新怎么处理？

**候选人：** 用户创建一个对话时，成员资格和元数据先写入本地区域的 Postgres，再异步复制。如果另一个区域的成员在复制完成前就给新对话发消息，他们会短暂地拿到"对话未找到"的错误。缓解办法：在复制赶上之前让初始消息通过发起区域路由，或者在第一条跨区域消息里内联包含元数据快照。

**面试官：** 如果 Cassandra 是多主，跨区域的消息排序呢？

**候选人：** 这里变得棘手。如果区域 A 的 Alice 和区域 B 的 Bob 同时给同一对话发消息，两个 Cassandra 各自分配自己的 clustering key 并复制。冲突解决取决于 schema —— 对消息来说，每行有唯一的 (conv_id, msg_id)，我们会把区域编码到高位，让 msg_id 全局唯一，类似 Snowflake。这样两条消息共存，按 msg_id 排序。

**面试官：** 产生的顺序是用户期望的吗？

**候选人：** 大致是 —— 基于时间戳加区域 tiebreaker。没有显式 happens-before 跟踪，就没有跨区域的因果保留。对聊天，我们接受几百毫秒内的交错是非确定的。

**面试官：** *（记录）* 让我回到之前的事情。你提到了每对话计数器。是按区域还是全局？

**候选人：** 在多区域模型里，按区域每对话计数器，加上区域编码让 msg_ids 全局唯一。我们放弃严格的全局单调性以换性能。

**面试官：** 这会影响历史拉取吗？

**候选人：** 历史拉取变成"给我所有 msg_id > X 的消息，按 msg_id 排序"。用 Snowflake 风格的 ID，排序近似因果顺序。来自不同区域的最近消息可能按区域 tiebreaker 交错，但视觉上看没问题。

**面试官：** 好。端到端加密 —— 既然有时间，简短聊聊？

**候选人：** 好。有了 E2EE，服务端就是密文中继；读不到消息内容。这会破坏服务端的功能：没有搜索、没有消息审核、没有基于内容的 fan-out 优化。标准做法是 Signal 协议，群组用 MLS。对群，MLS 比 Signal 扩展性更好，因为群密钥管理在成员数上是对数级的。

**面试官：** 为什么对数级？

**候选人：** MLS 用基于树的密钥结构 —— 每个成员是一个叶子，内部节点持有路径上的共享秘密。加或移一个成员更新 O(log N) 个树节点，而不是给每对都做配对会话。对 100 人群，那是约 7 次更新而不是 100 次。对 10 万人群，MLS 是 E2EE 唯一实用的选项。

**面试官：** 这怎么影响存储？

**候选人：** 从服务端视角看，消息体是密文 blob。schema 一样，但服务端能做的少了。元数据 —— 发送方、对话、时间戳 —— 仍可见，除非我们加 sealed-sender 这样的元数据保护层。

**面试官：** 搜索？

**候选人：** 客户端建索引，或者干脆没有搜索。一些实现把消息同步到客户端，让客户端本地搜索。这对单设备用户有效；多设备会很尴尬。

**面试官：** *（看时间）* 五分钟。如果有更多时间你会加什么？

**候选人：** 很多。*（思考）* 一：一个合适的离线优先客户端架构，带本地优先存储和同步 —— Web 上用 IndexedDB、原生用 SQLite —— 让客户端在连接糟糕时也始终有数据。二：消息反应和 threading，融入 schema 不难，但有各自的 fan-out 和通知语义。三：联邦 —— 通过 Matrix 或 XMPP 跟其他聊天网络互操作，这是一整层架构。四：反垃圾和滥用检测 —— 基于 ML 的内容扫描（用于非 E2EE 通道）以及基于行为的信号（连发、账号年龄等）。五：合规 —— 消息保留策略、法律保留、GDPR 删除工作流。

**面试官：** 挑一个，告诉我你会怎么开始。

**候选人：** 离线优先客户端。原则：每个 UI 操作先应用到本地状态，再同步。我们会做一个带乐观 UI 的本地消息日志 —— Alice 发一条消息时，立刻显示为"发送中"，本地 SQLite 用 `pending` 标志存下来。同步层推送到服务端，ACK 到达时更新本地记录。离线时消息保持 `pending`，直到连接恢复。

已读状态以本地模型为准 —— 用户实际看过什么。服务端是跨设备同步的协调者。客户端上线时，推送本地已读状态增量并拉取服务端的。冲突 —— 同一对话从两台设备以不同 last-read 位置读取 —— 解析为 max(local, server)。

**面试官：** 好。好，时间到了。谢谢一起走完这个。

**候选人：** 谢谢。我一直喜欢聊天设计题 —— 它有那么多层次。

## 做得好的地方

- 有条不紊的澄清阶段：在动笔设计之前就敲定了消息 vs 语音/视频、群大小上限、E2EE 范围、在线状态语义。
- 自信地处理了双写问题（Cassandra + Kafka）。直接提到 outbox 模式，并解释了备选方案（CDC）。
- 关于每对话 vs 全局排序的回答很强。识别了同时发送的因果问题，干净利落地提出 last-seen-msg-id 传播。
- 对超大群（10 万+）调整了 fan-out 设计且没有过度承诺 —— 切换到对在线者推送 + 重连时拉取，而不是坚持普遍推送。
- 具体地阐述了多区域取舍：按区域 msg_id 编码、LOCAL_QUORUM、Postgres 的异步跨区域复制。
- 加分项是把 WebSocket 网关当成唯一的有状态层，并讲清了部署故事（drain、canary、雷霆群冲击）。

## 可以更强的地方

- 最初的 msg_id 设计回避了热门对话上的争用。只有在被追问时候选人才切换到为高 fan-out 层级用 Snowflake 风格的 ID。值得主动提出。
- 大规模在线状态解决得还行，但在订阅流失上本可以更严谨 —— 用户关闭一个对话打开另一个时会发生什么，注册表上的流失。
- 推送通知本可以更明确地处理安静时段、通知分组和角标计数。这些在聊天规模下是真实的产品关切。
- 提到了 MLS 用于 E2EE 群密钥但没有解释为什么它比 Signal 扩展性更好 —— 更强的候选人会指出对数级密钥更新成本 vs Signal 的配对会话。
- 搜索范围窄了 —— 没有处理相关性排名、索引上的安全裁剪或附件索引（OCR、文件内容）。

## 关键要点

- 按对话分片是聊天的正确原语 —— 对话的所有写入在一个分片上串行化，排序是本地的，扩展来自并行的多个分片。
- 当单个用户操作需要持久写入和下游发布两者时，使用 outbox 模式（或 CDC）以避免双写问题。
- 对非常大的群，混合推送（在线接收方）和拉取（离线 + 后加入者）。不要试图永远向每个人推送。
- WebSocket 网关是唯一的有状态服务。仔细对待它：心跳、drain、金丝雀部署、区域固定。
- 因果排序需要显式跟踪 —— last-seen-msg-id 传播、向量时钟，或接受同时性是非确定的。
- 媒体通过 blob 存储和 CDN 走带外。不要把 50MB 视频推过消息总线。

## 参考架构

```
                            +-------------------------+
                            |   Mobile / Web / Desk   |
                            +-----------+-------------+
                                        |
                                  WebSocket (TLS)
                                        |
                                Geo-DNS / Anycast
                                        |
                  +---------------------+---------------------+
                  |                                           |
       +----------v----------+                    +-----------v----------+
       |      Region A       |                    |       Region B       |
       |                     |                    |                      |
       |  [L4 LB]            |                    |   [L4 LB]            |
       |     |               |                    |      |               |
       |  [WS Gateway pool]  |<---heartbeats--->  |   [WS Gateway pool]  |
       |     |               |                    |      |               |
       |     +-> [Session Registry (Redis)] <-replicate-> [Session Reg]  |
       |     |                                    |      |               |
       |  [Message Service] -gRPC-+               |   [Message Service]  |
       |     |                    |               |      |               |
       |  [Conversation Sharder]  |               |   [Conv Sharder]     |
       |     |                    |               |      |               |
       |  [Cassandra (RF=3)] <----+---xregion-----+-> [Cassandra (RF=3)] |
       |     |                                    |      |               |
       |  [Postgres (metadata)] <----replicate----+-> [Postgres]         |
       |     |                                    |      |               |
       |  [Kafka] -+                              |   [Kafka] -+         |
       |     |     |                              |      |     |         |
       |     |     +---> [Fan-out Service]        |      |     +-> [Fan-out]
       |     |     +---> [Push Notif Service] --> APNs/FCM                |
       |     |     +---> [Search Indexer] --> [Elasticsearch]            |
       |     |     +---> [Media Processor] --> [S3 + CDN]                |
       |     |                                                            |
       +-----+------------------------------------+----------------------+
             |                                    |
             +------ outbox sync / Kafka mirror --+
```

关键数据：
- 1 亿 DAU，峰值 30 万 msg/秒，每天 1 TB 存储增长
- 每网关 20 万 WebSocket 连接，500-1000 个网关
- 每对话 msg_id，高 fan-out 层级用 Snowflake 风格
- LOCAL_QUORUM 的 Cassandra 写入，Postgres 异步跨区域
- 按区域 Session Registry 加跨区域 pub/sub
- 消息 -> Kafka 发布使用 outbox 模式

热路径延迟预算（同区域 1:1）：
- WS 发送 -> 网关：约 10ms 客户端到边缘
- 网关 -> Message Service：约 1ms
- Cassandra 写入（LOCAL_QUORUM）：约 5-15ms
- Fan-out 查找 + 推送：约 5-10ms
- 接收方网关 -> 客户端：约 10ms
- 总计：约 30-50ms p50，约 150-300ms p99
