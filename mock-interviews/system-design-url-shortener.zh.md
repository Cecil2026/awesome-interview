# 模拟面试：设计短链接服务（TinyURL / bit.ly）

**类型：** 系统设计
**难度：** L4
**时长：** 45-60 分钟（模拟）
**标签：** #system-design, #scalability, #caching, #database-sharding, #base62

## 背景

候选人正在面试一家大型消费者互联网公司的高级软件工程师（L4）岗位。面试官是平台团队的 Staff 工程师。题目在这一轮开始时公布，没有提前准备。一个共享虚拟白板打开着。候选人有一杯水和一个笔记本；面试官有一摞追问问题，但大部分是即兴。

## 对话记录

**面试官：** 好，感谢加入。我希望你设计一个短链接服务 —— 类似 TinyURL 或 bit.ly。你想怎么发挥都可以。我会随时打断问问题。

**候选人：** 听起来不错。在我开始之前，我可以花几分钟澄清范围吗？

**面试官：** 请。

**候选人：** 好，那么大致来说，用户提交一个长 URL，我们返回一个短的。当有人访问短 URL 时，我们重定向他们到原始的。这是核心。你想让我也覆盖分析、自定义别名、过期、用户账户吗？

**面试官：** 这么说吧：重定向是必须的，自定义别名是 nice-to-have，基本的点击分析是延伸目标。暂时不要用户账户 —— 假设匿名或基于 API key 的使用。

**候选人：** 明白。我们的目标规模是多少？比如，我们是小创业公司还是要替代 bit.ly？

**面试官：** 假设我们在 bit.ly 的规模。做合理的估计。

**候选人：** 好。让我做信封背估算。*（开始在白板上写）* bit.ly 的公开数据是每月几亿新链接，每月大约 100 亿次点击。大致：每月 1 亿次写入约等于每秒 40 次写入的平均值。读取 —— 每月 100 亿次大约是每秒 4000 次的平均值。乘上 10 倍峰值因子，峰值算 400 写/秒、40K 读/秒。

**面试官：** 那个读写比例似乎很重要。它告诉你什么？

**候选人：** 大约 100:1 偏读。所以我的系统要为读积极优化 —— 缓存、副本，能上 CDN 就上 CDN。写相比之下可忽略，我可以集中化。

**面试官：** 好。存储呢？

**候选人：** 如果我们永远保留链接，每月增加 1 亿，每年就是 12 亿。每行大概 500 字节 —— 短码、长 URL、created_at、owner_id、可选过期。每年大约 600 GB。五年后 3 TB。可管理，但我最终会考虑分片。

**面试官：** 好。延迟目标？

**候选人：** 对重定向，我把目标设在端到端 p99 低于 50ms。重定向是用户可见的关键路径 —— 任何延迟都会被直接感受到。

**面试官：** 好。开始设计之前还有什么吗？

**候选人：** 还有一个：短码长度。6 字符 base62 下有约 10^11 个可能的码，7 字符下约 10^12 个。每年 12 亿链接、5 年视野，6 字符是临界；7 字符给我们冗余。我会目标 7。

**面试官：** 为什么具体选 base62？

**候选人：** Base62 —— a 到 z、A 到 Z、0 到 9 —— 不用转义就 URL-safe，大小写敏感所以比 base36 密度更高，并且比用 `+` 和 `/` 的 base64 避免了更少的混淆字符。一些短链接服务用 base58 来去掉像 0/O 和 1/l 的相似字符，但 base62 是常见选择。

**面试官：** 好，继续。给我勾勒 API。

**候选人：** *（在白板上写）*

```
POST /api/v1/shorten
  body: { "long_url": "https://...", "custom_alias": "optional", "expires_at": "optional" }
  returns: { "short_url": "https://tny.co/aB3xQ7p", "code": "aB3xQ7p" }

GET /{code}
  -> 301 或 302 重定向到长 URL

GET /api/v1/links/{code}/stats   （延伸）
  returns: { "clicks": 1234, "created_at": "...", "long_url": "..." }
```

**面试官：** 301 还是 302？

**候选人：** *（停顿）* 看情况。301 是永久 —— 浏览器激进地缓存它，对我们的负载很好，但如果我们想要点击分析就很糟糕，因为后续点击绕过我们的服务器。302 是临时 —— 每次点击都打到我们，可以计数，但失去缓存收益。

**面试官：** 你会选哪个？

**候选人：** 对短链接服务我会默认 302。分析价值是真实的，每点击开销可管理。如果客户不需要分析，我们可以作为升级选项提供 301 模式。

**面试官：** 合理。继续。

**候选人：** 现在数据模型。核心表很直接。

```
links (
  code VARCHAR(10) PRIMARY KEY,
  long_url TEXT NOT NULL,
  created_at TIMESTAMP NOT NULL,
  expires_at TIMESTAMP NULL,
  owner_id BIGINT NULL,
  is_custom BOOLEAN DEFAULT FALSE
)
```

对分析，单独的表或流：

```
clicks (
  code VARCHAR(10),
  clicked_at TIMESTAMP,
  ip_hash VARCHAR(64),
  referrer TEXT,
  user_agent_class VARCHAR(32)
)
```

**面试官：** 为什么 `code` 是主键而不是自增 ID？

**候选人：** 因为查找路径永远是按 code。把它做 PK，在大多数数据库里我们就免费拿到聚簇索引。没有其他需要单独 ID 的访问模式 —— 我们从不按"创建的第 5000 个链接"列出链接。

**面试官：** 如果你想把分析关联回用户呢？

**候选人：** 好观点。我会加 `owner_id` 作为二级索引；如果用户有很多链接，可能还要单独的 `user_links` 表。但热路径 —— 重定向 —— 只需要 code 查找。

**面试官：** 好。SQL 还是 NoSQL？

**候选人：** 我会从 SQL 开始 —— 比如 Postgres。数据每行小、结构化、访问模式简单。我们不需要 NoSQL 的灵活性，而在检查自定义别名的码冲突时也能受益于事务。

**面试官：** 如果我告诉你读取是每秒 10 万而不是 4 万呢？

**候选人：** SQL 仍然没问题 —— 带读副本的 Postgres 加上前面的 Redis 缓存可以轻松做每秒 10 万读。瓶颈不是数据库引擎，是我们是否打到磁盘。如果 95% 读命中缓存，DB 大概看到 5K rps，那是平凡的。

**面试官：** 走一遍 code 怎么生成。这里变得有意思。

**候选人：** 好。两个主要方法。一：用 MD5 或 SHA-256 哈希长 URL，取前 N 字节，base62 编码。二：维护一个全局计数器，每个新链接递增，base62 编码计数器值。

**面试官：** 你偏好哪个？

**候选人：** 哈希有个好性质：同样的 URL 总产生同样的 code，去重免费。但有冲突 —— 即使 MD5 截断到 7 个 base62 字符，在规模上也有有意义的冲突概率。所以你需要冲突处理：如果 `links[code]` 存在且 long_url 不同，你必须用盐重新哈希，或者退回另一种方案。

**面试官：** 好，那哈希。

**候选人：** 是，我会选哈希然后检查。计算 MD5，取前 42 位，base62，做 `INSERT ... ON CONFLICT` 并检测冲突。冲突且现有 long_url 匹配，返回现有 code；冲突且 URL 不同，附加一个计数器或随机 nonce 并重试。

**面试官：** *（记录）* 让我推一下。重试的最坏情况是什么？

**候选人：** 7 个 base62 字符我们有 3.5 万亿码。5 年 60 亿链接，我们用了空间的 0.17%。按生日悖论数学，冲突在几个百分点填充率以上才变得明显，所以重试应该很少 —— 远低于 1%。

**面试官：** 嗯。但每个写都必须做一次读来检查冲突。每次重试是又一次读。这不会伤害写吞吐量吗？

**候选人：** *（停顿）* 有道理。400 写/秒不是问题，但这是多一次往返。更干净的方法可能是计数器方案 —— 保证唯一性，关键路径无读。

**面试官：** 多说说那个。

**候选人：** 维护一个单调计数器。每次写时 fetch-and-increment，base62 编码计数器值，就是你的 code。按构造不可能有冲突。

**面试官：** 缺点是什么？

**候选人：** 三个问题。一，计数器是全局资源 —— 如果它住在一个 DB 里，就是一个写瓶颈。二，顺序码可猜 —— 任何人都能遍历链接。三，失去了 URL 级去重；同一个 URL 提交两次会得到两个不同的 code。

**面试官：** 你会怎么解决瓶颈？

**候选人：** 分发范围。每个应用服务器从中央分配器请求一个块，比如 10,000 个计数器值，然后本地分配。分配器每 10K 个链接才看到一次请求。你也可以分片分配器 —— 例如有 16 个分配器，每个拥有 `id mod 16 == allocator_id` 的计数器。

**面试官：** 可猜性呢？

**候选人：** 两个选项。在编码之前对计数器空间应用一个可逆双射 —— 比如 Feistel cipher 或简单的乘法打乱。Code 看起来随机，但每个仍 1:1 映射到一个计数器。或者接受它；对非敏感链接，可猜性是特性，不是 bug —— 短码本质上是公开的。

**面试官：** 去重损失呢？

**候选人：** 我会在长 URL 上加一个二级索引 —— `UNIQUE (url_hash)`，其中 `url_hash` 是规范化 URL 的 SHA-256。写入时先查这个索引；存在就返回现有 code。这样去重就拿回来了，代价是一次额外查找，而且可以缓存。

**面试官：** 去重索引 —— key 是什么？

**候选人：** 规范化长 URL 的 SHA-256。规范化本身就是一团乱：scheme 和 host 小写、剥离默认端口、查询参数排序、解码非保留百分号转义、丢弃片段、国际化域名用 punycode。不同的规范化选择会产生不同的去重行为，所以我们会挑一个约定并坚持。

**面试官：** 你把 `?utm_source=foo` 当作跟裸版本同一个 URL 吗？

**候选人：** 通常不 —— 营销团队把 UTM 参数当作不同的目的地来对待。我们会默认保留它们。如果用户想要激进去重，可以通过缩短调用上的参数主动选择加入。

**面试官：** 好的转折。好，勾勒高层架构。

**候选人：** *（开始画）*

```
[Clients]
   |
   v
[DNS / Anycast]
   |
   v
[L7 Load Balancer]    <- TLS 终止、限流
   |
   v
[App Servers (stateless)]
   |     \
   |      \-> [Counter Allocator Service]
   v
[Redis Cache]   <- code -> long_url
   |
   v
[Postgres Primary] -> [Read Replicas]
   |
   v
[Kafka] -> [Click Aggregator] -> [Analytics Store: ClickHouse]
```

重定向路径是：LB -> 应用服务器 -> Redis -> 302。缓存未命中穿透到一个读副本。

**面试官：** 讲讲缓存。

**候选人：** Redis 集群，key 是 code，value 是长 URL。TTL 可能 24 小时，带缓存击穿保护 —— singleflight 或概率性提前刷新。

**面试官：** 为什么具体选 24 小时？

**候选人：** 这是一个平衡。更长的 TTL 提高命中率，但在冷 key 上浪费内存。更短的 TTL 更快释放内存，但导致更多缓存未命中。24 小时大致匹配典型分享链接的半衰期 —— 大多数点击流量发生在第一天内。等有真实数据后再调整。

**面试官：** 命中率？

**候选人：** URL 流量是幂律分布 —— 一小部分链接占大多数点击。几百 GB 的 Redis 我预期预热后 95-99% 命中率。

**面试官：** 那个估计从哪来？

**候选人：** 粗略心智模型。如果前 1% 的链接驱动 80% 的点击 —— 典型幂律形状 —— 那么只缓存 60 亿链接的前 1% 就是 6000 万条目。Redis 里每个带开销约 200 字节，那是 12 GB，轻松装在一个节点里。缓存前 10% 抓住更多长尾，集群上是 120 GB。

**面试官：** 冷链接呢？没被看过的旧链接。

**候选人：** 它们穿透到 DB。延迟从亚毫秒到也许 5ms —— 仍然在 50ms 预算内。

**面试官：** 淘汰策略？

**候选人：** LRU 或 LFU。我会倾向 LFU，因为 URL 流量严重倾斜 —— 一个链接一旦病毒式传播，会保持热门一段时间；我们想留住这些，即使某些随机的新鲜链接最近被访问过。Redis 通过 `maxmemory-policy` 支持两者。

**面试官：** *（记录）* 好，深入一件事。假设 DB 有 100 亿行。你怎么保持重定向快？

**候选人：** *（思考片刻）* 三层。一，缓存吸收热尾 —— 那是我们 95%+ 命中率。二，DB 本身：code 上的聚簇索引，查找本质是 log(N) 加磁盘寻道。100 亿行约 33 层索引，但 B 树的顶部在内存里，所以每次查找大概付一次磁盘寻道。

**面试官：** 如果你有每秒 4 万读取，每次查找仍然多。

**候选人：** 对，但大多数是缓存命中。DB 只看到缓存未命中的残余 —— 比如 5%，就是 2000 读/秒。一台带 SSD 的 Postgres 节点从容处理。

**面试官：** 如果你不能把索引装进内存呢？

**候选人：** 那我会分片。按 code 分区 —— 取 code 的 hash 模 shard 数，让每个 shard 拥有一个不相交子集。每个 shard 有自己的索引，装得下内存，读取在 shard 间并行化。应用层路由知道哪个 shard 拥有哪个 code。

**面试官：** 为什么哈希 code 而不是范围分区？

**候选人：** 按 code 范围分区会造成热点。新创建的 code 往往比旧的更受欢迎，所以"最新"的 shard 拿到不成比例的流量。哈希让写和读均匀分布在所有 shard 上。范围分区在范围查询主导时有意义 —— 那不是我们的访问模式。

**面试官：** 你怎么选 shard 数？

**候选人：** 容量驱动。如果我想每个 shard 的工作集低于比如 50 GB 索引、整个索引是 2 TB，那我需要 40 个 shard。向上取整到 64 留冗余，也让模运算友好。

**面试官：** 重分片呢？

**候选人：** 能避免就避免 —— 这就是为什么我会从 64 开始，即使今天只需要 40。如果我们确实需要增长，一致性哈希或虚拟分片层让它不那么痛苦。我们把每个 code 映射到一个虚拟分片（比如 1024 个桶），每个物理 shard 拥有一段虚拟分片范围。要扩缩容，就跨新物理节点拆分虚拟分片。

**面试官：** 带我走一遍你拆分虚拟分片时的迁移。

**候选人：** 复制阶段：从源虚拟分片读所有行，写到目标。追赶阶段：通过逻辑复制或 CDC 工具 —— Debezium 之类 —— 流式增量变更。切换：短暂暂停该虚拟分片的写入 —— 毫秒级 —— 翻转路由表，恢复。该 shard 流量的总用户影响是亚秒级抖动。我们会在低流量窗口做，一次迁一个虚拟分片。

**面试官：** 如果你不能暂停写入呢？

**候选人：** 双写阶段：切换期间写入同时进旧和新 shard；翻转前读取仍走旧的。然后把读切到新的，经过验证窗口后再停止写旧的。迁移更慢，但零暂停。标准的在线 schema 变更模式。

**面试官：** 同步和异步的选择 —— 你怎么做？

**候选人：** 异步，除非业务明确要求更强保证。短链接的写入不涉及钱 —— 区域故障切换期间丢几秒的写入可以接受。同步复制的延迟成本对这种工作负载不值得。

**面试官：** 好。现在更深入。带我走一遍单次重定向端到端。我想要微秒级。

**候选人：** *（拉出一个新白板）*

```
Client sends GET /aB3xQ7p
  -> Anycast 路由到最近的 PoP                                ~5-30ms (网络)
  -> L7 LB 终止 TLS                                          ~1-2ms
  -> 转发到应用服务器（DC 内 HTTP keepalive）                ~0.5ms
  -> 应用服务器：解析路径，提取 code                          ~0.05ms
  -> Redis GET aB3xQ7p （本地网络上）                        ~0.5ms
  -> 缓存命中：解析长 URL                                     ~0.01ms
  -> 把点击事件发到 Kafka（fire-and-forget，批量）            ~0.1ms
  -> 返回 302 带 Location 头                                  ~0.05ms
  -> 网络回到客户端                                            ~5-30ms

总服务器时间：~1-3ms
总挂钟时间：~15-65ms（大部分是网络）
```

所以我们 50ms 的 p99 预算很宽裕，只要缓存命中且我们不同步等 Kafka。

**面试官：** 如果 Kafka 宕了呢？

**候选人：** 点击事件在重定向路径上应该是 fire-and-forget。Kafka 不可达时，我们在内存里缓冲，缓冲填满就丢最旧的。重定向本身从不阻塞分析 —— 分析是尽力而为。

**面试官：** 如果 Redis 宕了呢？

**候选人：** 穿透到 DB。延迟上升 —— 每请求大概加 5-10ms，DB 容量成为瓶颈。我们需要一个熔断器或慢路径，限制并发 DB 查询，避免雪崩。

**面试官：** 给我看缓存击穿问题。

**候选人：** 好。如果一个热 key 从 Redis 过期，10,000 个请求一起打，它们都看到未命中，都同时打 DB。DB 被噎住，延迟激增，重试级联。

**面试官：** 你怎么修？

**候选人：** 应用服务器上的 singleflight —— 把对同一 key 的并发未命中合并成一次 DB 查询。或概率性提前过期 —— 一小部分客户端在 TTL 之前稍微刷新缓存，让 TTL 命中时大多数客户端已经有新值。或用 Redis 的 `SETNX` 选举一个客户端去刷新，其他人服务陈旧值。

**面试官：** 如果一个 key 热到甚至本地缓存查找都是瓶颈呢？

**候选人：** 到那时我们说的是单台服务器环回上的 50K+ rps。启用热 key 复制：从 N 个副本服务同样的内容，load balancer 轮询。我们实际上是把那个单一 URL 水平扩缩容。CDN 方法也有帮助，因为边缘节点完全终止请求，不接触源。

**面试官：** 对非常热的 key 你能不能纯缓存？

**候选人：** 能 —— 钉住它们。如果一个 code 在我们的"始终热"列表上，就从不从任何层淘汰它。对这些条目的更新主动推送。我们会标记某层级以上账户的链接，或越过病毒阈值的链接。

**面试官：** 好。换档。你怎么处理写入？

**候选人：** 写路径：客户端 POST 到 `/api/v1/shorten` 带长 URL。应用服务器做校验 —— schema 检查、恶意软件/钓鱼域名黑名单、长度检查。然后计算一个规范化 URL 哈希，检查去重索引。如果 code 已存在，就返回它。否则，从分配器请求一个计数器，base62 编码，INSERT 到主 DB。

**面试官：** 计数器的操作顺序 —— 如果计数器递增了但 INSERT 失败呢？

**候选人：** 你泄漏一个 code。没事 —— code 空间巨大，泄漏不重要。我们要保证的是"无重复"，不是"无跳过"。

**面试官：** 如果 INSERT 成功但给客户端的响应丢失了呢？

**候选人：** 客户端重试。重试带同样的长 URL 进来；去重索引看到现有行，返回同样的 code。所以重试在 URL 级别是幂等的。为绝对安全，我们可以让客户端在请求头里发一个幂等 key，我们按它缓存响应，比如 24 小时。

**面试官：** 像 Stripe 的幂等 key。

**候选人：** 完全是。标准模式。每次写花我们一个 Redis 条目，那是小的。

**面试官：** 好。分配器呢？你提到范围。如果一个应用服务器拿了 10K 的范围然后崩了呢？

**候选人：** 一样 —— 我们泄漏那个范围内未用的 code。在我们的规模下不是问题。如果我们想整洁，分配器可以跟踪未处理的范围，长超时后回收它们，但不值得这个复杂度。

**面试官：** 讲讲分配器自己的一致性。如果有两个分配器实例且它们发出了重叠的范围呢？

**候选人：** 那会是个 bug。分配器需要在计数器递增上强一致。选项：单实例加热备，或用强一致存储支持它 —— etcd、ZooKeeper，或用 `SELECT ... FOR UPDATE` 更新的 Postgres 行。我会用 Postgres，因为我们已经有它；争用很低，因为每个请求是 10K 大小的范围。

**面试官：** 如果 Postgres 行变成瓶颈的失败模式是什么？

**候选人：** 在我们的规模下不应该。400 写/秒 ÷ 10K 块大小 = 每 25 秒一次分配器请求。即使 10 倍流量，也是每 2.5 秒一次请求 —— 一个单行锁几毫秒没问题。如果真的饱和了，我们会像我提到的那样分片计数器空间，或者转到 Snowflake 风格的 ID 生成系统，每个节点嵌入自己的机器 ID 和每节点序列。

**面试官：** Snowflake ID 是 64 位的。它们怎么装进我们的 7 字符 base62 空间？

**候选人：** 它们不直接装 —— Snowflake ID 太大。但我们可以在更小规模借用这个模式：42 位计数器空间分成比如 8 位机器 ID 和 34 位每机器序列。2^34 就是每台机器约 170 亿 code，够了。取舍是 base62 编码现在是每机器而不是全局单调，但单调性不是要求。

**面试官：** 如果一个 DB 行变得有争用，你怎么分片分配器？

**候选人：** 16 个逻辑计数器空间，每个有自己的行。每个空间分配 `code mod 16 == space_id` 的 code。在 base62 转换前把空间 ID 编码到计数器的高位，所以不同空间的 code 不冲突。

**面试官：** 好。自定义别名 —— 它们怎么交互？

**候选人：** 自定义别名是用户选的，所以它们在计数器方案之外。处理自定义别名请求时，我们用用户选的 code 做 INSERT；冲突就返回错误。`is_custom` 标志区分它们。我们也要保留一个命名空间，让自定义别名不能跟计数器生成的 code 冲突 —— 例如自定义别名必须至少 8 字符，或以特定前缀开头。

**面试官：** 如果自定义别名是 "google" —— 一个常见词呢？

**候选人：** *（笑）* 最简单情况下是先到先得。实践中你会有一个高价值词的黑名单，为高级客户保留它们，加上脏词过滤、商标纠纷流程。那更是产品问题而不是系统设计问题，但你会想要预留那个钩子。

**面试官：** 公平。讲讲一致性。如果我在主库上创建一个链接，我能立刻从副本用它重定向吗？

**候选人：** 异步复制下复制延迟通常是几十毫秒。有一个小窗口里新 code 存在于主库但还没在副本上。两种处理方式：在每用户短窗口里把写后读路由到主库，或通过会话 token 实现 read-your-writes。大多数用户不会注意 —— 间隙以几十 ms 计。

**面试官：** 缓存呢？你刚写了一个新 code；你能立刻读它吗？

**候选人：** 最简单的做法是写入时填充缓存 —— write-through。INSERT 成功后，在 Redis 里用 TTL 设置 code。第一次读就直接命中缓存。

**面试官：** 如果缓存写成功但 DB 写回滚了有风险吗？

**候选人：** 是 —— 你会在缓存里有一个幽灵条目。缓解：先写 DB，再写缓存。DB 写成功后缓存写失败，就降级到 cache-aside —— 下次读未命中再刷新。反过来 —— 先缓存再 DB —— 是危险方向。

**面试官：** 分析计数器呢？

**候选人：** 那是故意最终一致。点击事件进 Kafka，由流处理器 —— Flink 或自定义消费者 —— 按一分钟或五分钟滚动聚合到 ClickHouse。用户看到稍微陈旧的计数；这是分析仪表板的标准。

**面试官：** 如果一个恶意用户想吹高他们的点击计数呢？

**候选人：** 按 IP 限流，在短窗口内去重同 IP 的点击，可选地要求 CAPTCHA 保护的 referrer 才进入"已验证"计数。机器人检测是它自己的问题；你会想要下游分类器。对系统设计，注意我们留了一个扩展点。

**面试官：** 怎么区分点击和预取？浏览器和链接展开器拉取你的 URL 只是为了预览。

**候选人：** 好观点。最简单的信号是 User-Agent —— Slack 的展开器、Twitter 的卡片拉取器、Facebook 的预览机器人都标识自己。我们用 `is_bot` 标志标记这些点击，滚动到一个单独的"预览"计数器，不计入用户可见的点击数。对看起来像机器人的未知 UA —— 单 IP 高 RPS、无 referrer、HEAD 请求 —— 我们会启发式分类。

**面试官：** HEAD vs GET 呢？

**候选人：** 很多展开器用 HEAD 只拉取头。我们可以像 GET 那样用 302 响应 HEAD，但标记为非点击。Location 头仍让接收方在最终做 GET 时去他们想去的地方。

**面试官：** 点击管道本身 —— 它的容量是多少？

**候选人：** 峰值每秒 4 万事件，平均载荷大概 200 字节 —— 8 MB/秒的摄取。Kafka 在单 partition 上不费劲就能处理，但我会按 code 分区以启用并行消费。8 个 partition 给我们 10 倍增长的冗余。

**面试官：** 为什么按 code 分区而不是按时间戳？

**候选人：** 因为下游聚合按 code 滚动统计。如果我们按时间戳分区，每个消费者都会看到每个 code 的事件，就需要跨 partition 协调才能聚合。按 code 分区意味着每个 partition 的消费者各自维护自己的聚合切片，无需协调。

**面试官：** *（看时间）* 好。失败场景。带我走一遍如果主 DB 宕了会发生什么。

**候选人：** 写下线，直到故障切换完成。读继续从副本和缓存提供。一旦副本被提升为主 —— 通过 Patroni 或类似自动化 —— 写就恢复。整个窗口对托管 Postgres 应该不到一分钟。

**面试官：** 用户影响是什么？

**候选人：** 现有重定向：零，因为读从缓存和副本提供。新缩短：在故障切换窗口里返回 503 失败。客户端应该带退避重试。我们可以在队列里缓冲写请求，但那会带来最终一致的意外。

**面试官：** 如果整个区域宕了呢？

**候选人：** 用单一全局计数器实现多区域 active-active 很难，但可行。两种做法。一：每区域计数器空间，让每个区域从计数器范围的不相交子集分配自己的 code。二：全局完整读复制，但写去主区域；主区域失败时，提升副区域。

**面试官：** 第二种方法的 RPO 和 RTO 是多少？

**候选人：** RTO —— 恢复时间 —— 取决于你的故障切换自动化能多快提升副本。用 Patroni 或类似工具、经过良好测试，可以亚分钟。RPO —— 数据丢失 —— 取决于复制模式。异步复制意味着你最多丢到复制延迟那么多，区域间是几百 ms；在我们的规模下是几十到几百次写入。同步复制把 RPO 清零，但给每次写加上一次往返，会把写延迟从毫秒推到几百 ms。

**面试官：** 你会选哪个？

**候选人：** 每区域计数器空间。它给我们 write-local-read-anywhere，热路径上没有跨区域写。代价是更多运维复杂度 —— 每个区域需要自己的分配器，我们要跨区域合并分析。但对全球用户的延迟收益是显著的。

**面试官：** 好。讲讲那个延迟收益 —— 全球读是怎么工作的？

**候选人：** 每个区域有自己的 Postgres 副本集，从主区域异步复制喂养。缓存对每个区域是本地的。重定向通过 anycast 或 geo-DNS 到最近的区域。所以东京的用户打东京的 LB、东京的缓存、东京的副本 —— 从不跨太平洋。

**面试官：** 区域间的复制延迟是多少？

**候选人：** 跨区域复制通常 100-500ms。对短链接没问题 —— 用户在圣保罗创建一个链接并在 Twitter 上分享；孟买的接收方可能错过复制的第一秒，但等推文被读到时链接已经传播过去了。我们也可以在创建后头几秒全局 write-through 到缓存，以掩盖延迟。

**面试官：** *（记录）* 我想再推一件事。热 key。如果单个短 URL 病毒式传播得到 10 万 rps 呢？

**候选人：** 单个 Redis key，单个 shard。热 key 问题。几个缓解：跨多个 Redis 节点复制热 key，让应用服务器随机选一个；或把热值推送到应用服务器的本地内存，带短 TTL —— 比如 1 秒 —— 让大多数请求甚至不打 Redis；或卸载到带边缘缓存的 CDN，因为响应只是带 Location 头的 302。

**面试官：** 你怎么首先检测热 key？

**候选人：** 采样。每个应用服务器跟踪一个访问 code 的 count-min sketch；定期 —— 每几秒 —— 它发出 top-K。中央聚合器合并 sketch，识别超过阈值的 key，然后触发热 key 路径：钉到本地缓存、跨 Redis 节点复制，或推到 CDN。从检测到缓解的循环在调优良好的系统里是亚秒级。

**面试官：** Count-min sketch —— 为什么不就用 hash map？

**候选人：** 内存。每服务器 1 亿 code，完整 hash map 巨大。Count-min sketch 用精确计数换有界内存 —— 几 MB 给你近似频率，带可控误差。对热 key 检测我们只关心分布的顶部，那里 sketch 是准确的。

**面试官：** CDN 对 302 重定向有用吗？

**候选人：** 设了 Cache-Control 头，CDN 可以缓存 302 —— 大多数现代 CDN 支持。缺点是你失去被缓存请求的每点击分析。对非常热的链接，你会接受这个取舍，或在边缘做采样分析。

**面试官：** 本地应用服务器缓存 —— 如果 URL 变了不会有一致性问题吗？

**候选人：** 短码按设计是不可变的 —— 一旦发出，长 URL 永不变。所以本地缓存安全；唯一的失效是删除（链接过期或撤下），我们可以在 Redis pub/sub 频道上广播失效。本地缓存 TTL 几秒，意味着最坏情况一个已删除链接被服务几秒钟。

**面试官：** 等等，短码真的不可变吗？过期呢？

**候选人：** 过期是软删除 —— code 仍存在，但我们返回一个"此链接已过期"页面而不是重定向。检查时我们可以在缓存值里带上 `expires_at`，让应用服务器跟挂钟比较，这样过期不需要缓存失效。

**面试官：** 好捕捉。好，最后一个领域。安全。威胁模型是什么？

**候选人：** 几件要防的事。一：恶意长 URL —— 恶意软件、钓鱼。防御：跟 Google Safe Browsing 或类似服务集成；在重定向前拒绝或警告。二：开放重定向滥用 —— 攻击者用我们的短链接服务在钓鱼邮件里伪装恶意 URL。同样的防御，加上对缩短请求的基于声誉的限流。三：枚举 —— 有人遍历短码来收集 URL。防御：按 IP 限流、用打乱的计数器，考虑不公开暴露自定义分析端点。四：自定义别名创建上的暴力破解 —— 限流加 CAPTCHA。

**面试官：** TLS、密钥、所有那些呢？

**候选人：** TLS 在 LB 终止，内部流量可选 mTLS。分配器和 DB 的密钥放在密钥管理器里 —— Vault 或云等价物。应用服务器上无长期凭证，只用短期 token。

**面试官：** 日志呢？访问日志本身能泄漏 URL 吗？

**候选人：** 是 —— 每次重定向记录 code 没问题；但如果我们曾经记录长 URL（比如在调试分支里），那就是隐私泄漏。长 URL 可能包含会话 token、密码重置链接，或其他敏感路径。策略：从不在 INFO 级别记录解码的长 URL。如果调试需要记录，要么哈希它们，要么激进截断。

**面试官：** 你想过 GDPR 风格的删除吗？

**候选人：** 是。如果用户有账户并请求删除，我们需要从 links 表和分析事件中擦除他们的 owner_id。对分析，我们会用一个跨 Kafka 和 ClickHouse 的 delete-by-owner_id 任务。短 URL 本身可以留下 —— 它孤立来看不是个人数据 —— 但与用户的关联被移除。

**面试官：** 对缩短端点的限流呢？

**候选人：** 每 API key 令牌桶，加上对未认证流量的每 IP 全局上限。缩短端点容易被滥用；我们会分层 —— 匿名用户每天 100 个，认证每天 10K，付费在合理使用内无限。Redis 里的限流状态用原子 INCR + EXPIRE。

**面试官：** 好。*（看笔记）* 我们还有大约五分钟。如果你有更多时间会加什么？

**候选人：** 几件事。*（思考）* 一：真正的滥用检测管道 —— 对缩短模式做 ML 模型，在钓鱼团伙病毒式传播前抓住他们。二：分层服务 —— 免费用户拿 7 字符计数器 code 加共享限流；付费用户拿来自保留命名空间的短码、自定义域名和 SLA 支持的分析。三：链接版本化 —— 发出后更新长 URL 的能力，用于营销活动。它打破了不可变假设，但是客户的诉求。我们要加缓存失效，可能把本地缓存 TTL 降下来。

**面试官：** 链接版本化最棘手的部分是什么？

**候选人：** 跨区域缓存失效。一旦我们允许可变性，每一层缓存都必须失效，我们就回到缓存一致性问题。最简单的方案是让新长 URL 带短暂延迟生效 —— 比如"此更新将在 30 秒内生效" —— 靠 TTL 过期而不是主动失效。

**面试官：** 你能改用显式失效吗？

**候选人：** 可以。在 pub/sub 总线上推一条失效消息 —— Redis pub/sub 或 Kafka —— 每个缓存订阅者丢掉 key。对 Redis 和本地应用服务器缓存都有效。难点是让它可靠：如果一个订阅者错过消息，它就会服务陈旧值。所以你会把 pub/sub 跟短 TTL 作为兜底结合起来。

**面试官：** 短 TTL 的代价是什么？

**候选人：** 更高的缓存未命中率，DB 上更多负载。如果我们从 24 小时降到比如 5 分钟，对长尾内容的命中率会明显下降。所以链接版本化有真实成本 —— 不是免费的。我们可能会把它放在付费层后面，并把取舍写进文档。

**面试官：** 好。还有什么希望你有时间做的吗？

**候选人：** 我想在分析管道上花更多时间 —— 你如何建模漏斗、地理分解、referrer 链有很多结构。我也一点都没谈部署、监控、容量规划。那些通常是另一个完整的面试。

**面试官：** 最后一个问题。如果你必须选这个设计中最大的风险，会是什么？

**候选人：** *（思考）* 计数器分配器。它是单一争用点，也是写路径上的协调依赖。如果它有不顺的一天 —— 慢 Postgres 行、锁争用、分配器服务抖动 —— 每个写都受影响。它通过范围和分片缓解得很好，但仍是我在生产中会最仔细盯的地方。

**面试官：** 你会怎么监控它？

**候选人：** 分配器请求延迟 p50/p99、范围耗尽率、新范围分配时间、分配器可用性。p99 超阈值或任何可用性抖动都告警。再加混沌测试 —— 定期在 staging 杀掉分配器，验证应用服务器用现有范围服务、优雅降级。

**面试官：** 公平。好，时间到。干得好。

**候选人：** 谢谢。挺有意思的。

## 做得好的地方

- 一开始有力的澄清问题阶段 —— 在深入之前锁定了读写比、延迟预算、code 长度。
- 在被推问写时读取成本时，从基于哈希的 code 干净利落地转向基于计数器的。承认每种方法的权衡，而不是坚持第一个答案是对的。
- 通篇使用具体数字：400 写/秒、4 万读/秒、95% 缓存命中率、64 个 shard。数字让设计落地并传递经验信号。
- 平稳处理了关于热 key、缓存击穿和多区域的追问。展示了对经典模式的熟悉，又不显得机械。
- 在没有每个都被提示的情况下，明确考虑了失败模式（Kafka 宕、Redis 宕、DB 主宕、整个区域宕）。

## 可以更强的地方

- 计数器分配器有点被掩盖；本可以详细画出范围请求和崩溃在线上具体是什么样子。
- 在提议计数器方案时没有最初提到去重索引 —— 只有在面试官追问去重损失后才加。最好预见权衡。
- 301 vs 302 的权衡正确但本可以更尖锐地框定为业务决策（分析收入 vs 服务成本）。
- 自定义别名命名空间冲突是临场解决的；更强的候选人会更早提出它作为已知问题。
- 在"code 不可变性"问题上短暂绊倒 —— 恢复得很好，但暗示候选人没有完全想透过期语义。

## 关键要点

- 对高读系统，缓存命中率主导一切。围绕 95%+ 设计，其余架构自然跟随。
- Base62 + 7 字符给 ~3.5 万亿 code —— 对任何现实规模都绰绰有余。不要不做数学就选 5 或 6 字符。
- 在规模上基于计数器的 code 生成比基于哈希简单，但如果 URL 去重是一个特性，带一个去重索引。
- 通过每区域计数器空间，对读重工作负载多区域 active-active 可实现 —— 保持写本地并容忍短暂跨区域复制延迟。
- 永远为每个组件明确处理"如果 X 宕了"的问题。失败模式是初级候选人丢分的地方。

## 参考架构

```
                          +----------------------+
                          |  Clients / Browsers  |
                          +----------+-----------+
                                     |
                              Anycast DNS
                                     |
                +--------------------+--------------------+
                |                                         |
         +------v------+                          +-------v------+
         |  Region A   |                          |   Region B   |
         |  (us-east)  |                          |  (eu-west)   |
         +-------------+                          +--------------+
                |                                         |
        +-------v-------+                         +-------v-------+
        |  L7 Load Bal  |                         |  L7 Load Bal  |
        |  TLS, rate-lim|                         |  TLS, rate-lim|
        +-------+-------+                         +-------+-------+
                |                                         |
        +-------v-------+                         +-------v-------+
        | App Servers   |                         | App Servers   |
        | (stateless)   |                         | (stateless)   |
        +---+-------+---+                         +---+-------+---+
            |       |                                 |       |
            |       +-----> Counter Allocator        |       +-----> Counter Allocator
            |               (counter space A)         |               (counter space B)
            |                                         |
    +-------v-------+                         +-------v-------+
    |  Redis Cluster|                         |  Redis Cluster|
    |  (code -> URL)|                         |  (code -> URL)|
    +-------+-------+                         +-------+-------+
            |                                         |
    +-------v-------+                         +-------v-------+
    | Postgres      |                         | Postgres      |
    | Primary +     | <----- async repl ----> | Primary +     |
    | Read Replicas |                         | Read Replicas |
    | (64 shards)   |                         | (64 shards)   |
    +-------+-------+                         +-------+-------+
            |                                         |
            +-----> Kafka (clicks) -----> Flink -----> ClickHouse
                                                       (analytics)
```

热路径延迟（缓存命中）：~1-3ms 服务器时间，~15-65ms 挂钟时间。
冷路径（缓存未命中）：+5-10ms 用于 DB 读副本查找。
缓存命中率目标：95%+，通过幂律流量倾斜和 LFU 淘汰实现。
Shard 数：64（在当前规模每个 ~50GB 工作集，带冗余）。
区域故障切换：通过 Patroni 自动化，亚分钟 RTO。
