# 模拟面试：设计分布式限流器

**类型：** 系统设计
**难度：** L4 / L5
**时长：** 45-60 分钟（模拟）
**标签：** #system-design, #api-gateway, #redis, #token-bucket, #sliding-window

## 背景

候选人正在面试 API 平台团队的高级/Staff 工程师岗位。面试官是一位 L6，负责公司的 API 网关。题目是设计一个分布式限流器，放在公司公开 API 前面。候选人有 50 分钟；面试官计划最后 5-10 分钟深挖所选择的算法，并讨论在极高规模下会出什么问题。

## 对话记录

**面试官：** 我希望你设计一个限流器 —— 分布式的，坐在 API 网关层，保护后端服务免于过载并强制每客户的配额。你想怎么发挥都可以。

**候选人：** 明白。让我澄清一下我们在构建什么。"限流器"是指单个组件，还是我们要围绕它设计整个 API 网关？

**面试官：** 限流器作为一个组件，加上它如何与网关集成。不是整个网关，但足够展示请求如何流过。

**候选人：** 然后"每客户配额"—— 粒度是什么？每 API key、每 IP、每用户、每端点？

**面试官：** 上述全部，按规则可配置。客户应该能够定义多个限流规则 —— 例如，"每 IP 每秒 100 个请求、每 API key 每天 10K 个、每 IP 在 /login 端点上每秒 5 个"。

**候选人：** 好。所以是一个建在计数系统之上的规则引擎。规模呢？

**面试官：** 通过网关的总流量：每秒几十万请求，峰值大约 100 万。数万客户，每个平均大约 10 条规则。

**候选人：** 延迟预算呢？

**面试官：** 这是关键问题。限流器在每个请求的关键路径上。我们需要个位数毫秒的添加延迟，理想 p99 低于 2ms。

**候选人：** 很紧。*（思考）* 好。限流违规时会发生什么？

**面试官：** 返回 HTTP 429 Too Many Requests，带 Retry-After 头。可选地在所有响应上包含 `X-RateLimit-Limit`、`X-RateLimit-Remaining`、`X-RateLimit-Reset` 头。

**候选人：** 一致性要求呢 —— 如果客户的限制是 100/秒，偶尔放过 105 个可以吗，还是必须精确？

**面试官：** 可以有一些超调 —— 比如在 5% 以内。重点是保护下游免于过载，而不是精确计费。对付费配额层级强制更重要，但即使那里小缓冲也可以。

**候选人：** 这相当地改变了设计 —— 意味着我们在某些情况下可以用最终一致的计数器。好。还有其他需求吗？审计、调试、动态配置更新？

**面试官：** 动态配置：是的，客户通过管理 API 更新规则，变更应在一分钟内生效。决策审计日志：对前 1% 的客户或违规情况 nice-to-have，不是每个请求。调试：客户应该能看到他们当前的消耗。

**候选人：** 明白。让我先思考算法，再思考架构。

**面试官：** 开始吧。

**候选人：** 四个经典的限流算法：固定窗口、滑动窗口日志、滑动窗口计数器、令牌桶。让我勾勒一下权衡。

*（在白板上写）*

```
固定窗口：       计数每个 [now / window] 桶中的请求。
                 简单。边界问题：窗口边缘 2 倍突发。

滑动窗口日志：   存储最近 N 秒里每个请求的时间戳。
                 精确。昂贵：O(requests) 内存和写入。

滑动窗口计数器： 用流逝时间分数加权当前和上一个固定窗口。
                 近似但便宜。
                 内存：每规则 2 个计数器。

令牌桶：         令牌以速率 R 累积，容量 B。每个请求
                 消耗 1。允许 B 以内的突发；长期限速到 R。
                 状态：(tokens, last_refill_time)。
```

**面试官：** 你会选哪个？

**候选人：** 令牌桶作为默认。它给出最好的 UX —— 短暂的突发没问题，持续超速则限速 —— 这是客户实际期望的。滑动窗口计数器是一个不错的备选，如果我们想要严格的"每窗口不超过 X"。

**面试官：** 为什么不用固定窗口？

**候选人：** 边界问题。如果客户有 100/分钟，他们可以在 11:59:59 发 100 个，在 12:00:01 又发 100 个 —— 那是 2 秒内 200 个请求，按算法技术上在限制内但是真正的过载。滑动窗口或令牌桶平滑了这一点。

**面试官：** 滑动窗口日志 —— 你什么时候用？

**候选人：** 低速率、高精度的规则。比如"每账户每小时 5 次密码重置" —— 那里按事件的审计日志本身就有用，存储成本也微不足道。

**面试官：** 好。给我看令牌桶的数学。

**候选人：** 每规则状态：`(tokens: float, last_refill_ts: timestamp)`。请求时：

```
now = current_time()
elapsed = now - last_refill_ts
tokens = min(capacity, tokens + elapsed * refill_rate)
if tokens >= 1:
    tokens -= 1
    last_refill_ts = now
    ALLOW
else:
    DENY (return Retry-After = (1 - tokens) / refill_rate)
```

关键洞察：我们不跑后台定时器来加令牌。我们在请求时计算懒惰补充。每规则两个数字的状态。

**面试官：** 那状态住在哪里？

**候选人：** 这是核心问题。选项。一：每个网关节点上的内存。快，但流量分布到很多网关时不准确。二：像 Redis 这样的集中式存储。一致但增加延迟和单点。三：混合 —— 本地缓存加定期同步。

**面试官：** 让我推你做决定。你会从什么开始？

**候选人：** Redis 后端。大多数工作负载能容忍 0.5-1ms 的 Redis 往返，并且 Redis 可以从容处理每秒数十万操作。如果我们触到 Redis 上限，我们会加本地缓存层；我们不会从复杂性开始。

**面试官：** 走一遍一个请求。

**候选人：** *（画）*

```
Client request
   |
   v
[L4 LB]
   |
   v
[API Gateway Node]
   |
   |--> 识别 principal（API key、IP、user_id）
   |--> 查找适用规则（缓存在节点内存）
   |--> 对每条规则：
   |       Redis 里的 LUA 脚本：
   |         GET (tokens, last_refill_ts)
   |         计算新 tokens
   |         IF allow: tokens -= 1, SET, return ALLOW
   |         ELSE: SET (不递减), return DENY
   |--> 如果任何规则拒绝：返回 429
   |--> 否则：转发到后端
   |
   v
[Backend]
```

**面试官：** 为什么用 Lua 脚本而不是 pipelined GET + SET？

**候选人：** 原子性。读-改-写必须是一个原子操作，否则两个并发请求都看到相同的 tokens，都递减，我们就少算了。Lua 在 Redis 服务端运行，相对同一个 key 上的其他命令是单线程的。

**面试官：** Redis cluster 呢 —— 它影响 Lua 方法吗？

**候选人：** 一个 Lua 脚本中接触的所有 key 必须在同一个 Redis slot 上。我们确保每个脚本接触一个限流 key。如果客户有多条规则，每条是一次单独的脚本调用，可能到不同的 shard。我们会 pipeline 它们以最小化往返。

**面试官：** 最坏情况下每个请求多少次往返？

**候选人：** 每条规则一次。每客户平均 10 条规则，所以每个请求最多 10 次 Redis 操作。100 万 rps 时，那是 1000 万 Redis 操作/秒 —— 分布到 Redis 集群上，高但可行。

**面试官：** 那延迟预算可以吗？

**候选人：** 如果所有规则在同一个 shard 上一次往返 pipelined，总共 ~1ms。如果它们击中不同的 shard，我们并行发出并等最慢的，仍然 ~1ms。都在我们的 2ms 预算内。

**面试官：** 如果 Redis 慢或宕了呢？

**候选人：** 选择是 fail-open 还是 fail-closed。Fail-open：如果 Redis 不可用，网关让请求通过。风险：Redis 故障期间后端被压垮。Fail-closed：拒绝所有请求。风险：一次 Redis 故障让整个 API 下线。

**面试官：** 哪个？

**候选人：** Fail-open 是限流器的典型默认，因为 fail-closed 的失败模式是灾难性的 —— 一次 Redis 抖动让一切下线。但我们会结合本地兜底：每个网关维护一个粗粒度的本地限流，只用它自己观察到的流量。所以在 Redis 故障期间，我们不是没保护，只是不那么准确。

**面试官：** 多说说本地兜底。

**候选人：** 每个网关在进程内跑一个每 (principal, rule) 的令牌桶。本地速率按假设请求分散在 N 个网关上来缩放 —— 所以如果全局限制是 1000/秒、我们有 10 个网关，每个网关本地许可 100/秒。Redis 故障期间，我们把本地限制作为硬上限。流量在网关间不均匀分布会有一些超调，但不会全面崩盘。

**面试官：** 如果一个客户因为粘性路由只击中一个网关呢？

**候选人：** 那他们的本地限制是 100/秒而不是 1000/秒 —— 我们对他们限流得比应该的多。他们在故障期间收到 429。这是我们为 Redis-down 健壮性接受的权衡。我们也可以基于观察到的历史分布预分配"份额配额"。

**面试官：** 好。讲讲规则引擎。规则怎么加载？

**候选人：** 客户通过管理 API 定义规则。规则存在配置数据库里 —— Postgres。网关定期轮询配置服务的变更（每几秒），或订阅一个变更流。

**面试官：** 大规模轮询 —— 每个网关每几秒轮询是在压垮配置服务？

**候选人：** 是的 —— 1000 个网关每 5 秒轮询，那是 200 rps 到配置，可管理但浪费。更好：一个配置推送服务。网关开一个流式连接到配置服务；新规则被推送。或者用 Kafka 或 NATS 这样的 pub/sub 做配置事件。

**面试官：** 拉 vs 推 —— 哪个更有韧性？

**候选人：** 拉更有韧性，因为网关是主动方 —— 如果连接断了，它就再轮询。推需要连接管理和重连逻辑。我会为效率构建推，但带一个每几分钟跑一次的拉兜底来捕获丢失的更新。

**面试官：** 规则在网关上住在哪里？

**候选人：** 内存数据结构，按 principal 索引以快速查找。请求到来时，我们识别 principal（API key、IP、JWT 中的 user_id）并查找匹配的规则。查找必须亚微秒。

**面试官：** 你怎么处理飞行中的规则变更？

**候选人：** 原子交换。规则是不可变快照；更新时，网关构建一个新快照并原子地交换指针。飞行中的请求使用它们开始时当前的版本。标准的 read-copy-update 模式。

**面试官：** 好。让我推一下 Redis key 设计。一条规则的 key 是什么？

**候选人：** 类似 `rl:{principal}:{rule_id}` —— 例如，`rl:apikey-abc123:per-second-100`。Principal 限定计数器的范围；rule_id 区分同一 principal 上的多条规则。

**面试官：** 值是什么？

**候选人：** 一个小 struct：tokens 和 last_refill_ts。存储为序列化形式或两个单独的 Redis key。我会用 Redis hash：`HMSET key tokens ts`。然后 Lua 脚本做 HGET、计算、HMSET。

**面试官：** TTL 是多少？

**候选人：** TTL 设得比桶的补充时间长。对一个 100/秒的规则，几秒不活动后桶反正是满的，所以状态可重建。我们可以把 TTL 设为，比如，10 分钟；之后 key 消失，下一个请求从满桶开始。

**面试官：** 存储占用是多少？

**候选人：** 每条活跃规则：包括 Redis 开销约 100 字节。10K 客户 * 每个 10 条规则 = 10 万活跃 key，那是 10MB。可忽略。即使按 IP 规则有更高基数 —— 比如 100 万活跃 IP —— 那是 100MB。一个 Redis 节点从容容纳。

**面试官：** 规则变更时的 TTL 呢？

**候选人：** 如果客户更新规则的容量或补充速率，Redis 里运行中的令牌桶状态现在跟旧规则语义关联。我们有几个选项。最干净的：在 key 里包含规则版本 —— `rl:apikey-abc:rule-7:v3`。规则更新时，版本递增，旧 key 自然过期。状态重置为一个新桶，这是正确的，因为新规则有不同的参数。

**面试官：** 这不会导致一个短暂的过度允许窗口吗，因为版本递增时桶立刻是满的？

**候选人：** 是的 —— 一个桶补充周期。缓解：首次访问时用旧桶的剩余容量（按新容量缩放）初始化新桶。增加一点复杂度但平滑过渡。

**面试官：** Redis 本身的扩展呢？

**候选人：** Redis Cluster 按 hash 分片 key。限流操作 100K rps，单个 Redis 节点处理 ~10 万操作/秒，我们至少要 3-5 个 shard 以留冗余。100 万 rps 每请求 10 个操作，1000 万 Redis 操作/秒 —— 那是 100+ 个 shard。

**面试官：** 那是很多 Redis。有办法减少负载吗？

**候选人：** 有 —— 本地近似加定期对账。每个网关持有一个本地令牌桶；请求时本地递减。每 N 个请求或每 T 毫秒，把本地消耗同步到 Redis。全局状态滞后，但对大多数工作负载滞后可接受。让我们摊销 Redis 命中。

**面试官：** 多说说 —— 协议是什么？

**候选人：** 每个网关跟踪每 (principal, rule) 的 `(local_consumed, last_sync_time)`。每第 N 个请求或每 T 毫秒，网关用批量消耗调用 Redis：按批量计数 DECRBY，拿到新的全局剩余。如果剩余为负或低，减少或停止本地配额直到下次同步。否则为下个间隔补充本地配额。

**面试官：** 权衡是什么？

**候选人：** 较低的准确性。如果客户突然从 0 到 1000 rps，本地桶可能允许一些突发，在同步赶上之前。超调与同步间隔成比例。1 秒同步间隔和 1000 rps 限制，最坏情况超调是 1000 个请求。那是 100% 超调 —— 不够好。所以这个技巧对高速率限制更好，对紧配额不理想。

**面试官：** 那你什么时候用它？

**候选人：** 对粗限制 —— 比如"每小时 100 万请求" —— 同步间隔相对于窗口很小。或者对 IP-based DDoS 保护，那里精确度比总体吞吐量保护重要性低。

**面试官：** 好。换一下，深入滑动窗口计数器，因为我想确保你理解权衡。

**候选人：** 好。算法：维护两个计数器 —— 当前窗口和上一个窗口 —— 两者都在固定边界上（例如，每分钟）。时间 t 的请求时，计算：

```
weight = (window_size - elapsed_in_current) / window_size
estimated_count = previous_count * weight + current_count
if estimated_count < limit:
    current_count += 1
    ALLOW
else:
    DENY
```

它通过按上一个窗口与实际以"现在"结束的滑动窗口的重叠多少来加权上一个窗口的计数来近似滑动窗口。

**面试官：** 为什么是近似？

**候选人：** 因为我们不知道上一个窗口内的请求什么时候来的。我们假设均匀分布，这很少为真。如果上一个窗口的所有请求都接近末尾来，实际当前负载比估计高，我们就限流不足。反之如果在开始来，我们就过度限流。

**面试官：** 误差多严重？

**候选人：** 由上一个窗口的计数有界。对高度突发流量最坏情况两个方向各 ~50% 偏差，但实践中误差很小。对大多数工作负载，每规则 2 个计数器就是很好的近似。

**面试官：** 你什么时候选滑动窗口而不是令牌桶？

**候选人：** 当客户用"每分钟 X"的方式思考且我们想紧密强制时。令牌桶允许容量 B 以内的初始突发；滑动窗口不允许。

**面试官：** 如果你想要两者 —— 突发容忍和严格窗口限制呢？

**候选人：** 多条规则。客户有一条令牌桶规则（补充 100/秒，容量 200）用于突发容忍，还有一条滑动窗口规则（每分钟 10K）用于窗口上限。两者都必须允许；任一拒绝则请求被拒绝。

**面试官：** 好。*（记录）* 现在讲讲边缘情况。网关节点间的时钟偏差呢？

**候选人：** 每个网关用本地时钟在 Lua 脚本中计算时间戳 —— 等等，不，脚本在 Redis 里跑。所以 Redis 的时钟是权威。只要 Redis 在 NTP 上，时钟偏差在毫秒内。

**面试官：** 如果你有多个 Redis shard，每个有自己的时钟呢？

**候选人：** 每个 shard 拥有自己的 key；不同 shard 上的规则有独立的时钟。NTP 让它们保持在几毫秒内，在我们可接受的准确度内。我们不需要跨 shard 时间协调。

**面试官：** 如果一个 Redis 节点有时钟跳跃 —— NTP 校正呢？

**候选人：** 向后跳跃会让一些本不该的请求短暂通过。向前跳跃可能错误地补充桶。两者都是瞬态的。我们会监控时钟异常并告警。在 Lua 里，我们可以从 Redis 自己读 `TIME`，那是 shard 内最一致的。

**面试官：** 好。一个有数十亿不同 IP 的客户呢 —— 比如来自僵尸网络的 DDoS 攻击？

**候选人：** 那种规模的按 IP 规则让 Redis 状态爆炸。每个 IP 是一个 key，带一些开销。防御：限制基数。如果规则是按 IP 的，我们把 IP hash 到一组有界的桶 —— 比如 10 万 —— 并按桶限流。这失去了每 IP 公平性但限制了内存，并保护免受状态爆炸攻击。

**面试官：** 这不会伤害与攻击者共享桶的合法用户吗？

**候选人：** 是的，这是权衡。更好的 DDoS 防御在上游 —— CDN、WAF，或限流器之前基于 ML 的机器人检测。限流器本身不应该是 DDoS 的前线；它在保护后端免于逻辑过度使用，而不是对抗性洪流。

**面试官：** WAF 在你的架构里放哪里？

**候选人：** *（画）*

```
[CDN / Anycast]
   |
   v
[L4 LB]
   |
   v
[WAF / DDoS protection]   <-- IP 信誉、签名检测、CAPTCHA
   |
   v
[L7 LB]
   |
   v
[API Gateway + Rate Limiter]
   |
   v
[Backend]
```

WAF 丢掉明显的坏流量。限流器处理每客户配额执行。

**面试官：** 好。*（看时间）* 思考一下可观测性。你暴露什么指标？

**候选人：** 每规则：允许的请求、拒绝的请求、当前消耗、补充速率。每客户：总请求、总拒绝、按消耗排名的顶部规则。网关范围：Redis 延迟 p50/p99、Lua 脚本执行时间、本地兜底激活率。

**面试官：** 每客户可见性呢？

**候选人：** 客户通过管理 API 或仪表板看到自己的消耗。数据来自同样的 Redis 状态（或其快照） —— 我们可以按百分比暴露每规则的 `current_tokens / capacity`。

**面试官：** 响应头呢 —— 你在每个响应上设 X-RateLimit-Remaining 吗？

**候选人：** 是的 —— 但我们从做检查的同一个 Lua 脚本读它，所以没有额外的往返。Lua 脚本在一次调用中返回 `(allowed, remaining, reset_time)`。网关只是把它们复制到头里。零额外成本，客户端可以自我节流。

**面试官：** 调试呢 —— 一个客户说"我收到 429 但我在限制以下"。你怎么调查？

**候选人：** 把拒绝的样本记录下来，带触发的规则、principal、当前桶状态。采样而不是每个拒绝 —— 1% 采样或对头部客户始终记录。有日志，你可以重建发生了什么。我们也会暴露一个调试端点："对这个 principal，给我看现在所有匹配规则的状态"。

**面试官：** 那些拒绝日志去哪里？

**候选人：** Kafka -> 一个查询友好的存储如 ClickHouse 或 Elasticsearch。按 (principal, rule, time) 索引以快速查找。保留 30 天左右，取决于存储预算。

**面试官：** *（记录）* 让我再探一件事。你说 1ms p99 延迟。什么让你达到？

**候选人：** 几件事。一：Redis 必须跟网关在同一个网络上 —— 同一 DC，理想同一机架。往返 0.2-0.5ms。二：多条规则适用时 pipeline。三：Lua 脚本保持读-改-写原子性，所以我们不为争用的乐观重试买单。四：本地兜底摊销以减少高速率 principal 的 Redis 调用。

**面试官：** 具体到 p99 —— 是什么造成尾部？

**候选人：** Redis GC 暂停、网络抖动、热 key 上的 Lua 脚本争用、偶尔的 Redis 集群 slot 迁移。最大的尾部风险是热 key 争用 —— 如果一个客户发送的请求太多以至于他们的桶 key 成为热 Lua 脚本目标，Redis 串行化它们，延迟上升。

**面试官：** 你怎么处理？

**候选人：** 把热 key 分散到多个 shard：不是一个 key 对应客户的一条规则，而是把请求 hash 到 K 个子 key 之一。每个子 key 持有 1/K 的配额。客户看到稍微不同的限流行为（K 个 shard 间的公平性），但没有单个 key 是瓶颈。

**面试官：** 这是自动的还是手动的？

**候选人：** 对超过流量阈值的客户自动。我们监控每 key 请求速率；当 key 超过，比如，10K rps，我们把它提升到分片。客户不需要知道 —— 网关抽象了它。

**面试官：** 跨数据中心的分布式配额呢？

**候选人：** 跨 DC 限流很难，因为 DC 间往返对每请求关键路径太慢。两种方法。一：每 DC 配额（把全局限制分到各 DC） —— 更简单但流量不均匀分布时不公平。二：最终一致的全局计数器加本地近似 —— 每个 DC 基于其份额强制本地上限，定期与其他 DC 对账。

**面试官：** 哪个？

**候选人：** 每 DC 配额作为默认；仅对关心的高层级客户切换到全局最终一致。全局一致性增加复杂度和滞后；大多数客户的限流足够粗，每 DC 拆分就够了。

**面试官：** 如果一个 DC 宕了 —— 客户能拿回他们的配额份额吗？

**候选人：** 能 —— 一个协调者（或客户的管理界面）重新分配配额。如果 DC-A 宕了，持有客户配额的 30%，那 30% 应该流向剩余的 DC。我们会有一个控制平面监视 DC 健康并重新平衡配额。

**面试官：** 分区时 DC 互不可达但都在线呢？

**候选人：** 每个 DC 继续用其现有份额。分区期间总服务可能超过全局配额，因为两个 DC 都独立运行并假设其他健康。短期内可接受；分区愈合后，控制平面对账，可能临时降低份额来补偿。

**面试官：** 好。让我推你思考误用。客户怎么能玩弄系统？

**候选人：** *（思考）* 几种方式。一：在 Redis 故障期间分散他们的流量到多个网关，大幅超过每网关本地兜底。缓解：本地兜底不应太宽松 —— fail-open 带硬上限，不是无界。二：如果我们批量本地消耗同步，发送巨大突发可以在同步赶上前落地。缓解：有界本地缓冲 —— 一旦本地消耗超过网关的份额，本地拒绝，不等 Redis。

**面试官：** 还有吗？

**候选人：** 更微妙的一个：如果客户可以配置规则，恶意客户可能定义一条成本非常低的规则 —— 比如"每小时 100 万亿请求" —— 有效地禁用限流。缓解：基于层级的规则配置，每层级带硬最大值。

**面试官：** IP 方面呢？

**候选人：** TCP 上的 IP 欺骗不控制网络路径就不太可能，但从云提供商轮换 IP 是平凡的。所以每 IP 限制应该聚合到云范围的每 /24 或每 /16。我们维护已知云范围的列表，并在那里应用更广泛的聚合。裸 IP 限流主要对单机误用有效，不是分布式误用。

**面试官：** *（看时间）* 最后一个话题。如果你有更多时间会加什么？

**候选人：** 几件事。*（思考）* 一：更智能的算法 —— 自适应限流，下游服务健康时提升，受压时降低。我们会与后端健康信号集成。二：基于成本的限流 —— 不同端点有不同成本（搜索查询昂贵，健康检查便宜），所以我们按请求收取"权重"而不只是计数。

**面试官：** 多说说基于成本的。

**候选人：** 每个端点有配置的成本；限流器从桶中扣除该成本。所以搜索查询可能花 10 个令牌，简单的 GET 可能花 1 个。桶容量和补充速率以令牌表示。客户配额以令牌而不是请求计价。与现有令牌桶实现自然配合 —— 只把"递减 1"改为"递减 cost"。

**面试官：** 第三件呢？

**候选人：** 一个自助规则游乐场 —— 客户可以针对他们实际的近期流量测试"如果我有这条规则呢？"，看会拒绝多少请求。帮助他们调整限制而不必发布并破坏生产。

**面试官：** 好。还有什么希望你有时间做的吗？

**候选人：** 我想聊聊基于队列的限流 vs reject-with-429 模型。一些工作负载偏好排队（带最大队列长度和超时）而不是直接拒绝 —— 它以延迟为代价平滑突发。那是同一个问题的不同形态，值得建模。

**面试官：** 好，到此结束。表现不错。

**候选人：** 谢谢。限流器悄悄地很深 —— 每次对话都会发现新的边缘情况。

## 做得好的地方

- 一开始清晰、有条理的算法对比。在做决定前勾勒了所有四个经典算法及其权衡 —— 这展示了广度。
- 用合理的推理选了令牌桶作为默认，然后处理了关于何时滑动窗口或滑动窗口日志更好的追问。
- 在原子性讨论上强 —— 立即提到了 Redis 上的 Lua 脚本，并解释了为什么 pipelined GET+SET 是错的。
- 本地兜底讨论有微妙之处：fail-open 带有界本地上限，不是无界 —— 抓住了明显的反模式。
- 识别了热 key 问题并提出了 key 拆分作为缓解。
- 基于成本的限流是结尾一个干净的延伸 —— 表明候选人理解如何组合原语。

## 可以更强的地方

- 提出跨 DC 关切有点慢；面试官不得不引导。
- 带定期同步的本地缓存被正确识别为不精确，但本可以更清晰地说明确切何时使用它 —— 只在追问后才提到高速率/粗限制。
- DDoS 缓解在 WAF 处被掩盖 —— 本可以展示限流器本身能做什么的更多深度（每 /16 聚合、指数加权移动平均、CAPTCHA 升级）。
- 没有明确指出 Lua 脚本部署的运维关切 —— Redis 上的脚本变更很微妙。
- 滑动窗口计数器数学正确，但误差界限的解释有点模糊。

## 关键要点

- 对大多数工作负载，令牌桶是正确的默认 —— 它自然地容纳突发并维护便宜。
- 原子性很重要。读-改-写必须是一个原子操作；在 Redis 上意味着 Lua 脚本（或 Redis 7+ 函数）。
- Fail-open 带有界本地兜底。纯 fail-open（Redis 故障期间无保护）让后端淹没；纯 fail-closed 让 Redis 抖动把整个 API 拖下水。
- 热 key 是主要的 p99 风险。用采样检测，通过把 key 跨 shard 拆分来缓解。
- 跨 DC 限流最好避开热路径。用每 DC 配额加 DC 失败时的中央重新平衡。
- 基于成本的限流是令牌桶的小延伸，给出更丰富的语义 —— 推荐给任何端点成本异构的平台。

关键数据：
- 网关层峰值 100 万 rps
- 每请求平均 10 条规则 -> 1000 万 Redis 操作/秒
- 约 100 个 Redis 集群 shard 留冗余
- p99 添加延迟目标：<2ms
- Redis 不可用时本地兜底启动，带每网关有界上限

算法选择：
- 默认：令牌桶（懒惰补充，每 key 2 个数字）
- 严格窗口：滑动窗口计数器（每 key 2 个计数器，~5% 误差）
- 审计关键的低速率：滑动窗口日志
- 成本感知：带加权递减的令牌桶

失败处理：
- Redis 宕：fail-open 带每网关有界本地兜底
- 网关宕：流量通过 LB 转移到其他网关
- DC 宕：控制平面把每 DC 配额重新分配给幸存 DC
