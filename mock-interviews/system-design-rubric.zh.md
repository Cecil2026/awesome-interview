# 系统设计评分表 V1

一份可落地、区分级别的系统设计面评分表。用法：模拟面试后给自己（或别人）打分，读 transcript 时核对维度，或在面试中途当 checklist 提醒自己漏了什么。

评分表包含 **8 个维度**，每维 0–5 分，总分 40。**级别看的是阈值 + 分布**——高级别不是总分更高就行，而是每个维度的最低标准都抬高。

## 怎么用

1. 跑或读一场 45–60 分钟的系统设计完整 round。
2. 按下面描述给每个维度打 0–5 分。**不要算平均**，承认薄弱项。
3. 把总分 + 维度分布对照 [级别映射](#级别映射)。
4. 选**最低的两个维度**作为下一步重点学习方向。

每次 mock 和每篇 transcript 都做一遍。**round 之间的提升幅度**比绝对分数更重要。

## 8 个维度

### 1. 需求与范围

有没有先把题切清楚再动手画？

| 分 | 表现 |
| --- | --- |
| 0 | 直接开画。 |
| 1 | 问了一两个问题，没区分功能与非功能。 |
| 2 | 列了功能需求，忽略约束（一致性、延迟、多地域）。 |
| 3 | 功能/非功能清晰区分，明确标出 out of scope。 |
| 4 | 区分了 must-have 与 nice-to-have，主动暴露歧义，题目太大时主动缩范围。 |
| 5 | 把一个含糊的题目改写成更锐利的版本，并让面试官认同。 |

### 2. 规模估算

要数字，不要形容词。

| 分 | 表现 |
| --- | --- |
| 0 | 全程没有数字。 |
| 1 | 只甩了一个含糊数字（"很多用户"）。 |
| 2 | 只算了 DAU + QPS。 |
| 3 | QPS（读/写）、年存储、带宽、峰值 vs 平均。 |
| 4 | 上述 + 单分片负载、热点 key 估计、3–5 年增长。 |
| 5 | 数字直接驱动了一个具体设计决策（"40K 读/秒，一台 Postgres 不够，所以..."）。 |

### 3. 高层架构

读者能否仅凭图重建系统？

| 分 | 表现 |
| --- | --- |
| 0 | 没图，或一个大方块。 |
| 1 | 框画了，箭头缺失或语义不清。 |
| 2 | 只有 Client → API → DB 的 happy path。 |
| 3 | 读路径和写路径分开，异步路径明确标出。 |
| 4 | 组件职责清晰，API（资源 + 动词 + 鉴权）写明。 |
| 5 | 整场图始终成立——deep dive 是在原图上扩展，而不是推倒重画。 |

### 4. 数据模型与存储

最难蒙混过关的地方。

| 分 | 表现 |
| --- | --- |
| 0 | "用数据库存"。 |
| 1 | 选了 SQL 或 NoSQL，没说为什么。 |
| 2 | 表结构有了，主键定了。 |
| 3 | 索引明确，schema 由访问模式驱动，SQL vs NoSQL vs blob vs cache 给出理由。 |
| 4 | 分片键 + 再平衡方案、热分片处理、二级索引策略。 |
| 5 | 讨论了写放大、GC/compaction 行为，或所选存储的具体失效模式。 |

### 5. 可扩展与性能

峰值下读写到底落在哪。

| 分 | 表现 |
| --- | --- |
| 0 | "加机器"。 |
| 1 | 提了一层缓存，没有淘汰/失效策略。 |
| 2 | 缓存 + 副本，没 CDN，没异步管道。 |
| 3 | CDN/边缘、读副本、写时异步扇出、热路径之间的队列。 |
| 4 | 缓存失效策略（TTL vs 主动失效）、惊群处理、过载时的反压。 |
| 5 | 指出了**对应建模负载**的真实瓶颈，并用对工具（不是"Redis 万能"）。 |

### 6. 可靠性与权衡

说"最终一致"必须能讲清楚不一致时具体会坏什么。

| 分 | 表现 |
| --- | --- |
| 0 | 没提失败。 |
| 1 | 说了"复制"，没说一致性模型。 |
| 2 | 抽象地提了 CAP。 |
| 3 | 按数据类型分别给出一致性模型（强 / read-your-writes / 最终），并解释为什么。 |
| 4 | 走过失败场景（节点挂、分区、整 region 挂），讲清楚每种行为。 |
| 5 | 把 trade-off 主动说出来："我们牺牲 X 换 Y，业务能容忍 X 因为 Z。" |

### 7. 深入展开

"你是不是真高级"的核心信号。

| 分 | 表现 |
| --- | --- |
| 0 | 一整场都在框层面。 |
| 1 | 尝试了一次 deep dive，但讲得空。 |
| 2 | 一个组件下钻到算法/协议级（一致性哈希、leader election、OT/CRDT、token bucket 等）。 |
| 3 | 上述 + 考虑了替代方案并说明为什么不选。 |
| 4 | 两次独立 deep dive，都是生产级。 |
| 5 | Deep dive 揭示了面试官原本不打算测的东西。 |

### 8. 工程运维

决定系统能不能熬到周一。

| 分 | 表现 |
| --- | --- |
| 0 | 从没提。 |
| 1 | "我们会做监控"。 |
| 2 | 关键路径的 SLI（延迟、错误率）点名。 |
| 3 | 发布 + 回滚策略，on-call 信号（什么时候 page）。 |
| 4 | 滥用与安全（限流、授权边界），数据生命周期（PII、删除、保留）。 |
| 5 | 事故视角的叙事："凌晨 3 点 X 挂了，会通过 Y 触发 page，runbook 是 Z。" |

## 级别映射

**总分 + 维度分布**一起看。35 分但两个维度是 0，仍然是 no hire。

| 级别 | 总分 | 必须达到的下限 | 典型画像 |
| --- | --- | --- | --- |
| L3 / E3 / 新人 | 18–24 | 维度 1–4 每项 ≥ 3 | 基础干净，deep dive 可选，运维基本没讲。 |
| L4 / SDE II | 24–30 | 维度 1–6 每项 ≥ 3，至少一项 ≥ 4 | 基础扎实，一个真正的 deep dive，trade-off 能说出口。 |
| L5 / 高级 | 30–35 | 每维 ≥ 3，两项 ≥ 4 | 两个 deep dive，明确的一致性模型，主动认 trade-off，运维有提。 |
| L6 / Staff | 34+ | 每维 ≥ 4，至少一项 5 | Deep dive 像在跑生产，有独到洞察，运维成熟。 |

## 常见红旗（命中一条降一级）

- 还没厘清需求就开始画。
- 数字从未驱动设计选择。
- 嘴上"最终一致"，说不出不一致时会坏什么。
- 不论访问模式，所有数据都用同一种存储。
- 加了缓存但没失效策略。
- 选了分片键但没做热点 key 分析。
- 没走过任何失败场景。
- 完全没有 box 之下的 deep dive。
- "我们会加监控"但没点名任何 SLI。
- 面试官 pushback 时只是把原答案声音变大重复一遍。

## 自我打分技巧

- 面试一结束**立即**打分，记忆衰减很快。
- 不要根据"想说什么"打分——只看**实际说出来或画出来**的。
- 录下来回放；你以为说了 vs 你真的说了，之间的差距就是分数。
- 建一份跨多场 mock 的滚动维度评分表。同一个维度连续三场卡在 2 = 这就是你的瓶颈。
