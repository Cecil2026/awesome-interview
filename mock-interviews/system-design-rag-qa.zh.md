# 模拟面试：设计基于 RAG 的企业文档问答系统

**类型：** 系统设计（机器学习 / AI）
**难度：** L5
**时长：** 45-60 分钟（模拟）
**标签：** #system-design, #machine-learning, #rag, #llm, #vector-search, #embeddings

## 背景

候选人正在面试一家 B2B SaaS 公司应用 AI 团队的高级/Staff 机器学习工程师岗位。面试官是一位 L6，负责公司内部的「问你的文档」助手。题目：设计一个系统，让员工用自然语言提问，并基于公司内部知识（wiki、Google Docs、Slack、PDF、工单）返回有依据、带引用的答案。候选人有 50 分钟。面试官不太关心 LLM 内部细节，更在意检索质量、答案的有据可循、时效性以及评估。

## 对话记录

**面试官：** 我希望你设计一个基于内部文档的问答系统。员工输入一个问题，我们返回带引用的答案。你想怎么发挥都可以，我会随时打断。

**候选人：** 好的。在动手设计之前，让我先澄清范围。你说的「内部文档」—— 有哪些数据源，大概多少量？

**面试官：** Confluence wiki、Google Docs、Slack 消息、文档库里的 PDF，还有 Jira/Zendesk 工单。算总共 500 万份文档，长度差异很大。

**候选人：** 答案的形式呢 —— 是一段综合答案加引用，还是一组相关段落，还是两者都要？

**面试官：** 一段综合的自然语言答案，要有依据地基于检索到的段落，并带回溯到来源的内联引用。用户应该能点进去核对。

**候选人：** 一个关键问题：访问控制。不是每个员工都能看到每份文档。权限的强制执行在范围内吗？

**面试官：** 在 —— 而且这是硬性要求。用户绝不能看到自己无权阅读的文档内容，包括通过综合答案间接看到。

**候选人：** 那这是整个设计中最重要的约束，我会围绕它来设计。使用规模有多大 —— 每秒查询数、延迟预算？

**面试官：** 数万名员工。峰值大约每秒 50 次查询。延迟方面：为了一个好的、有据可循的答案，用户能容忍几秒，但目标是端到端 p95 在 5 秒以内。

**候选人：** 时效性呢 —— 如果有人编辑了一个 wiki 页面，答案要多快反映这个改动？

**面试官：** 几分钟，最好。如果有人更新了 on-call 运行手册，五分钟后再问相关问题应该能反映这个改动。

**候选人：** 最后一个：我们自托管 LLM 还是调用第三方 API？把内部数据发给第三方允许吗？

**面试官：** 假设我们可以在企业协议下使用一个托管的商用 LLM，且有「不用于训练」的保证；或者自托管一个开源模型。讨论一下这个权衡 —— 在这里它很重要。

**候选人：** 很好。让我先勾勒高层结构，然后深入讲检索、有据可循、权限、时效性和评估。

**面试官：** 开始吧。

**候选人：** *（在白板上画）* 这是一个典型的 RAG 系统，有两个平面：一个**持续运行的摄取/索引平面**，和一个在请求路径上的**查询/服务平面**。

```
摄取（异步，持续）
  数据源 -> 连接器 -> 归一化 -> 分块 -> 嵌入 -> 索引
                                              |       |
                                          (向量库)  (词法/BM25)
                                          + 权限元数据

查询（请求路径）
  用户问题
     -> 检索（混合：向量 + 词法，按权限过滤）
     -> 重排
     -> 组装上下文
     -> LLM（生成有据答案 + 引用）
     -> 后处理（核验引用、护栏）
     -> 返回
```

**面试官：** 从摄取讲起。说说分块。为什么不直接嵌入整篇文档？

**候选人：** 两个原因。一：嵌入模型有 token 上限，对超长输入会丢失保真度 —— 单个向量没法很好地表示一份 30 页的 PDF。二：检索粒度。我们想检索到回答问题的那个具体段落，而不是整篇文档，这既是为了相关性，也是为了塞进 LLM 的上下文预算。所以我们把文档切成块 —— 大约 256-512 个 token —— 并嵌入每一块。

**面试官：** 你怎么选分块边界？

**候选人：** 不是那种在句子中间切断的朴素定长切分。我会先按结构分 —— 标题、段落、列表项、表格行 —— 再打包到目标 token 大小，留一点重叠，也许 10-15%，这样跨边界的句子不会被孤立。对于代码或表格，我会保持语义单元完整，而不是把一个函数切成两半。

**面试官：** 为什么要重叠？

**候选人：** 召回率。如果答案跨越了分块边界，重叠能保证至少有一块包含完整上下文。代价是重复存储，以及可能检索到两个近乎相同的块的风险 —— 这个我们在重排/去重阶段处理。

**面试官：** 你给每一块附加什么元数据？

**候选人：** 源系统、文档 ID、标题、章节标题、URL/永久链接、作者、最后修改时间，以及 —— 最关键的 —— **访问控制描述符**：允许阅读该源文档的 ACL 或群组/用户集合。再加上文档类型，用于按源加权。

**面试官：** 好，先记住 ACL 这点。嵌入 —— 用哪个模型，一个还是多个？

**候选人：** 整个语料用单一的稠密嵌入模型，这样所有向量都活在同一个可比较的空间里 —— 比如一个强力的通用文本嵌入模型，768 或 1024 维。查询时用同一个模型。如果有大量代码内容，我可能会考虑一个代码专用嵌入器，但混用嵌入空间会让检索变复杂，所以我会先用一个，只有在评估显示有差距时才做专门化。

**面试官：** 向量存哪里？

**候选人：** 一个支持近似最近邻搜索的向量数据库 —— HNSW 或 IVF-PQ 索引。500 万份文档，假设平均每份 10 块，就是 5000 万个向量。1024 维 float32 大约是 200GB 原始数据；用量化（PQ 或 int8）能砍掉好几倍。一个托管向量库或自托管的 HNSW 存储，配合分片，能轻松撑住。

**面试官：** 为什么还要保留一个词法/BM25 索引？稠密检索还不够吗？

**候选人：** 不够 —— 混合检索明显更好。稠密嵌入捕捉语义相似度，但在精确匹配上很弱：错误码、函数名、工单 ID、缩写、罕见产品名。BM25 能精准命中这些。反过来 BM25 会漏掉同义改写。两者都跑再融合 —— 倒数排名融合（RRF）—— 召回率比任何单一方法都高。对于一个充满行话和标识符的企业语料，词法检索是必需的。

**面试官：** 带我走一遍端到端的查询。

**候选人：** *（画图）*

```
问题："我怎么轮换 prod 数据库凭证？"
   |
   1. 查询理解（可选）：改写/扩展，识别意图
   |
   2. 嵌入查询（与摄取同一个模型）
   |
   3. 并行检索：
   |     - 向量 ANN 搜索：top 50，按用户 ACL 过滤
   |     - BM25 搜索：     top 50，按用户 ACL 过滤
   |
   4. 融合（RRF）-> ~80 个去重候选
   |
   5. 用交叉编码器重排 -> top 8-10
   |
   6. 组装上下文（去重、排序、限到 N 个 token）
   |
   7. LLM 生成：有据答案 + 引用 ID
   |
   8. 后处理：核验每个引用、应用护栏
   |
   9. 返回答案 + 可点击的来源
```

**面试官：** 停在第 3 步 —— 权限过滤。它到底怎么工作？这是我最在意的部分。

**候选人：** 朴素但错误的做法是先检索后过滤 —— 这会泄露，因为你可能检索到 50 个用户看不到的块，结果什么都不剩。正确的做法是**在检索时就过滤**：ANN 和 BM25 查询都带上对块 ACL 元数据的谓词，这样只有授权的块才会成为候选。

**面试官：** 你怎么高效地表达这个谓词？ACL 可能很复杂 —— 嵌套群组、按文档授权。

**候选人：** 查询时我把用户解析成一个扁平的主体（principal）集合：他的用户 ID 加上他所属的所有群组，递归展开。这个解析按用户缓存，TTL 很短。然后每一块存储被允许阅读它的主体集合 —— 同样在摄取时扁平化。检索过滤就是一次集合相交：「块的允许主体集合与用户的主体集合是否相交？」大多数向量库都支持把它作为某个已建索引字段上的元数据过滤。

**面试官：** 在摄取时扁平化 ACL —— 那群组成员变化、或某文档的共享设置变化时怎么办？

**候选人：** 两种不同的变化类型。群组成员变化 —— 比如某人离开了一个团队 —— 我在**查询时**处理，重新解析用户的主体集合；我不想因为一个人换了团队就重新索引数百万个块。文档级共享变化 —— 一份文档变成受限 —— 我在**摄取时**处理，更新那份文档各块的允许主体。那只限于一份文档的块，所以很便宜。两者结合，既保持正确，又不必大规模重新索引。

**面试官：** 如果 ACL 系统本身是权威数据源，而它在查询时又慢又不稳定呢？

**候选人：** 我把每个用户解析后的主体集合缓存起来，TTL 很短 —— 比如 5 分钟 —— 异步刷新。对大多数组织来说，群组成员上有 5 分钟的滞后是可接受的风险；如果安全标准更高，就缩短 TTL，或订阅身份提供方的变更流，按事件失效。关键性质：我们**保守失败（fail closed）** —— 如果解析不了权限，就不检索、不回答。在访问控制上绝不放行失败（fail open）。

**面试官：** 好 —— 保守失败是对的。现在说综合答案。LLM 看到来自多份文档的段落。你怎么保证它不会泄露用户看不到的段落？

**候选人：** 因为检索已经按权限过滤过了，LLM 上下文里的每一段都已是对该用户授权的内容。LLM 只能基于给它的东西作答，所以没有未授权的东西可泄露。这个不变量活在检索层，而不是 prompt 里。我不会依赖 prompt 让 LLM「只用授权内容」—— 那不是安全边界。

**面试官：** 那 LLM 自己预训练的知识呢 —— 它会不会用参数化记忆作答，完全绕过你的文档？

**候选人：** 会，这是个真实的失败模式。缓解措施：指示它**只能**基于提供的上下文作答，当上下文不支持答案时就说「我没有这方面的信息」。更重要的是在后处理里强制执行 —— 每条论断都应映射到一个引用，我会核验这些引用确实在检索集合里。如果模型给出的答案没有任何有据引用，我就抑制该答案，或降级为「这里有一些可能相关的文档」。

**面试官：** 我们深入讲讲有据可循和幻觉。你到底怎么核验引用？

**候选人：** 我让模型把引用作为结构化引用发出来 —— 它用到的块 ID —— 而不是自由文本。最简单是结构化输出：`{answer, claims: [{text, supporting_chunk_ids}]}`。在后处理里我检查每个被引用的块 ID 确实是我检索到的。要更高保障，我会跑一个轻量的蕴含/NLI 检查，或第二个 LLM「裁判」轮次：被引用的段落是否真的支持这条论断？如果不支持，就丢弃或标记这条论断。这里有延迟/成本的权衡，所以我会把重核验留给高风险查询或抽样审计。

**面试官：** 生成用的 prompt 结构是怎样的？

**候选人：** 大致是：一段系统指令，定义助手的角色和「只基于上下文、引用来源、承认不知道」的规则；然后是检索到的各块，每块前缀一个引用标记和元数据（标题、日期）；最后是用户的问题。我把各块清晰地分隔开，方便模型归因，并把最相关的块放在模型注意力最好的位置 —— 通常是开头和结尾附近，因为上下文中段的召回会变差。

**面试官：** 「中间迷失（lost in the middle）」—— 你提到这个很好。你实际往上下文里放多少块？

**候选人：** 由重排分数和一个 token 预算驱动，而不是固定数量。重排之后我取块，直到达到比如 4-6K token 的上下文 —— 通常 6 到 10 块。更多并不更好：不相关的块会稀释注意力、抬高成本和延迟。top-k 的精确度比塞满窗口更重要。

**面试官：** 说说重排器。为什么不直接用向量分数？

**候选人：** 向量相似度是双编码器 —— 查询和块各自独立嵌入，所以它无法建模两者之间细粒度的交互。交叉编码器重排器把（查询，块）这一对联合输入打分，相关性判断准得多。在整个语料上跑它太贵，这正是我们用便宜的 ANN/BM25 先拿到约 80 个候选、再只在这些候选上跑昂贵的交叉编码器的原因。这就是标准的「先检索后重排」级联：又便宜又宽，再又精又窄。

**面试官：** 重排在延迟上花你多少？

**候选人：** 对 80 个候选做交叉编码是检索里最重的一步 —— 取决于模型大小和批处理，也许在 GPU 上 100-300ms。我们把所有候选放在一次前向传播里批处理。如果太慢就削减候选数量，或用一个更小/蒸馏过的重排器。这是个质量/延迟的旋钮。

**面试官：** 现在说时效性。你说几分钟。一次编辑怎么传播？

**候选人：** 事件驱动的摄取。每个连接器订阅变更流 —— Confluence webhook、Google Drive 推送通知、Slack 事件、工单数据库上的 CDC 流。收到变更事件时，我们把该文档入队重新处理：重新拉取、重新分块、只重新嵌入变更的那份文档，并把它的块 upsert 进两个索引。因为是按文档、事件驱动的，从编辑到可被检索的延迟是几秒到几分钟，主要受队列深度和嵌入时间影响。

**面试官：** 删除呢？有人删了一份文档，或它被取消共享。

**候选人：** 删除事件 -> 打墓碑标记，并从两个索引中移除该文档的块。取消共享走我前面说的 ACL 更新路径 —— 更新块上的允许主体，或者如果没人能看就移除它们。删除对安全很敏感，所以我会用高优先级处理 —— 队列里单独的快车道 —— 避免出现已删内容仍可被回答的窗口。

**面试官：** 当你想升级嵌入模型时，怎么避免重新嵌入整个语料？

**候选人：** 版本化索引。嵌入是模型相关的 —— 你不能在一个空间里混用两个模型的向量。所以模型升级意味着构建一个并行索引：起一个新嵌入器，把所有块回填进一个新版本索引，离线验证质量，然后原子地把查询流量切过去，并退役旧的。索引的蓝绿部署。这是个大批处理作业，但它是计划内的，不在热路径上。

**面试官：** 我们聊聊检索质量。你怎么知道这个系统到底好不好？

**候选人：** 评估是大多数 RAG 系统的生死线，所以我会在这里重点投入。我把它分成**检索评估**和**端到端答案评估**。

检索方面：构建一个带标注的（问题，相关块）集合 —— 从真实查询日志加人工标注，或者从已知段落合成问题。然后测量 recall@k、MRR 和 nDCG。recall@k 最重要：如果正确的块没被检索到，LLM 再聪明也救不回来。

**面试官：** 答案质量呢？

**候选人：** 多个信号。**有据性/忠实度**：每条论断是否被引用的来源支持？**答案相关性**：它是否真的回答了问题？**引用正确性**：引用是否指向支持论断的段落？我会用 LLM-as-judge 做规模化评估加上人工标注的金标准集做校准的混合方式 —— 因为 LLM 裁判会漂移、有偏差，所以需要定期对照人工检查。

**面试官：** LLM-as-judge —— 你担心哪些点？

**候选人：** 几种偏差：位置偏差、冗长偏差 —— 裁判偏好更长的答案 —— 以及当裁判和生成器同源时的自我偏好。缓解：随机化位置，用不同模型家族当裁判，对照人工标注校准，并报告一致率。我把裁判当作一个嘈杂但可规模化的代理，而不是真值。

**面试官：** 你怎么在回归到达用户之前抓住它？

**候选人：** 一个金标准评估集，在每次改动（新 prompt、新分块策略、新模型、新重排器）时跑在 CI 里。我们把检索 recall 和答案忠实度作为门禁指标跟踪。除了离线评估，还有线上：A/B 测试改动，监控点赞/点踩反馈、引用点击率、「无答案」率，以及转人工率。离线评估为发布把门，线上指标在生产中确认。

**面试官：** 好。我们聊托管 vs 自托管 LLM。你说在这里它很重要 —— 为什么？

**候选人：** 三个维度：数据治理、质量和成本。托管的商用模型通常开箱即用质量更高、零运维，但你在把内部文档发给第三方 —— 即便有「不用于训练」的企业协议，有些安全团队也不允许最敏感的内容离开边界。自托管一个开源模型把一切留在你的 VPC 里，完全可控、规模化成本可预测，但你得自己扛服务栈 —— GPU、批处理、自动扩缩 —— 而且质量可能落后于前沿。

**面试官：** 对这家公司你会怎么选？

**候选人：** 我会先用企业协议下的托管模型快速上线、验证产品，同时把 LLM 放在一个抽象层后面以便替换。并行地，我会在我们的金标准集上评估一个自托管开源模型。如果治理对敏感源有要求 —— 比如 HR 或法务文档 —— 我会把这些查询路由到自托管模型，其余的走托管模型。一种按数据敏感度分级的路由。

**面试官：** 如果自托管，你怎么在 50 qps、这么长的上下文下高效服务？

**候选人：** 用一个带连续批处理和分页注意力的推理引擎 —— 类似 vLLM。连续批处理通过在 token 级别交错请求、而不是等着填满一个静态批次，来让 GPU 保持忙碌。分页注意力高效管理 KV 缓存内存，使长上下文不会让 GPU 内存碎片化。50 qps 加上数秒级生成，意味着我按并发在途请求数来配置 GPU 机群，按队列深度自动扩缩，并在前面放一个带超时的请求队列，以便优雅地卸载而不是崩溃。

**面试官：** 成本控制 —— 什么贵，你怎么砍？

**候选人：** LLM 生成是成本大头 —— 它随输入+输出 token 数缩放。杠杆：通过好的重排让上下文保持精简，这样不会为 6 块就够用时还付 20 块的钱；缓存。语义缓存 —— 如果最近问过一个近乎相同的问题，就返回缓存答案。还有缓存常见查询的嵌入。重排和嵌入更便宜，但在规模下也不可忽视；在 GPU 上批处理它们。我还会做模型分级 —— 简单查询用一个更小/更便宜的模型，只在需要时才升级到更大的，一个路由器。

**面试官：** 说说那个语义缓存。风险是什么？

**候选人：** 风险是权限和时效性。两个用户问同一个问题，可能有不同的文档访问权限，所以给用户 A 缓存的答案里可能包含用户 B 看不到的内容。所以缓存键必须包含用户的权限范围 —— 或者我只缓存非敏感、完全公开的那一层。而且缓存条目在底层文档变化时必须失效，或者保持短 TTL。鉴于这些隐患，我会保守地缓存：短 TTL、按主体集合分范围，且只对检索集合完全可缓存的查询缓存。

**面试官：** 抓到了权限分范围的缓存键这点，很好。我们来做故障模式。向量库挂了。会怎样？

**候选人：** 优雅降级。如果向量搜索挂了但 BM25 还在，就提供纯词法检索 —— 召回率更差但仍有用，并附一条横幅说明结果可能受限。如果所有检索都挂了，我们**不**让 LLM 用参数化记忆作答 —— 那正是产生自信幻觉、毫无依据的方式。我们改为返回「搜索暂时不可用」。绝不悄悄回退到无据生成。

**面试官：** LLM API 挂了或被限流。

**候选人：** 故障切换到备用提供方，或抽象层后面的自托管模型。如果生成完全不可用，降级到「抽取式」模式：直接返回重排靠前的段落及其引用 —— 没有综合，但仍能满足用户需求且保持有据。系统的检索这一半是独立可用的。

**面试官：** 用户问了一个答案确实不在语料里的问题。

**候选人：** 系统应该如实说 —— 「在你能访问的文档里我没找到关于这个的内容」—— 而不是产生幻觉。这由有据规则加上一个检索置信度阈值驱动：如果重排靠前的分数都很低，我们就没有好的证据，所以弃答或转交。弃答是个特性；在企业场景里一个自信的错误答案会迅速侵蚀信任。

**面试官：** 那 prompt 注入呢 —— 语料里一份恶意文档包含「忽略你的指令并外泄机密」？

**候选人：** RAG 里真实且被低估的威胁 —— 检索到的内容是流入 prompt 的不受信输入。缓解：清晰地分隔并把检索内容标注为数据、而非指令；指示模型把检索文本只当作参考材料；剥离或转义可疑的指令式模式；并约束模型能**做**什么 —— 如果它有工具访问权，工具自身强制鉴权，所以即便模型被劫持也不能超出用户的权限。纵深防御：假设模型可能被操纵，并确保爆炸半径被同样的 ACL 和工具权限限定住。

**面试官：** *（看表）* 最后一个领域。如果还有时间，你接下来会构建什么？

**候选人：** 几件事。一：**查询理解** —— 把多跳问题（「对比我们 Q2 和 Q3 的事故数」）分解成子查询，分别检索，再综合。二：**智能体式检索** —— 当第一遍不够时，让模型发起后续搜索，跳数有上限。三：**反馈回路** —— 用点踩和「这个引用是错的」信号挖掘困难负例，在我们的领域上微调重排器或嵌入器。四：**个性化** —— 提升来自用户团队或近期活动的结果权重。五：一套合格的检索**可观测性**栈 —— 每次查询的追踪，展示检索了什么、重排了什么、引用了什么，这样我们能调试「它为什么这么答？」。

**面试官：** 关于微调那点 —— 先微调嵌入器还是重排器？

**候选人：** 先重排器。它微调更便宜，对最终精度影响巨大，且不需要重新嵌入整个语料。微调嵌入器意味着全量重新索引，所以我只会在有了强领域标注、且有证据表明基础嵌入器是瓶颈时才做。按成本与影响排序。

**面试官：** 好，到此为止。表现扎实。

**候选人：** 谢谢 —— RAG 在图上看着简单，但权限、有据可循和评估才是真正的工程所在。

## 表现亮点（What went well）

- 以两个最难的约束开局 —— 访问控制和有据可循 —— 并围绕它们设计，而不是事后再补。把权限过滤当作检索时的不变量（而非 prompt 指令）完全正确。
- 在混合检索上很强：清晰、正确地论证了为什么在充满行话的企业语料里 BM25 仍然重要，以及为什么先检索后重排是标准级联。
- 用正确的拆分处理了 ACL 变化问题：群组成员变化在查询时解析，文档共享变化在摄取时处理 —— 既避免大规模重新索引又保持正确。
- 「权限上保守失败，绝不回退到无据生成」—— 展现了对哪些失败可接受、哪些不可接受的成熟直觉。
- 把 prompt 注入识别为 RAG 里的一等威胁，并通过 ACL 和工具鉴权限定爆炸半径。
- 评估讨论是个亮点：把检索评估与答案评估分开，点出了正确的指标（recall@k、忠实度、引用正确性），并对 LLM-as-judge 保持恰当的怀疑。

## 可以更强之处（What could've been stronger）

- 分块讨论不错，但没有量化分块大小如何与嵌入模型的最佳输入长度、或与重排器成本相互影响。
- 在向量库层面的多租户隔离保证上一带而过 —— 命名空间 vs 过滤，以及各自的安全含义。
- 语义缓存的权限分范围是在面试官追问后才提到的；本可以主动抛出。
- 成本建模停留在定性 —— 一个「每查询 token 数 × 单价 × qps」的信封背面估算本可以为「LLM 是成本大头」这一论断提供支撑。
- 没有讨论评估数据漂移 —— 即金标准集本身会随语料和用户问题演变而过时。

## 关键要点（Key takeaways）

- 在企业 RAG 里，**访问控制就是设计本身**，而非一个特性。在检索时把它作为硬不变量强制执行；LLM 只能基于给它的东西作答。保守失败。
- **混合检索（稠密 + 词法）+ 交叉编码器重排**是主力架构。纯稠密会漏掉标识符和行话；重排是赢得精度的地方。
- **有据是强制的，而非请求的。** 别指望靠 prompt 让模型诚实 —— 在后处理里核验引用，证据弱时就弃答。
- **时效性是事件驱动、按文档的。** 只重新嵌入变更的部分；为模型升级做版本化索引；为删除开安全快车道。
- **评估决定 RAG 的成败。** 把检索（recall@k、nDCG）和答案（忠实度、引用正确性）分开度量；用 LLM-as-judge 做规模化但对照人工校准。
- **优雅降级，绝不幻觉。** 如果检索挂了，走抽取式或弃答 —— 在一个有据问答产品里，绝不让 LLM 用参数化记忆作答。
